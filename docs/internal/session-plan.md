## M17 — Workspace ergonomics
**Goal.** Four workspace changes under one umbrella, all delivered against the 2026-05-20 spec (which supersedes the narrower 2026-05-14 9p-securityModel spec): (1) make the 9p `securityModel` of `shared:` selectable per session — default `mapped-xattr`, opt-in `none` for real-symlink interop both directions; `passthrough` / `mapped-file` deliberately not exposed; (2) make the `shared:` guest path configurable with a breaking default of *preserving the host path* inside the guest (so build artefacts referring to the working directory survive a host↔guest round-trip) — pre-0.1.0 breaking-default change accepted; (3) add a new `local:` workspace mode — rsync-based host-snapshot at session create plus operator-driven `sandbox workspace push` / `pull` (`-f`/`-n` safety gate, `--safe-links`, `--no-gitignore`), no 9p or bind-mount surface, no live host writes; (4) add a workspace-lock subsystem — daemon in-memory per-session lock that serialises push/pull and atomically state-gates workspace ops at acquire time, refuses `sandbox stop` / `sandbox delete` against a session with an active workspace op, and surfaces a `sandbox workspace unlock --force` recovery path for orphan locks left by a crashed CLI. The expansion also restructures `SessionConfigDto` with a `workspace_mode_detail` field (DTO separation per the project convention) and renames `WorkspaceSecurityModel::None` to `NoneMapping` to remove the `Option::None` collision. CLI grammar grows to `shared:<host>[:<guest>][:<model>]`, `local:<host>[:<guest>]`, and a new `sandbox workspace unlock <session> [--force]` subcommand. M17-S1 ships the `shared:` parser + guest-path + securityModel + breaking default + DTO scaffold; M17-S2 ships `local:` mode core (variant, parser, daemon-side rsync orchestration at create, describe rendering, capability advertisement) on both backends with integration + create-side E2E; M17-S3 ships the push/pull CLI together with the workspace-lock subsystem they consume (acquire/release/force endpoints, lifecycle interaction with stop/delete, `unlock --force` CLI, orphan-lock recovery, lock unit + integration + E2E); M17-S4 is the six-track review across S1+S2+S3; M17-S5 is the spec-delivery verification. Terminal milestone for the 2026-05-20 spec.
**Status.** not_started · **Sessions.** 5 · **Details.** [milestones/M17.md](milestones/M17.md)

## M18 — Cross-user CLI access
**Goal.** Implement the 2026-05-24 cross-user CLI access spec: fix the six CLI commands (`sandbox ssh`, `cp`, `sync`, `workspace push`, `workspace pull`, and transitively `git-remote-sandbox`) that break when the daemon runs as the `sandbox` system user but the CLI is invoked by an unprivileged operator. The fix adopts the daemon-mediated approach used by Docker/libvirt/LXD/Kubernetes/systemd-machined: every cross-session operation tunnels through a new `GET /sessions/{id}/proxy` WebSocket endpoint via `ProxyCommand`, and the daemon distributes a per-session SSH config block via a new `GET /sessions/{id}/ssh-config` endpoint that the CLI persists under `~/.ssh/sandbox/` with an `Include` line inserted at the very top of `~/.ssh/config` between marker comments. Both backends unify on sshd by baking openssh-server into the lite-image (and renaming the in-container guest user from `agent` to `sandbox` so the SSH-config template's `User sandbox` line works without per-backend branching). M18-S1 reshapes the e2e harness to launch sandboxd via its existing systemd unit (or `sudo -u sandbox sandboxd` where systemd is unavailable) and lands the three Phase-1 acceptance tests (Lima ssh, Lima `git-remote-sandbox`, container ssh) so the bug is reproducible in CI before any production code change; M18-S2 bakes sshd into the lite-image and lands the guest-user rename; M18-S3 ships the `V007__add_ssh_keypair.sql` forward-only migration, per-session ed25519 keypair generation on container create, tmpfs `/run/sandbox/authorized_keys` injection, and the `GET /sessions/{id}/ssh-config` endpoint returning `SshConfigDto { config, private_key }` with `404 SSH_NOT_AVAILABLE` for pre-migration container sessions; M18-S4 ships the `GET /sessions/{id}/proxy` WebSocket byte mover with `tokio::io::copy_bidirectional` between the WebSocket and `tokio::net::TcpStream` (Lima) / `docker exec … socat` async-pipe child (container), with the deliberate `tokio::process::Command` carve-out from the project's `spawn_blocking` convention documented in inline code comments; M18-S5 ships the CLI `sandbox proxy <id>` hidden shim plus the persistent `~/.ssh/sandbox/` management module (flock + tempfile+rename atomic writes, marker-delimited `Include` block at the top of `~/.ssh/config`); M18-S6 rewrites the six broken CLI commands to use the `sandbox-<id>` alias and wires the single-retry drift-recovery path matched only at outermost CLI dispatch (never inside `sandbox proxy`, so `git-remote-sandbox`'s nested invocations cannot stack retries); M18-S7 wires the lifecycle hooks — `sandbox rm` cleanup of the per-session key + config block, `sandbox ls` opportunistic reconcile against the daemon's session list (full-list query only, silently skipped on unreachable daemon, `--no-reconcile` opt-out), and `sandbox proxy <id>` lazy-404 cleanup; M18-S8 is the six-track review across S1..S7 — six parallel subagents covering implementation-vs-spec, code quality, unit-test quality (with non-tautological-assertion focus and code-coverage-tooling setup), e2e-test quality (with non-tautological + coverage-gap focus), docs quality, and workarounds + deprecated patterns — followed by aggregation and fix-up; M18-S9 is the spec-delivery verification — claim-to-code map at `.tasks/specs/2026-05-24-cross-user-cli-access-design/2026-05-24-cross-user-cli-access-design-delivery.md` shipped, full e2e matrix executed; the matrix surfaced a second cross-user gap one layer below the CLI↔daemon surface: the daemon drives `limactl` directly as uid 999, but Lima's `_config/user` SSH identity, the per-VM instance dirs, and the QEMU process all need the operator's uid for OpenSSH `StrictKeyfileMode` and 9p `shared:` host-side DAC to hold. This is delivered as a second spec — `2026-05-29-m18-s10-sandbox-lima-helper-design` — across M18-S10..S15, superseding the supervisor-fork-as-operator sketch the post-S9 close first proposed. The fix is a dedicated `sandbox-lima-helper` setcap `cap_setuid+ep` binary (nine typed daemon-only subcommands, hardcoded argv, no pass-through) that pivots the daemon to the operator uid before exec'ing `limactl`, plus a per-operator `LIMA_HOME` at `/var/lib/sandboxd/<op-uid>/lima/` with POSIX ACLs and a per-operator golden base image; the load-bearing decision is that the daemon never invokes `limactl` directly. M18-S10 unifies the two `install_guest_agent` variants (refactor prerequisite); M18-S11 builds the helper crate standalone with full unit + integration coverage, unwired; M18-S12 lands the daemon helper resolver (hard-error on missing, no soft fallback) and the per-operator LIMA_HOME substrate (mkdir + setfacl + the `Mutex<HashMap<u32, Arc<LimaManager>>>` base-image-build registry); M18-S13 is the atomic flip — every call-site inventory row routes through the helper, `LimaManager::limactl_path()` is deleted, `sandbox-spawn-helper` is fully removed, `relax_lima_instance_perms` deleted, the startup orphan scan loops per-operator, a new V009 migration hard-deletes pre-migration null-`operator_uid` rows (V008's columns already shipped, so the cutover cannot edit it in place), and the cross-user e2e matrix goes green on both backends; M18-S14 is the seven-track review (impl-vs-spec, code quality, unit-test quality, e2e-test quality, docs + workspace-docs audit, workarounds + deprecated patterns, and a dedicated security audit of the privileged helper surface) with every BLOCKER and SHOULD-FIX fixed in-session under a no-open-todos rule; M18-S15 is the spec-delivery verification with a stricter conjunctive gate (every claim maps to (a) code+test or (b) a Non-goals citation — no `(c)` tracked-follow-on escape) and flips M18 to completed. Terminal milestone for both the 2026-05-24 spec (verified at S9) and the 2026-05-29 spec (verified at S15). **Close (2026-05-31).** Beyond the plan, S14 unified the Lima guest home to `/home/sandbox` (matching the container backend, retiring the `/home/agent` divergence that had caused a container `--repo` clone-path bug) and caught + fixed a release-integration regression the spec assumed already in place — `sandbox-lima-helper` was packaged/installed by no path, so a fresh install crash-looped the daemon; it is now wired into the release tarball, `release.yml`, `install.sh` (setcap `cap_setuid+ep`), and `uninstall.sh`. S14 also cleared all 17 outstanding deferred todos under the no-open-todos rule (one filed forward as GH issue #17 — manual VS Code Remote-SSH G9 verification). S15's claim-to-code map (154 claims → 140 (a) / 14 (b) / 0 (c) / 0 blockers) found one stale spec example: the per-operator `LIMA_HOME` `setfacl` prescribed a default named-user ACL (`d:u:<op>:rwx`) that breaks OpenSSH `StrictKeyfileMode` via the POSIX mask; the implementation was correct (`u:<op>:rwx,d:g::---,d:o::---`), so the spec example was amended in-branch. All gates green: e2e matrix 147/0/2, install-e2e 33/33, integration 182/182, nextest 1822, clippy `-D warnings` + fmt clean.
