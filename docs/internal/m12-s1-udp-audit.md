# M12-S1 UDP audit

Audit of UDP behavior end-to-end in `sandboxd` as it stands on master at the
start of M12-S1. This is the read-only inventory that the M12-S1
implementation phase (conntrack lookup), M12-S2 (e2e parity + doc
reconciliation), and M12-S3 (doc coherence) will all depend on. Citations
are `file:line` against the working tree at audit time.

A few orienting facts before the section walks:

- The deny-logger is a separate Rust binary (`sandbox-deny-logger`,
  `sandboxd/sandbox-deny-logger/src/main.rs`) that runs *inside* the
  per-session gateway container alongside Envoy / CoreDNS / mitmproxy. It
  emits JSONL (`/var/log/gateway/events/deny-logger.jsonl`) which sandboxd
  on the host tails via the per-session ingest watcher.
- nftables rules are generated by `sandbox-core` and pushed into the
  gateway container via `docker exec nft -f -`. There are three tables:
  `sandbox` (deny-all baseline + service-port input), `sandbox_dnat`
  (PREROUTING DNAT to CoreDNS / Envoy / deny-logger), `sandbox_policy`
  (egress allow on `output`).
- Envoy is TCP-only today: the listener config emits only
  `envoy.filters.network.tcp_proxy` filters with no `udp_listener_config`
  / `udp_proxy` shape. The `policy_allow_udp` set + UDP DNAT-to-Envoy
  rule are *forward-compat scaffolding*, not a live UDP-through-Envoy
  path.
- mitmproxy is gated implicitly: it is reached only via Envoy's L3
  CONNECT tunnel (`tcp_proxy.tunneling_config`). UDP never enters that
  tunnel.

prompt does not exist; the actual design doc is
`/home/olek/Projects/claude-sandbox/networking-design.md` at the repo

## 1. End-to-end UDP path map

Reference scenario: a process inside the sandboxed VM sends a UDP datagram
to an external `(dst_ip, dst_port)`. Walked layer by layer.

### 1.1 VM → bridge → gateway PREROUTING

Traffic arrives at the gateway container's network namespace via the
per-session Docker bridge. The kernel's PREROUTING hook fires before any
DNAT; the `sandbox_dnat.prerouting` chain is at `priority dstnat`.

The chain shape is generated in two equivalent places (same byte-shape;
the policy compiler rebuilds the whole table on apply):

- Initial / pre-policy default — `sandboxd/sandbox-core/src/gateway.rs:1449`
  (`fn generate_dnat_ruleset`), rules at lines `1462-1486`.
- Compiled-from-policy (post-apply, with set elements) —
  `sandboxd/sandbox-core/src/policy.rs:725` (`fn render_two_table_ruleset`),
  rules at lines `747-767`.

UDP-relevant rules in PREROUTING, in evaluation order:

1. **DNS to CoreDNS** — `gateway.rs:1466`, `policy.rs:751`:
   `ip saddr {vm_subnet} udp dport 53 dnat to {gateway_ip}:53`. Forces all
   port-53 UDP through CoreDNS regardless of original destination.
2. **Policy-allowed UDP to Envoy** — `gateway.rs:1473`, `policy.rs:756`:
   `ip saddr {vm_subnet} meta l4proto udp ip daddr . udp dport
   @policy_allow_udp dnat to {gateway_ip}:10000`. The
   `policy_allow_udp` concat set is keyed on
   `(ipv4_addr . inet_service)` and is populated by:
   - `PolicyCompiler::compile_nftables` —
     `sandboxd/sandbox-core/src/policy.rs:1007-1037` (CIDR rules go in at
     compile time; domain rules wait for DNS).
   - DNS propagation —
     `sandboxd/sandbox-core/src/dns_propagation.rs:368-403`
     (`udp_elements` accumulator) and `:427` (renderer call).
3. **Catch-all UDP to deny-logger** — `gateway.rs:1479`, `policy.rs:760`:
   `ip saddr {vm_subnet} meta l4proto udp dnat to
   {gateway_ip}:10002`. Fail-closed: any UDP that didn't match (1) or (2)
   gets DNATted to the deny-logger UDP listener.
4. **Cloud metadata block** — `gateway.rs:1482`: `ip daddr 169.254.169.254
   drop` (TCP/UDP-agnostic).
5. **IPv6 drop** — `gateway.rs:1485`: `ip6 daddr != ::1 drop`.

After PREROUTING, `nat hook postrouting priority srcnat` masquerades
egress (`gateway.rs:1488-1493`). Forward chain
(`gateway::generate_forward_allow_ruleset`,
`sandboxd/sandbox-core/src/gateway.rs:1565`) requires
`ip daddr {gateway_ip}` so any non-DNATted UDP is rejected at FORWARD
(`gateway.rs:1580`). This is the safety net referenced at
`tests/e2e/test_networking.py:401-444`.

The two `policy_allow_udp` concat-set declarations live in *both*
`sandbox_dnat` and `sandbox_policy` because nftables 1.0.6 (pinned in the
gateway image) rejects cross-table set references — see
`gateway.rs:1441-1448` and `policy.rs:713-719` for the rationale.

### 1.2 Input chain — admit DNATted UDP to local sockets

Once DNAT rewrites the destination to the gateway's own IP, the packet
hits `inet sandbox` `chain input`, which by default rejects everything.
The allow-list is generated by `generate_input_allow_ruleset`
(`sandboxd/sandbox-core/src/gateway.rs:1522`).

UDP-relevant accept rules:

- DNS — `gateway.rs:1537`: `ip saddr {vm_subnet} udp dport 53 accept`.
- Deny-logger UDP — `gateway.rs:1547`: `ip saddr {vm_subnet} udp dport
  10002 accept`. **No accept for `udp dport 10000`** (Envoy port). That
  is consistent with Envoy not having a UDP listener — even if a UDP
  packet reaches `policy_allow_udp` and is DNATted to `:10000`, it would
  be rejected at the input chain. This is a latent issue; see §3.

### 1.3 DNAT — where UDP gets its destination rewritten

PREROUTING DNAT is the *only* place UDP destinations are rewritten on
the VM-egress path. After DNAT, the kernel's conntrack machinery records
the original 5-tuple under `IP_CT_DIR_ORIGINAL` so reverse-path
translation (return packet d-NAT) can be applied — but the kernel does
**not** propagate the original destination to a UDP listener via
`IP_ORIGDSTADDR` for conntrack-DNAT'd UDP. The cmsg surfaces the
*post-DNAT, as-delivered-to-the-socket* destination, which after
DNAT-to-self equals the listener's bind address (`gateway_ip:10002`).

This is the asymmetry with TCP. For TCP, the kernel exposes the
pre-DNAT destination via `getsockopt(SOL_IP, SO_ORIGINAL_DST)` on the
accepted socket, populated by
`netfilter/nf_conntrack_l4proto_tcp.c`. There is no equivalent
syscall-style accessor for UDP; the documented recovery path is
**netfilter conntrack netlink** (`NFNL_SUBSYS_CTNETLINK` →
`IPCTNL_MSG_CT_GET` keyed on the post-DNAT 5-tuple, returning the
`CTA_TUPLE_ORIG` attribute).

### 1.4 Deny-logger UDP listener

Defined in `sandboxd/sandbox-deny-logger/src/udp.rs`. Bind site
(`sandbox-deny-logger/src/main.rs:180`): `udp::bind(args.bind_ip,
args.udp_port)`. Listener loop (`udp.rs:56-88`).

The socket is opened with `IP_RECVORIGDSTADDR=1`
(`sandbox-deny-logger/src/udp.rs:43`). On `recvmsg`
(`udp.rs:101-109`), the loop walks ancillary cmsgs looking for an
`Ipv4OrigDstAddr` cmsg (`udp.rs:117-126`). If present, the cmsg's
sockaddr is used as `orig_dst_ip:orig_dst_port`
(`udp.rs:121-124, 132-141`). The peer (`src_ip:src_port`) comes from
the recvmsg's `address` field
(`udp.rs:111-114`).

**Why this doesn't recover the pre-DNAT destination.** The kernel fills
`IP_ORIGDSTADDR` with the destination address actually written into the
IP header at the moment of delivery to the socket. For non-DNATted UDP
this equals the original destination (which is why the cmsg is
documented as carrying "the original destination"). For DNATted UDP
this equals the post-DNAT destination — the deny-logger's own bind
address. There is no kernel-side "leak" of the pre-DNAT 4-tuple into
the UDP cmsg path; conntrack stores it but does not surface it via
`IP_ORIGDSTADDR`.

The result: every UDP deny event today carries
`orig_dst_ip = gateway_ip`, `orig_dst_port = 10002`. The 5-tuple's
`src_ip:src_port` is correct (the recvmsg `address`), but the
destination half is the listener's own address — useless for
attribution.

After emitting (or dropping at the rate cap), the datagram payload is
discarded (`udp.rs:62-87`). No reply is sent. The fixed receive buffer
is 128 bytes (`udp.rs:34`).

### 1.5 Why TCP works (and what UDP would need)

The TCP deny path lives at
`sandboxd/sandbox-deny-logger/src/tcp.rs`. After `accept(2)`
(`tcp.rs:126`), `read_original_dst` (`tcp.rs:222-249`) calls
`getsockopt(SOL_IP, SO_ORIGINAL_DST)` on the accepted socket. The
kernel returns the pre-DNAT destination from conntrack
(see `tcp.rs:226-230` for the safety / contract comment). The peer
address (`src_ip:src_port`) is the `accept(2)` argument, passed in via
`peer` at `tcp.rs:158`.

Concretely: TCP can recover pre-DNAT because the kernel exposes a
syscall accessor against conntrack that's keyed on the *socket file
descriptor* — accept gives you a stateful conntrack-anchored fd, and
`SO_ORIGINAL_DST` reads from the conntrack entry the kernel already
threaded through that fd. UDP has no per-flow fd; the listener socket is
shared by every datagram. So even though conntrack *has* the pre-DNAT
tuple (it has to, to do reverse-path translation), there's no socket
option to read it from. The substitute is to query conntrack by 5-tuple
out-of-band — netlink.

### 1.6 Envoy chains — UDP bypass

Envoy's `policy_listener` is generated by
`PolicyCompiler::compile_envoy_listener`
(`sandboxd/sandbox-core/src/policy.rs:1236`). The listener YAML is
emitted at `policy.rs:1796-1827` (`listener_yaml_for_filter_chains`).
The `address.socket_address` block (`policy.rs:1811-1814`) does **not**
specify `protocol`; Envoy defaults `socket_address.protocol` to TCP.
There is no `udp_listener_config` block, no `udp_proxy` filter, and no
UDP-shape filter chain.

All filter chains across L1/L2/L3 emit
`envoy.filters.network.tcp_proxy` filters
(`policy.rs:1299, 1416, 1438, 1526, 1546`). The L3 path uses
`tcp_proxy.tunneling_config.hostname = %DOWNSTREAM_LOCAL_ADDRESS%`
(`policy.rs:1311-1320`) which depends on the `original_dst` listener
filter populating `%DOWNSTREAM_LOCAL_ADDRESS%` from
`SO_ORIGINAL_DST` — itself TCP-only.

So Envoy doesn't bypass UDP via an explicit gate; it simply has no UDP
listener. UDP DNATted to `gateway_ip:10000` would arrive at a kernel
socket that no Envoy worker is listening on, and get dropped. **And in
fact it can't even reach there**: the input chain at
`gateway.rs:1522-1554` only accepts UDP on `:53` and `:10002`. Any UDP
DNATted to `:10000` is rejected at INPUT before the kernel even
attempts socket dispatch. So `policy_allow_udp` is effectively wired to
"DNAT to a destination that the input chain rejects". See §3.

### 1.7 mitmproxy — UDP not applicable

mitmproxy is reached via Envoy's L3 CONNECT tunnel
(`policy.rs:1311-1320`), specifically `tcp_proxy.tunneling_config`
which emits an HTTP/1.1 `CONNECT` preface on a TCP upstream stream to
`127.0.0.1:18080`. UDP never enters that tunnel because (a) UDP never
reaches Envoy (see §1.6), and (b) the tunneling shape itself is
TCP-only.

There is no explicit UDP gate in
`networking/mitmproxy/passthrough_addon.py`,
`networking/mitmproxy/policy_addon.py`, or
`networking/mitmproxy/events.py`; the only `udp` substring in
`networking/mitmproxy/*.py` is incidental (a docstring and a comment
about TCP tunnels). UDP is **implicitly** non-applicable: it never
reaches mitmproxy.

`docs/guides/network-policies.md:174` explicitly notes "Non-TCP
protocols (QUIC/HTTP/3, raw UDP) cannot be inspected at this level"
which is consistent with the implementation.

### 1.8 Diff vs the TCP path (summary)

| Concern               | TCP path                                                                 | UDP path                                                            |
|-----------------------|--------------------------------------------------------------------------|---------------------------------------------------------------------|
| Pre-DNAT recovery     | `getsockopt(SOL_IP, SO_ORIGINAL_DST)` on accepted socket                 | `IP_ORIGDSTADDR` cmsg returns post-DNAT (broken)                    |
| Per-flow fd           | Yes, from `accept(2)`                                                    | No, single shared listener socket                                   |
| RST on close          | `SO_LINGER{1,0}` then drop (`tcp.rs:257-265`)                            | None — datagrams discarded silently (`udp.rs:62-86`)                |
| Concurrency cap       | RAII `ConnGuard` against `AtomicU32` (`tcp.rs:74-106`)                   | None — UDP rate-capped only via per-process `RateCap`               |
| Allow path            | TCP DNAT to Envoy:10000, INPUT accepts :10000                            | UDP DNAT to Envoy:10000, but INPUT rejects :10000 → DNAT is dead    |
| Allow set population  | `policy_allow_tcp` from CIDR rules + DNS-resolved IPs                    | `policy_allow_udp` populated identically — but unreachable today    |

## 2. Deny-event attribution gap (#29)

### 2.1 The use site

Pre-DNAT recovery in the UDP listener happens at
`sandboxd/sandbox-deny-logger/src/udp.rs:117-126`. The cmsg parser
walks the `Ipv4OrigDstAddr` ancillary message and extracts a
`SocketAddrV4`. For DNATted UDP, the kernel populates this with the
*post-DNAT* destination (see §1.3 / §1.4 above): the deny-logger's own
bind address, `gateway_ip:10002`. The resulting `DenyRecord` carries
`orig_dst_ip = gateway_ip`, `orig_dst_port = 10002` — useless for
discovery / attribution.

The setsockopt that enables the cmsg is at `udp.rs:43`. The setsockopt
itself works (the cmsg is delivered); the bug is that the cmsg's
contents are post-DNAT, not pre-DNAT.

### 2.2 The Phase 8 test that documents the workaround

`sandboxd/sandboxd/tests/gateway_deny_pipeline.rs:609-732`. The relevant
asserted tuple is at lines `711-715` and `724-729`:

```
d.protocol == DenyProtocol::Udp
    && d.orig_dst_ip == gateway_ip
    && d.orig_dst_port == 10002
    && d.src_ip == want_src_ip
    && d.src_port > 0
```

The verbatim explanatory comment, lines `670-694`:

> The UDP deny-logger was designed to read `IP_ORIGDSTADDR` from the
> `recvmsg` cmsg and emit the **pre-DNAT** destination (expected:
> `203.0.113.1:9999`).
>
> Observed behavior on the current gateway image: the Linux kernel
> does not expose the pre-DNAT destination via `IP_ORIGDSTADDR` for
> conntrack-DNAT'd UDP the same way it does for TCP's
> `SO_ORIGINAL_DST` (TCP queries conntrack via
> `nf_conntrack_l4proto_tcp`; UDP's cmsg only surfaces the packet's
> destination as delivered to the socket, which after DNAT equals
> the listener's own bind address — `gateway_ip:10002`).
>
> The deny-logger emits `orig_dst_ip=<gateway_ip>,
> orig_dst_port=10002` for every UDP deny. Capturing the pre-DNAT
> tuple would require a conntrack netlink lookup in the deny-logger
> or a different nftables marking strategy — both out of scope for
> M10-S3 (handoff prohibits touching the deny-logger wire shape).
>
> Like Test 1's stderr-signature relaxation, we pin the primary
> contract (a `deny` event **is** emitted for a blocked UDP
> datagram, with the correct src_ip and protocol) and accept the
> DNAT-target orig_dst rather than the intended pre-DNAT
> destination. The 5-tuple's pre-DNAT accuracy is tracked upstream
> as a deny-logger UDP follow-up.

There is also a guard comment at `gateway_deny_pipeline.rs:696-699`:

> NOTE: if this assertion ever fails with orig_dst = 203.0.113.1:9999,
> the UDP pre-DNAT conntrack fix (deferred todo #29) has landed —
> update this test to the pre-DNAT tuple instead of reverting.

### 2.3 Proposed fix mechanism — conntrack netlink lookup

After `recvmsg` produces the post-DNAT 4-tuple
`(src_ip:src_port, gateway_ip:10002)`, the deny-logger needs to ask
the kernel: *given this current 5-tuple, what was the original
destination before DNAT rewrote it?*

The supported answer is the netfilter conntrack netlink protocol
(`NFNL_SUBSYS_CTNETLINK`). The deny-logger sends an
`IPCTNL_MSG_CT_GET` request to a `NETLINK_NETFILTER` socket carrying
a `CTA_TUPLE_REPLY` (or `CTA_TUPLE_ORIG`, depending on direction)
matching the post-DNAT flow; the kernel replies with the conntrack
entry, and `CTA_TUPLE_ORIG.CTA_IP_V4_DST` +
`CTA_TUPLE_ORIG.CTA_PROTO_DST_PORT` give the pre-DNAT destination.

For DNATted VM→gateway UDP:
- Post-DNAT (what the listener sees as a flow): src=VM-IP:VM-port,
  dst=gateway-IP:10002, proto=UDP. This is the `REPLY` direction's
  inverse, i.e. the `ORIG` direction.
- The kernel's `ORIG` tuple after DNAT mutation actually carries the
  pre-DNAT destination as the original destination address. The
  precise CTA selection (ORIG vs REPLY) depends on which direction
  the kernel labels as "original" for a VM-initiated flow that DNATs
  the destination — empirically `ORIG.dst` carries the pre-DNAT IP
  (the VM dialled), `REPLY.src` carries the post-DNAT IP (the
  gateway).

In implementation terms, the lookup happens once per datagram:
construct a netlink request keyed on the post-DNAT 4-tuple, parse the
reply for the original destination tuple, populate `DenyRecord` with
that. Failure modes (entry expired, kernel races) need a documented
fallback — log + drop, or emit with a sentinel orig_dst to make the
deny visible without claiming a wrong destination.

### 2.4 Candidate Rust crates

**No conntrack / netlink crate is currently in the workspace.**
`sandboxd/Cargo.toml:17-46` lists no `netlink-*`, `conntrack`, or
`nfnetlink` dependency. The deny-logger's deps
(`sandboxd/sandbox-deny-logger/Cargo.toml:15-35`) are
`nix`, `socket2`, `clap`, `serde`, `tokio`, `tracing*`. A new
dependency is required.

Candidate crates available on crates.io:

- `nfct` — conntrack-netlink with high-level entry semantics; pulls in
  `mnl`-style C bindings. Active.
- `netlink-packet-netfilter` + `netlink-sys` (the `rust-netlink`
  ecosystem) — pure-Rust netlink encode/decode + raw netlink socket.
  More plumbing but no C deps. Active (`rtnetlink` family).
- `nflog` / `conntrack-rs` — narrower scope, may or may not match.

The plan-prompt names `nfnetlink_conntrack` (the C library, libmnl
family) — wrapping it would mirror the libnetfilter-conntrack C
interface. The Rust-native `netlink-packet-netfilter` route avoids
linking a C library inside the gateway image (which is currently
`debian:bookworm-slim` with the deny-logger as a static-ish binary).

The implementation phase needs to make this dependency call. Notable
constraint: the deny-logger runs inside the gateway container, which
already has `CAP_NET_ADMIN`, so opening a `NETLINK_NETFILTER` socket
should work without further capability bumps.

## 3. Latent issues found during audit

### (a) Fix in M12-S1 (tightly coupled to conntrack work)

**a1. UDP pre-DNAT attribution itself.** `sandbox-deny-logger/src/udp.rs:117-126`.
This is the headline #29 fix. Replace the cmsg-only path with a
conntrack lookup keyed on `(src_ip:src_port, gateway_ip:10002, UDP)`.
Document the netlink crate choice and the failure-mode policy in the
listener docstring at `udp.rs:1-13`.

**a2. Phase 8 test 2 rewrite.** `sandboxd/sandboxd/tests/gateway_deny_pipeline.rs:610-735`.
After (a1) lands, rewrite the assertion (lines 711-715, 724-729) to
expect the pre-DNAT tuple `203.0.113.1:9999` and remove the
explanatory comment block at lines 670-699. This is M12-S1's named
deliverable.

**a3. Operator-log degradation path on cmsg-missing.**
`sandbox-deny-logger/src/udp.rs:74-79`. Today, a missing
`Ipv4OrigDstAddr` cmsg silently drops the deny event with only a
`tracing::warn!`. After the conntrack-lookup change the cmsg is no
longer the source of truth, but the listener still needs a degraded
mode for the case where the conntrack lookup fails. Decide whether
"netlink lookup failed" should still emit a deny event with a
sentinel destination (visible attribution gap, but the deny itself is
not lost) or drop+log (current behaviour for missing cmsg). M12-S1
should pin this in the same commit as a1.

### (b) Defer to M12-S2 (e2e coverage parity / doc reconciliation)

**b1. UDP allow-path is currently unreachable.** `sandboxd/sandbox-core/src/gateway.rs:1547`
is the only UDP `accept` rule on the input chain (besides DNS). UDP
DNATted to Envoy on `:10000` is rejected at INPUT. Either (i) add an
`udp dport 10000 accept` to `generate_input_allow_ruleset` if
"policy-allowed UDP through Envoy" is supposed to actually work
(requires Envoy UDP listener too, see b2), or (ii) explicitly remove
the `policy_allow_udp` → Envoy DNAT rule (`gateway.rs:1473`,
`policy.rs:756`) and the egress-allow rule (`policy.rs:806`) to
reflect that UDP allow-path doesn't exist yet. M12-S2's e2e parity
work needs to settle this — *what does "allow UDP" actually mean
today*?

**b2. Envoy has no UDP listener.** `sandboxd/sandbox-core/src/policy.rs:1796-1827`
emits `socket_address` with the default TCP protocol and only
TCP filter chains. If b1.(i) is the path, an Envoy
`udp_listener_config` + `udp_proxy` filter is needed too.
Alternatively, route policy-allowed UDP straight through nftables
(masquerade-only, no DNAT) and only DNAT non-allowed UDP to the

### (c) Defer to M12-S9 (new follow-on, doesn't fit S1 or S2)

**c1. UDP listener has no per-source rate limit / concurrency cap.**
`sandbox-deny-logger/src/udp.rs:60-87` shares the per-process
`RateCap` with TCP (`limits.rs`). A high-rate UDP storm from one
VM could exhaust the deny-logger's per-second emit budget,
crowding out denies from other layers (CoreDNS, mitmproxy, Envoy)
that share the same JSONL. TCP has at least a connection cap
(`tcp.rs:117-153`). UDP has no equivalent — no per-source bucket.
Tractable as a M12-S9 hardening item: per-source token bucket on
UDP, similar shape to the existing per-process `RateCap`.

**c2. UDP recv buffer fixed at 128 bytes silently truncates.**
`sandbox-deny-logger/src/udp.rs:34`. The truncation is intentional
(payload is discarded anyway) but `msg.bytes` returned at
`udp.rs:140` is the *truncated* count, not the original datagram
size. If we ever want to surface "this datagram was N bytes" in
the deny event for diagnostic purposes, we'd need `MSG_PEEK` first
or `MSG_TRUNC`-aware accounting. Low-priority M12-S9 item.

**c3. No explicit policy validation that http+udp is rejected at the
boundary.** There *is* such validation (`policy.rs:2386`,
`validate_rejects_http_level_with_udp`) but it's a unit test, not an
e2e contract. If the policy DTO ever loses that check, e2e wouldn't
catch it. Could fold into M12-S2's UDP test additions or stand
alone as M12-S9 polish. Low priority.

**c4. Forward-compat dead code: `policy_allow_udp` set populated by
DNS propagation.** `sandboxd/sandbox-core/src/dns_propagation.rs:368-403`
populates `policy_allow_udp` from domain rules with `Protocol::Udp`.
Until UDP-through-Envoy actually works (b1, b2), DNS-resolved IPs
landing in `policy_allow_udp` only steer UDP into a DNAT-then-INPUT-reject
path, not a real allow path. The set population is correct
forward-compat scaffolding, but operators looking at
`nft list set inet sandbox_dnat policy_allow_udp` may believe UDP
is being allowed when it's not. Doc-only fix unless b1/b2 land. M12-S9
note.

## 4. Doc claims vs reality

Cross-referenced against `docs/concepts/networking.md`,
`docs/guides/network-policies.md`, `docs/concepts/policy-model.md`,
`docs/guides/troubleshooting.md`, and `networking-design.md` (repo root —
the M12-S1 plan-prompt mentioned `docs/internal/networking-design.md`
which does not exist).

### Untrue today, tag "S2 reconciliation"

**4.1 `docs/concepts/networking.md:141`** (already cited at b3):

> recovers the pre-DNAT destination via `SO_ORIGINAL_DST` /
> `IP_ORIGDSTADDR`

True only for TCP. **Tag: S2 reconciliation.**

**4.2 `docs/concepts/networking.md:130`** (already cited at b4):

> The deny-logger records the pre-DNAT 5-tuple

For UDP, the destination half is post-DNAT. **Tag: S2 reconciliation
(or auto-resolves once M12-S1 lands).**

**4.3 `docs/concepts/networking.md:191`**:

> Even if an application skips DNS and dials an IP directly, its
> `(daddr, dport)` must be in `policy_allow_{tcp,udp}` … for the
> DNAT to Envoy to fire. Anything else is redirected to the
> deny-logger, which records the attempt and RST-closes the flow.

"records the attempt" is true (modulo 4.1/4.2). "RST-closes the flow"
is TCP-only. UDP is silently discarded. Doc could say "RST-closes the
flow (TCP) / drops without reply (UDP)". **Tag: S2 reconciliation.**

**4.4 `docs/guides/network-policies.md:174`**:

> Non-TCP protocols (QUIC/HTTP/3, raw UDP) cannot be inspected at
> this level. QUIC is blocked by the deny-all firewall since no rule
> opens UDP/443.

The first sentence is correct ("cannot be inspected"). The second
sentence is structurally true (no rule opens UDP/443) but
operationally misleading: even if a rule did add `(host, 443, udp)`
to `policy_allow_udp`, UDP would still not work end-to-end because
the input chain rejects `udp dport 10000` (b1, §1.6, §3.b1). **Tag:
S2 reconciliation.**

**4.5 `docs/concepts/policy-model.md:60, :108`** — describe `protocol`
as `tcp | udp` without UDP-specific behavior caveats. Not strictly
untrue, but understates the asymmetry. **Tag: S2 reconciliation
(additive).**

**4.6 `networking-design.md:392, :411, :417, :479`** — repeat that "UDP
is handled purely by nftables (IP/port allow/deny)". This is the
intended end-state. Today the deny path goes through the deny-logger
binary (a userland process), not "purely nftables" — the nft DNAT
points userland-ward. The allow path in nft is wired but unreachable
(see b1). Wording should describe what the implementation actually
does. **Tag: S2 reconciliation (doc-side).**

**4.7 `networking-design.md:1395`** (in §17 Security model):

> UDP policy is enforced purely by nftables (IP/port allow/deny)
> with no userland proxy

Same drift as 4.6. **Tag: S2 reconciliation.**

**4.8 `networking-design.md:1221`**:

> use REJECT (TCP RST for TCP, ICMP unreachable for UDP) rather than
> DROP where possible

Today the deny-logger drops UDP without ICMP unreachable
(`sandbox-deny-logger/src/udp.rs:62-87`). The deny-logger is the
endpoint — it doesn't reject upstream. **Tag: S2 reconciliation
(decide whether to honor the original intent by sending ICMP, or
document that the silent-drop is the correct behavior).**

### Already accurate (no action)

- `docs/concepts/networking.md:127-130` — the table description of
  `sandbox_dnat` and the concat sets is accurate.
- `docs/concepts/policy-model.md:159` — the `policy_propagated` event
  description correctly references both `policy_allow_tcp` and
  `policy_allow_udp`.
- `docs/guides/troubleshooting.md:25` — accurate at the rule-set level.
- `docs/guides/troubleshooting.md:156` — DNS interception description
  is accurate.
