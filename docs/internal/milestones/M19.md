# M19 — macOS foundations: operator-execution seam + per-operator agent

The first milestone of the macOS support arc. macOS reaches Linux-equivalent security guarantees through different mechanisms, and the load-bearing difference is the **operator-execution model**: where Linux pivots to the operator's uid via a `cap_setuid` `sandbox-lima-helper`, macOS runs each operator's `limactl` inside a **per-operator LaunchAgent** that already runs as the operator, brokered by an unprivileged system daemon. M19 lands that foundation — the cross-platform build baseline, the `LimaExecutor` seam that abstracts "run this limactl op as operator N" (Linux behaviour preserved verbatim), the agent binary + authenticated broker channel, the hardened broker protocol (the macOS analog of the most-reviewed file in the Linux tree), and the macOS `AgentBrokerExecutor` that completes the seam. **No sandbox VM boots in M19** — that is M20. This milestone is the architectural pivot the rest of the arc builds on, and it is the part that refactors *Linux* code, so it lands and is reviewed in isolation.

This is **not** the terminal milestone of the spec (the arc spans M19–M23), so M19 carries no spec-delivery verification session; the claim-to-code gate lands at M23. The arc is executed as a single unit and **published only after M23 closes** — no milestone ships independently (§ macOS arc execution model in `session-plan.md`).

## M19-S1 — Cross-platform workspace cleanliness (build + clippy on macOS)

**Entry criteria.** Fresh.

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — § M15 → "Cross-platform workspace cleanliness (clippy)"; § Overview (scope / Non-goals: Apple-Silicon-only, dev-only).

**Rationale.** Nothing macOS-specific can be developed until the Rust workspace compiles and lints cleanly on an Apple-Silicon host, and three crates ship Linux-only binaries that pull Linux-only deps (`nix`, `caps`, `netlink-sys`) which will not build under `cfg(target_os = "macos")`. Landing the cfg-gating sweep first unblocks every later macOS session and keeps `cargo clippy --workspace` green in the coming `build-macos` CI lane. The change is mechanical and confined to those three crates, so it is a clean, low-risk opener that leaves every other crate and all Linux behaviour untouched.

**In scope.**

- Apply the three-step cfg-gating treatment to `sandbox-route-helper`, `sandbox-nft-deny-logger`, and `sandbox-nft-allow-logger` exactly as the spec's § Cross-platform workspace cleanliness enumerates (Linux-only deps under `[target.'cfg(target_os = "linux")'.dependencies]`; Linux code relocated into a `#[cfg(target_os = "linux")] mod imp`; non-Linux stub `fn main()` printing the Linux-only message and exiting 2).
- Confirm the cross-platform-clean baseline the same section asserts: `sandbox-guest` and `sandbox-event-emitter` already compile on macOS, and `cargo fmt --workspace` already works.

**Explicitly deferred.**

- The `build-macos` CI job that runs this on a real runner → M22-S4 (§ CI matrix).
- The `/usr/local/libexec/sandboxd/sandboxd` daemon-path move (a cross-platform install cleanup the spec folds into macOS work) → M22-S1.

**Exit criteria.**

- `cargo build --workspace`, `cargo clippy --workspace`, and `cargo fmt --workspace --check` all pass on an Apple-Silicon macOS host; the three Linux-only binaries' stub mains exit 2 with the spec's message when run on macOS.
- Linux build/clippy/fmt and the full hermetic + integration suites remain green — no behavioural change to any Linux path.
- Code review approved; the macOS build baseline noted where contributor docs describe the workspace.

## M19-S2 — Extract the operator-execution seam (`LimaExecutor`)

**Entry criteria.** M19-S1.

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — § Platform detection → "The operator-execution seam (`LimaExecutor`)"; § Glossary (operator-execution seam, pivot, per-operator LIMA_HOME).

**Rationale.** The daemon today reaches `limactl` only through the post-M18 setuid helper at Linux-shaped call sites; macOS needs the same operations delivered by a per-operator agent instead. Factoring "run this limactl op as operator N" into a `LimaExecutor` trait with a behaviour-preserving Linux `SetuidHelperExecutor` is the load-bearing refactor that lets the macOS executor slot in later without re-touching the call sites. Keeping Linux byte-for-byte unchanged here is the whole point — the seam is introduced, not new behaviour.

**In scope.**

- Introduce the `LimaExecutor` trait with the full op set the spec's seam table enumerates (`Create`, `Start`, `Stop`, `Delete`, `Clone`, `List`, `InstallGuestAgent`, `Copy`, `ReadUserKey`, `RunRsync`, `GuestSocat`) per § The operator-execution seam.
- Implement `SetuidHelperExecutor` as the verbatim post-M18 helper invocation and route every existing direct-helper call site through the trait, per the seam's "Linux is unchanged" row.
- Carry the seam through the shared registry structure (`LimaManagerRegistry` → `LimaManager` → its `LimaExecutor`) the spec sketches; have `LimaRuntime` construct the Linux executor at startup (resolve + verify helper, fatal on missing) exactly as today.

**Explicitly deferred.**

- `AgentBrokerExecutor` (the macOS impl) → M19-S5.
- The macOS gateway degenerate case (`op_uid == _sandbox` runs limactl directly) → M19-S5.

**Exit criteria.**

- Full hermetic + integration suite green on Linux; the Lima integration tests that exercise the helper pass unchanged, proving the refactor is behaviour-preserving.
- `cargo clippy --workspace` clean on both Linux and macOS (the trait + Linux impl compile on macOS even though only the Linux executor is constructed there).
- Code review approved; the seam documented at its definition site in its own terms (no milestone tags).

## M19-S3 — Per-operator agent binary, broker channel, and authenticated check-in

**Entry criteria.** M19-S2.

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — § Privilege model on macOS → The per-operator agent (launchd scoping, Check-in, Anti-spoof + authorization, limactl invocation); § M15 → Install paths (agent binary + LaunchAgent plist) and install.sh changes (per-operator agent install).

**Rationale.** macOS replaces the setuid pivot with a LaunchAgent that runs as the operator and checks in to the daemon's broker socket; this session stands up that binary, its launchd scoping, and the authenticated check-in channel — the substrate the broker protocol rides on. The authentication rules are security-critical (kernel peer-cred uid, `getgrouplist` enrollment gate, `uid < 500` rejection, last-writer-wins eviction, heartbeat) and must be built exactly as specified, because the channel admits anything running as an enrolled operator. The protocol payloads come next; here the channel must be correct first.

**In scope.**

- Build the `sandbox-agent` binary and the `io.sandboxd.agent` LaunchAgent with the mandatory `LimitLoadToSessionType: Aqua` scoping and the clean-exit `_sandbox`-membership self-gate, per § The per-operator agent → launchd scoping.
- Implement the agent→daemon check-in over the broker socket at the path/owner/mode the spec fixes (`/var/run/sandbox/broker.sock`, 0660 `_sandbox:_sandbox`, daemon recreates before listening), incl. single-connection-per-uid eviction and the heartbeat/liveness ping, per § Check-in.
- Implement daemon-side anti-spoof + authorization exactly as § Anti-spoof + authorization requires: peer uid from `LOCAL_PEERCRED` (via `peer_cred()`), reject uid 0 / `< 500`, re-resolve `_sandbox` membership via `getgrouplist(3)` (never the peercred group array).
- The agent's own `limactl` PATH-establishment (launchd does not inherit the operator's interactive PATH) per § limactl invocation.

**Explicitly deferred.**

- The wire protocol crossing the channel (subcommand enum, validation, streaming) → M19-S4.
- Wiring the channel into the `LimaExecutor` seam → M19-S5.
- install.sh provisioning of the agent + LaunchAgent on a real install → M22-S2 (this session ships the binary/plist as source artifacts plus a dev-time registration path).

**Exit criteria.**

- Hermetic + integration tests cover the daemon-side check-in authentication (peer-cred uid gate, `getgrouplist` membership, `< 500` rejection, duplicate-uid eviction, heartbeat-driven drop) without requiring a GUI session.
- The agent builds as an ad-hoc-signed Mach-O (per the signing prerequisite) and self-exits cleanly for a non-enrolled user with no respawn loop.
- Code review approved (with attention to the authentication path as a trust surface); the broker-socket contract documented at its definition site.

## M19-S4 — Hardened broker protocol (the trust surface that replaces the setuid helper)

**Entry criteria.** M19-S3.

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — § Privilege model on macOS → Broker protocol (closed subcommand enum, agent-side re-validation, daemon-authored networking, NON-FEATURES, one-shot vs streaming); § Trust model.

**Rationale.** The broker is the macOS analog of the most carefully-reviewed file in the Linux tree, and the spec holds it to the same hardened bar; this session specifies the wire protocol and, critically, the agent-side enforcement that makes it safe. The load-bearing properties — closed subcommand enum with typed fields (no argv/shell pass-through), agent-side re-validation of every field, the agent acting only for its own uid, and daemon-authored networking — must be implemented and reviewed as the helper's NON-FEATURES block is. Getting the `GuestSocat` streaming framing right (distinct from one-shot request/response) is part of the same surface.

**In scope.**

- Define the tagged-enum wire protocol over the check-in channel covering the full op set, with typed/named fields and per-subcommand hardcoded `limactl` flag construction, per § Broker protocol → Closed subcommand enum, no pass-through.
- Implement the agent-side re-validation the spec mandates (VM-name regex, absolute-no-`..` paths confined to the operator's own `LIMA_HOME`, numeric range checks, reject-not-clamp) and the own-uid gate (agent learns its uid from `getuid()`, rejects mismatched `op_uid`/`LIMA_HOME`) per § Agent-side re-validation and § Agent verifies the work is for its own uid.
- Implement daemon-authored networking handling: the agent writes the daemon-rendered `lima.yaml` verbatim after re-validating it (socket_vmnet-only NICs, no extra NICs, asserts the sandboxd-shipped `--qemu-wrapper`), per § Networking is daemon-authored.
- Implement the explicit NON-FEATURES contract and the `GuestSocat` long-lived bidirectional sub-stream framing distinct from one-shot ops, per § NON-FEATURES and § One-shot vs streaming ops.
- Include the **post-boot `restrict=on` verification op** in the closed enum now (the agent reads the wrapper marker / queries QMP `query-netdev` and returns the management-netdev `restrict` state) with the same own-uid + VM-name re-validation — so M20-S6 wires a *reviewed* op rather than introducing a new trust-surface variant *after* the M19-S6 security review (§ vzNAT and SLIRP → Post-boot verification).

**Explicitly deferred.**

- Daemon-side routing of ops to the right agent and the gateway degenerate case → M19-S5.
- The actual rendered macOS `lima.yaml` / qemu-wrapper the agent validates → M20-S3 (this session validates the shape; M20 produces the content).

**Exit criteria.**

- Hermetic + integration tests prove the agent **rejects** (not clamps) malformed VM names, traversal paths, foreign-uid requests, extra NICs, and a missing/mismatched qemu-wrapper assertion; the NON-FEATURES contract is enforced in code, not by convention.
- The `GuestSocat` stream framing round-trips bidirectionally under test, distinct from one-shot framing.
- Code review approved with a dedicated pass treating the protocol + NON-FEATURES block as the load-bearing review surface the spec demands.

## M19-S5 — `AgentBrokerExecutor` and login-session execution routing

**Entry criteria.** M19-S4.

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — § Platform detection → operator-execution seam table (macOS `AgentBrokerExecutor`, gateway degenerate case); § Privilege model → Login-session constraints; § Session lifecycle → "Operator identity & execution routing"; § Daemon startup sequence (step 1: AgentBrokerExecutor construction + broker-channel bind); § Dev workflow (single-uid foreground case).

**Rationale.** This session completes the seam on macOS: the daemon routes each `LimaExecutor` op to the target operator's checked-in agent, and special-cases its own uid (gateway infra, foreground dev) by running `limactl` directly. The login-session constraint — that an operator's limactl work is available only while their agent is checked in — must surface as the spec's actionable error, never a silent hang, because it is the defining cost of the no-pivot model. With this, the macOS seam is functionally complete even though no VM boots until M20.

**In scope.**

- Implement `AgentBrokerExecutor` routing per-`op_uid` to that operator's checked-in agent connection, with the `op_uid == _sandbox` degenerate case running `limactl` directly as the daemon, per the seam table and § Gateway instance is the degenerate case.
- Construct the macOS executor at startup and bind the agent-broker channel (no helper to resolve on macOS) per § Daemon startup step 1; thread the single-uid foreground dev case (`op_uid == self` → direct) per § Dev workflow.
- Surface the agent-not-checked-in condition as the actionable `session agent for <user> is not running` error at every routed op, per § Login-session constraints and § Per-operator agent unavailable.

**Explicitly deferred.**

- Agent-crash / fast-user-switching reconcile semantics → M21-S2.
- The session-create flow that drives these ops end-to-end → M20-S4.

**Exit criteria.**

- Integration tests cover op routing to a test agent, the `_sandbox` degenerate direct path, and the actionable error when no agent is checked in — without requiring a GUI session.
- The macOS daemon starts, binds the broker channel, and accepts an agent check-in end-to-end in the harness; Linux remains on `SetuidHelperExecutor` unchanged.
- Code review approved; the seam's two implementations documented at the trait definition.

## M19-S6 — Security review of the agent + broker trust surface

**Entry criteria.** M19-S5.

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — § Privilege model on macOS (Broker protocol, Trust model, Anti-spoof + authorization); § M13 (`NGROUPS` cap vs DAC, `getgrouplist` authority).

**Rationale.** The agent/broker pair is the macOS replacement for the setuid helper, and the spec repeatedly insists it be reviewed to the same bar; a dedicated security-focused review before the networking/VM work builds on top of it catches trust-surface defects while the surface is still small. The review is adversarial against a *buggy daemon* and a *uid-impersonating non-agent process* — the two threats the spec's boundary actually scopes against — not against a malicious agent (out of model). Findings are fixed in-session under a no-open-blocker rule.

**In scope.**

- Adversarial review of the broker protocol enforcement against § Broker protocol NON-FEATURES and § Trust model: confirm no argv/shell/env pass-through, agent acts only for its own uid, daemon-authored networking cannot be subverted by a daemon-routing bug, and no cross-session safety decision rests on agent-reported state.
- Review the check-in authentication against § Anti-spoof + authorization and § M13: kernel peer-cred uid, `getgrouplist` membership authority, `< 500`/uid-0 rejection, eviction + heartbeat correctness.
- Confirm the no-launchd-hardening-hazard and no-setuid-binary-on-host invariants hold (§ Trust model → No launchd hardening hazard).
- **Assert the `LimitLoadToSessionType: Aqua` scoping** prevents a `LoginWindow`/non-Aqua (root) load — a *distinct* failure mode from the `_sandbox`-membership self-gate — so the agent can never load as root and attempt a uid-0 broker check-in (§ The per-operator agent → launchd scoping).

**Explicitly deferred.**

- The full multi-track arc review → M23-S1; the spec-delivery verification → M23-S2.

**Exit criteria.**

- Every BLOCKER and SHOULD-FIX from the review is fixed in-branch (no open blockers carried out of the session); trust-surface invariants are backed by tests where testable.
- Full hermetic + integration suite green on Linux and macOS; clippy `-D warnings` + fmt clean on both.
- Code review approved; any spec inaccuracy found during review corrected in the spec in-branch.
