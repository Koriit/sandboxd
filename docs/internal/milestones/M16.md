# M16 — Update infrastructure

Spec 5 closes the five-spec arc. It adds the `sandbox update` CLI subcommand (the only supported in-place upgrade path after Spec 4's install.sh), the config migration framework that applies Spec 1's V001 transform and future transforms to `/etc/sandboxd/users.conf` and `/etc/qemu/bridge.conf`, the lock-file mutex that guards concurrent updates, the backup mechanics that make the documented manual rollback recipe possible, and the daemon-side schema-mismatch refusal that closes the safety loop. M16 is the terminal milestone for Spec 5 and for the entire five-spec arc.

**Spec references:**

- [`../specs/2026-05-11-update-infrastructure-design/2026-05-11-update-infrastructure-design-spec.md`](../specs/2026-05-11-update-infrastructure-design/2026-05-11-update-infrastructure-design-spec.md) — M16-S1..S3, M16-S4..S5

## M16-S1 — Config migration framework + daemon schema-mismatch refusal + `rebuild-image --backend gateway` refusal

**Entry criteria.** Two independent gates, both required:

1. **M14 complete** (Spec 3) — `load_users_config()` and `ensure_base_dir_layout` call sites must exist in `sandboxd/src/main.rs` before `validate_users_conf_schema_version` can be wired in immediately after the config load; the `SandboxError` / `UsersConfigError` mapping machinery must be in place.
2. **M15 complete** (Spec 4) — `scripts/install.sh` must exist so that `scripts/lib.sh` has source material to extract the cosign pin constants from; Spec 4's tarball layout and MANIFEST format must be finalised so the cosign constants in `lib.sh` match what the release pipeline produces and what M16-S2's fetch/verify logic must verify.

**Spec reference.** `../specs/2026-05-11-update-infrastructure-design/2026-05-11-update-infrastructure-design-spec.md` — §§ 4.1–4.7 (config migration framework: registry location, `ConfigMigration` trait, apply loop, atomic write semantics, version detection per file type, preserving operator content, daemon-side schema-mismatch refusal), § 8 (`rebuild-image --backend gateway` refused), § 13 (implementation notes: `cfg_migrations/`, `users_conf.rs`, `bridge_conf.rs`, `main.rs` wiring).

**Rationale.** The migration framework and daemon schema validation are pure Rust library code with no shell scripting and no privileged operations — they are the most isolated deliverable in Spec 5 and the natural starting point. The daemon-side `validate_users_conf_schema_version` call ships here because it must precede the update orchestration (M16-S2): the validator is the convergence anchor that forces operators to run `sandbox update` when the config file drifts behind the binary's supported version. The `rebuild-image --backend gateway` refusal is a one-variant CLI change that belongs alongside the framework it documents as the "right path instead."

## M16-S2 — Lock file + `sandbox update` pre-flight + `--check`/`--dry-run` modes

**Entry criteria.** M16-S1 (the migration framework must exist for the pre-flight's migration dry-run to call; `--dump-migration-set` must exist for the `--dry-run` stopped-session classification; `scripts/lib.sh` must exist for the cosign bootstrap to source it).

**Spec reference.** `../specs/2026-05-11-update-infrastructure-design/2026-05-11-update-infrastructure-design-spec.md` — §§ 2.1–2.5 (CLI surface, `--check` output, `--dry-run` output, confirmation prompt, privilege model), § 2.6 (log file destination — same file as install.sh), §§ 3.1.1–3.1.12 (pre-flight steps: arg parse, dev-mode detect, install state read, target version + fetch, version compare with `--check` exit gate, active sessions check, stopped session compat enumeration, disk space, cosign bootstrap, sigstore verify + extract, migration dry-run, confirmation prompt), §§ 6.1–6.4 (lock file: path/shape, acquisition shell pseudo-code, ordering rule flock-before-payload, acquisition outcomes, release, sticky `was_running` flag), §§ 10.1, 10.3 (lock mode rationale, self-replacement semantics).

**Rationale.** The read-only half of `sandbox update` — the pre-flight, `--check`, `--dry-run`, and the lock mechanics — forms a self-contained unit whose exit criteria do not require stopping the daemon or writing backups. Separating it from the stateful steps (M16-S3) gives reviewers a focused PR: "does `--check` produce the right output without side effects? does the lock correctly serialize concurrent invocations?" The lock file module is included here (not S3) because it is acquired as the first stateful step and its unit tests are hermetic.

**In scope.**

- `sandboxd/sandbox-cli/src/update/lock.rs` per Spec 5 §§ 6.1–6.4: lock-file path (`/var/lib/sandbox/.update.lock`, mode `0664 sandbox:sandbox`), acquisition (non-blocking `flock -n -x` via shell invoked from Rust `Command`, JSON payload write, dead-PID adoption with sticky `was_running`, stale-payload detection at >24h), release (`rm -f + exec {fd}>&-`). Unit tests: `lock_file_acquisition_refuses_on_live_holder`, `lock_file_acquisition_adopts_on_dead_pid_payload`, `lock_file_acquisition_preserves_was_running_across_adopt`, `lock_file_flock_acquired_before_payload_write`, `lock_file_released_on_process_exit`.
- `sandboxd/sandbox-cli/src/update/fetch.rs` per Spec 5 §§ 3.1.4, 3.1.9, 3.1.10: release tarball fetch (GitHub Releases API for latest-tag resolution, `--from` local path, `--source-url` mirror), cosign bootstrap (downloads cosign v2.4.1, sha256-verifies, stores in tmpdir; sources constants from `scripts/lib.sh` via a Rust `include_str!` constant or a parallel `const`-duplication with a comment pointing at lib.sh), sigstore `verify-blob`, tarball extraction, MANIFEST sanity + per-file sha256 verification. MANIFEST arch cross-checked against install-state `installed_arch`, not `uname -m`.
- `sandboxd/sandbox-cli/src/update/mod.rs` pre-flight logic per Spec 5 §§ 3.1.1–3.1.12: full arg-parse (superset of install.sh flags), dev-mode detection and refusal, install-state read, target-version determination, version compare (up-to-date short-circuit), `--check` early-exit gate, active sessions query, stopped session compat enumeration using current daemon's `DAEMON_GUEST_PROTO_VERSION` (for `--check`) and target binary's constants via `--dump-migration-set` (for `--dry-run`), disk-space check, cosign bootstrap delegation to `fetch.rs`, sigstore verify + extract, migration dry-run via `apply_pending` in memory (read + transform + validate, no write), confirmation prompt (`Proceed? [y/N]:`, `--yes` bypass, `--dry-run` prints plan + exits 0).
- `Command::Update { ... }` variant in `sandbox-cli/src/main.rs` per Spec 5 § 13; dispatch arm calling `update::run(args)`.
- `--check` output verbatim per Spec 5 § 2.2 (including exit codes: 0=up-to-date, 1=error, 2=arg-parse-failure, 3=update-available).
- `--dry-run` output verbatim per Spec 5 § 2.3 (would-execute/skip labels, § 3 step numbers referenced).
- Update log appended to `/var/log/sandbox-install.log` with `sandbox-update` second token per Spec 5 § 2.6.
- Unit test `version_lifecycle_check_then_dry_run_then_apply` (mocked daemon `/version`; assert each phase output shape; assert no privileged calls in `--check`/`--dry-run`).

**Explicitly deferred.**

- Stateful steps §§ 3.2.13–3.2.30 (stop daemon, backup, install binaries, start daemon) → M16-S3.
- Lima E2E tests → M16-S3.

**Exit criteria.**

- All lock-file unit tests pass under the default nextest profile.
- `sandbox update --check` from a lower version reports "Status: update available", exits `3`, and does not acquire the lock (lock file absent after invocation).
- `sandbox update --check` at current version reports "Status: up to date", exits `0`.
- `sandbox update --dry-run` prints the full plan with `would execute`/`skip` labels for all 18 stateful steps, exits `0` when all pre-flight passes.
- `sandbox update --from <tarball>` with a mismatched arch (per MANIFEST) dies with "MANIFEST arch mismatch" before any state mutation.
- `sandbox update` (no flags) after a successful pre-flight presents the confirmation prompt with `Proceed? [y/N]:` and exits without acting on `N`.
- `cargo nextest run --workspace` default profile clean; `cargo clippy --workspace` clean; code review approved.

## M16-S4 — Multi-track review

**Entry criteria.** M16-S3 complete — all three implementation sessions landed; `cargo nextest run --workspace` default and integration profiles pass; Lima E2E harness tests pass.

**Spec reference.** `../specs/2026-05-11-update-infrastructure-design/2026-05-11-update-infrastructure-design-spec.md` (all sections).

**Rationale.** The update command is the most security- and reliability-sensitive deliverable in the arc: it runs with `sudo`, stops the daemon, swaps binaries, and applies config transforms — all while maintaining idempotency. The six review tracks are calibrated to catch failure modes specific to this scope (TOCTOU races in the lock, partial-backup states that break the retention filter, config migration transforms that silently drop operator content, security gating on the hidden CLI flags).

**In scope.**

Six tracks, each delegated to a separate review agent in parallel:

- **Track 1 — Implementation vs. spec**: Re-read Spec 5 §§ 1–11 section by section; compare every concrete requirement (step order, log line format, file mode, JSON field name, error message token, lock-file acquisition outcome, retention filter predicate, atomic-write guarantee) against the implementation; flag any gap or divergence.
- **Track 2 — Code quality**: Review the `update/` module, `cfg_migrations/`, `lock.rs`, `backup.rs`, and `migrate.rs` for: ownership/borrowing correctness in the lock acquisition (no FD closed prematurely), correct use of `std::process::Command` for all privileged `sudo -k` calls, error propagation (no silent `unwrap` in the error path of stateful steps), idiomatic Rust. Check that every `sudo -k` elevation has a corresponding log line before it fires. Review the hidden subcommand security gates for completeness (all four conditions from § 4.3).
- **Track 3 — Unit test quality**: For every unit test in §§ 9.2 covering the lock, framework, schema-mismatch, and subcommand gates: verify the assertion is non-tautological (the tested property is not reachable without the production code path). Run `cargo-llvm-cov` on `cfg_migrations/`, `update/lock.rs`, `update/backup.rs`, and `sandbox-core/src/users_conf.rs`; identify uncovered lines and document each with a reason or a new test. Specifically: verify `apply_pending_atomic_write_visible_only_after_complete` actually injects a fault and observes the pre-write state (not just a successful path with a comment).
- **Track 4 — E2E / integration test quality**: For each Lima E2E test in `test_update_*.py`, verify the assertion covers the full observable post-condition (not just exit code): `test_update_interrupted_then_resumed` must parse the second run's log and count `action=skip` lines; `test_update_then_manual_rollback` must verify daemon version, sessions.db row count, and doctor exit code post-rollback; `test_update_concurrent_refused` must read stderr and verify the "in progress" message substring. Verify `test_update_check_does_not_mutate` inspects the lock file path directly (not inferred). Check that `test_update_backup_retention_prunes_oldest` counts the backup-set directories, not just trusts the prune log.
- **Track 5 — Docs quality**: Verify the rollback recipe in `docs/` is copy-pasteable without modification on a standard Ubuntu 22.04 VM (trace through each step; confirm path names, file modes, and tool invocations match the spec's § 7.2 verbatim); verify `docs/start/installation.md` update is accurate; check that the refusal message for `sandbox update` on a dev install (§ 11) matches the spec's verbatim text including the migration URL.
- **Track 6 — Workarounds + deprecated patterns**: Grep `sandbox-cli/src/update/` and `sandbox-cli/src/cfg_migrations/` for: (a) any `unwrap()` or `expect()` in non-test paths (report each; the production paths should propagate errors via `?`); (b) any cosign sha256 constants duplicated outside `scripts/lib.sh` (both the Rust cosign-bootstrap and install.sh must source/reference the same values — flag if they diverge); (c) any `TODO`/`FIXME` introduced in M16 sessions; (d) any place where the `--apply-config-migration` root-check (§ 4.3) could be bypassed by calling an internal function directly (audit the call graph). For tests: flag any tautological test.

- Every "must-fix" finding addressed, re-implemented where needed, and re-tested.
