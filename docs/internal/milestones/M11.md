# M11 — Lite mode: container backend

Introduce a second session backend for `sandboxd` — a Docker container selected via `--lite` (or `--backend container`) — sitting alongside the existing Lima/QEMU VM backend behind a new backend abstraction. The user promise is full UX parity with VM sessions and much faster session creation, traded for container-level isolation rather than VM-grade isolation. The four-phase rollout (abstraction refactor → container runtime feature-flagged off → user-facing feature → parametrization + polish) is preserved as the linear S1..S5 ordering, with a terminal spec-delivery verification session in S6.

**Spec references:**

- [`2026-04-22-lite-mode-container-backend-design/2026-04-22-lite-mode-container-backend-design-spec.md`](../../../.tasks/specs/2026-04-22-lite-mode-container-backend-design/2026-04-22-lite-mode-container-backend-design-spec.md) — primary spec; M11-S1..S6.

## M11-S2 — Route helper, users.conf, daemon startup validation

**Entry criteria.** M11-S1 complete.

**Spec reference.** `.tasks/specs/2026-04-22-lite-mode-container-backend-design/2026-04-22-lite-mode-container-backend-design-spec.md` — Context §§ "Install-time setup", "Operating constraints → Deployment model"; Architecture § "Paths summary" (`sandboxd/sandbox-route-helper/` row); Container specifics § "Networking" subsections "Why Docker's default route is wrong for lite", "Fix: setcap route helper", "Helper authorization flow" (the eight ordered steps + PID TOCTOU closure + cross-user MITM closure), "Config file: `/etc/sandboxd/users.conf`", "Attack table", "Daemon privilege (unchanged)"; Rollout § "Phase 2 — container runtime (feature-flagged off)" (helper + users.conf bullets only).

**Rationale.** Lands the privilege-bounded primitives the container backend depends on, ahead of the runtime impl that consumes them. The route helper and `users.conf` form a self-contained security boundary that can be tested in isolation: the helper performs the eight-step authorization flow on demand, the config parser is shared between daemon (subnet-scope lookup at startup) and helper (per-invocation authorization). Splitting these from the container runtime keeps each PR within a reviewable size and keeps the privileged-binary surface (one ~200-line crate with one entry point) reviewable on its own merits without being entangled with Docker plumbing.

**In scope.**

- Architecture § "Paths summary" — new standalone crate at `sandboxd/sandbox-route-helper/`, ships as a binary alongside `sandboxd` with packaging guidance for `setcap cap_sys_admin+ep` (file capabilities only — no setuid bit).
- Container specifics § "Helper authorization flow" — full eight-step flow implemented: `getuid()` → username resolve via `getpwuid`; parse `<gateway-ip>` argument; load `/etc/sandboxd/users.conf` and find the subnet whose `cidr` contains `<gateway-ip>` (no match → deny); check caller's username is in that subnet's `allow_users` (not present → deny); `pidfd_open(container_pid, 0)` + `setns(pidfd, CLONE_NEWNET)` atomically (PID TOCTOU closure via pidfd); enumerate all non-loopback interface addresses in the target netns and require every non-`lo` address to live in the matched subnet (cross-user MITM closure); `ip route replace default via <gateway-ip>`; exit 0. Any failed step is a deny: non-zero exit, no action taken on the container's netns. Numeric uid comparison internally; usernames in the config are admin readability only.
- Container specifics § "Config file: `/etc/sandboxd/users.conf`" — JSON shape per project convention (see CLAUDE.md "JSON-only config" rule), root-owned, mode `0644`. Single loader implementation in `sandbox-core` shared between daemon and helper; one parse path, one struct, one set of tests.
- Context § "Install-time setup" — kernel 5.8+ floor enforced (helper uses `pidfd_open` from 5.3+ and `setns(pidfd, CLONE_NEWNET)` from 5.8+); install docs spell out the `setcap cap_sys_admin+ep` step alongside the existing `qemu-bridge-helper` setuid-root step in the same install runbook (mirrors the operator contract operators already follow for the VM backend).
- Daemon-startup validation — daemon loads `users.conf` at startup, finds the subnet entry whose `allow_users` contains the daemon's own user's username, and uses that `cidr` to scope `NetworkManager`'s per-session `/28` allocation. If no matching section exists, the daemon refuses to start with an error pointing at `/etc/sandboxd/users.conf` and the install docs. Daemon remains unprivileged; only reads.
- Container specifics § "Daemon privilege (unchanged)" — verify (and document in the daemon-side glue) that no new capabilities are granted to the daemon process; the helper is the only new privileged surface; `docker` and `kvm` group memberships are unchanged.

**Explicitly deferred.**

- Daemon invoking the helper from a container's lifecycle (between `docker start` and the agent-ready wait) → M11-S3.
- `ContainerRuntime` + `ContainerTransport` + Dockerfile + `ensure_image()` → M11-S3.
- `GET /backends` endpoint → M11-S3.
- All CLI surface (`--lite`, capability errors, isolation warning, list/inspect, config file precedence, `rebuild-image --backend`) → M11-S4.
- User-facing docs `docs/lite.md` → M11-S4.

**Exit criteria.**

- `cd sandboxd && cargo nextest run --workspace` green; `make test-integration` green; `make test-e2e` full suite green (Lima path unchanged — the helper is not yet wired into any session-create path, so no E2E regression surface).
- New unit/integration coverage in `sandbox-core` for the `users.conf` loader: valid multi-subnet shape; missing file is a hard error with a pointer to the path; malformed JSON is a hard error with the same pointer; subnet-by-`<gateway-ip>` lookup returns the right entry; numeric-uid `allow_users` check is case- and rename-stable.
- Three route-helper authorization tests covering the deny branches called out in the spec's Testing § "Route-helper authorization", each driving the helper against a tmp `users.conf` fixture (test-only env var or bind-mounted config — does not touch `/etc/sandboxd/users.conf` on the host): cross-user MITM rejected at step 6 (container netns IP outside caller subnet); caller not in `allow_users` rejected at step 4; gateway IP outside any defined subnet rejected at step 3. Each test asserts non-zero exit, the prescribed stderr pattern, and that the target netns default route was not modified.
- Daemon-startup test: daemon refuses to start when `users.conf` is missing, malformed, or has no subnet matching the daemon's user; the error message points at the file path and the install docs.
- Grep sweep: no `CAP_SYS_ADMIN` granted to the daemon process at any call site; helper's `allow_users` check uses numeric `getuid()` ground truth, not username strings.
- `/review` (or `kio-reviewer`) run over the full diff; every finding addressed in-branch.

## M11-S3 — ContainerRuntime, lite image build, `GET /backends`

**Entry criteria.** M11-S2 complete.

**Spec reference.** `.tasks/specs/2026-04-22-lite-mode-container-backend-design/2026-04-22-lite-mode-container-backend-design-spec.md` — Architecture § "Two implementations" (Container impl row); Architecture § "Paths summary" (container.rs and Dockerfile rows); Image building § entire section ("First-use build, not startup-time", "Mechanics", "First-use warning", "Dockerfile shape", "Explicit non-goals"); Container specifics § "Networking" container-side subsections ("Per-session IP layout", "Flags and phases" table — bridge, IP, DNS, default-route invocation), "Hardening", "Workspace", "Per-session home volume", "Resource defaults — container only", "Lifecycle"; Capabilities model § "CLI learns capabilities via `GET /backends`" (endpoint side only — CLI consumption deferred); Persistence § "Handle persistence" (container row); Testing § "Integration tests"; Rollout § "Phase 2 — container runtime (feature-flagged off)" (container parts only — no CLI surface yet, test-only via direct API).

**Rationale.** Lands the container backend end-to-end behind the abstraction from S1, consuming the security primitives from S2. Image build, runtime impl, transport, hardening defaults, networking-and-route-installation lifecycle, resource defaults, home-volume β semantics, and the `GET /backends` endpoint are one cohesive deliverable: the daemon can create container sessions via direct API call, with policy / gateway / CA / propagation paths reused unchanged from Lima. CLI surface is deliberately deferred to S4 so this PR's review surface stays bounded to the runtime, the image, and the discovery endpoint.

**In scope.**

- Architecture § "Two implementations" (Container) — `ContainerRuntime` + `ContainerTransport` in `sandbox-core/src/backend/container.rs`. Transport wraps `docker exec <container> socat - TCP:127.0.0.1:5123`, mirroring Lima's `limactl shell <vm> -- socat - TCP:127.0.0.1:5123` exactly. The `sandbox-guest` agent binds TCP on `127.0.0.1:5123` in both backends; no agent-side changes.
- Image building § "Mechanics" — `ensure_image()` serialized by `container_image_lock: Mutex<()>`, sibling to the existing `base_image_lock`; image tag `sandboxd-lite:<daemon-version>`; build context staged at `{runtime_dir}/images/lite/`; Dockerfile baked into `sandboxd` via `include_str!` and written out at staging time; `sandbox-guest` binary copied from `{exe_parent}/sandbox-guest` into the staging dir (same convention Lima already uses); `docker build -t sandboxd-lite:<ver>` against the staged dir. **No** `build.rs` cross-workspace embedding, **no** binary bloat in `sandboxd`.
- Image building § "First-use warning" — daemon emits `lite: first use on this daemon version — building lite image` once per first-use rebuild; surfaced via the HTTP create response so the CLI (S4) can echo it verbatim. Subsequent creates on the same daemon version skip the warning entirely.
- Image building § "Dockerfile shape" — Ubuntu 24.04 (noble) base; package list `bash coreutils git socat ca-certificates iproute2 curl tini`; non-root `agent` uid 1000 / gid 1000; `tini` as PID 1; `ENTRYPOINT ["/usr/bin/tini", "--", "/usr/local/bin/sandbox-guest"]`. No flags or paths beyond what the spec prescribes.
- Container specifics § "Flags and phases" — `--network sb-{session_id}` (per-session bridge from `NetworkManager`); `--ip` from the bridge subnet (`.3`); `--dns` set to gateway container's bridge IP (`.2`).
- Container specifics § "Hardening" table — every default applied at `docker create` time: `--read-only`, `--tmpfs /tmp` (`rw,nosuid,nodev,size=256m`), `--tmpfs /run` (`rw,nosuid,nodev,size=16m`), `--security-opt no-new-privileges`, `--security-opt seccomp=default`, `--cap-drop=ALL`, `--user 1000:1000` (or calling uid/gid when host uid ≠ 1000), `--pids-limit 512`, `--memory` and `--cpus` from resource defaults, `--restart no`. Operators cannot relax these.
- Container specifics § "Workspace" — bind mount `/host/path → /workspace`; `--user <host-uid>:<host-gid>` when host uid is not 1000 so files written inside the container are owned by the operator. No userns-remap.
- Container specifics § "Per-session home volume" — `/home/agent` lives in `sandbox-home-{session_id}` named Docker volume; survives `stop` / `start`, deleted with `sandbox delete`. The β middle-ground that matches what a Lima session offers (state within a session, clean slate between sessions).
- Container specifics § "Resource defaults — container only" — `memory_mb` default `host_ram × 0.8` rounded down to whole MB; `cpus` default `host_cpus × 0.8` rounded to 1 decimal place. Computed once at daemon startup. Treated as ceilings — multiple concurrent lite sessions share the 80% envelope. Lima defaults unchanged.
- Container specifics § "Lifecycle" — `create`/`start`/`stop`/`delete`/`exec_interactive` shapes per the spec; `start` invokes `sandbox-route-helper <container-pid> <gateway-ip>` between `docker start` and the agent-ready TCP probe (`docker inspect -f '{{.State.Pid}}'` to read PID, helper invocation, helper exit-code check, then proceed). `--cap-drop=ALL` envelope is absolute — `CAP_NET_ADMIN` is never inside the container at any point.
- Container specifics § "Networking → Timing invariant" — verify that the agent does no outbound I/O before the daemon signals ready; `docker exec` readiness probe does not traverse the container's default route, so the pre-helper window is benign.
- Capabilities model § "CLI learns capabilities via `GET /backends`" — endpoint implemented and serves `[{"kind": "lima", "capabilities": {...}}, {"kind": "container", "capabilities": {...}}]`. Authoritative source; CLI consumption lands in S4.
- Persistence § "Handle persistence" — `RuntimeHandle` for container derives from `sandbox-{session_id}`; `docker inspect` recovers state on daemon restart.
- `AppState.runtimes` registers `BackendKind::Container` alongside Lima. Daemon dispatches container-row sessions to the container runtime end-to-end.

**Explicitly deferred.**

- All CLI surface (`--lite`, `--backend`, capability errors, isolation warning, list/inspect, config file precedence, `rebuild-image --backend`, `--no-cache` rejection) → M11-S4.
- User-facing `docs/lite.md` and "What this breaks" docs → M11-S4.

- Orphan cleanup on daemon start → M11-S5.

**Exit criteria.**

- `cd sandboxd && cargo nextest run --workspace` green; `make test-integration` green — including new `integration_*` tests for `ContainerRuntime` lifecycle (`create`/`start`/`stop`/`delete` round-trip against real Docker), `ensure_image()` (first build when missing, no-op when tag present, rebuild on daemon-version bump), and `GuestTransport` for container (round-trip the agent JSON protocol end-to-end: ping, trivial exec, file upload). Tests use the `integration_` prefix and the integration nextest profile per the convention in `sandbox-core/tests/validators.rs`; `TestContainer` RAII wrappers reused.

**In scope.**

- Testing § "E2E tests" (parametrization) — `@pytest.fixture(params=["lima", "container"]) def backend(request): return request.param` applied to backend-agnostic existing tests; Lima-specific tests (nested-virt checks, `--hardened` behavior) gain `@pytest.mark.skipif(backend != "lima", reason="VM-only feature")`.
