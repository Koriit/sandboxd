# M21 — macOS robustness & lite mode

The third milestone of the macOS support arc. M20 proved a single full-backend session boots with enforced egress; M21 makes macOS *robust* and adds the second backend. The single gateway Lima instance hosts all gateway containers, so its crash is the only cross-session availability failure in the architecture and needs explicit recovery; the per-operator agent is a long-lived process with no Linux analog, so its failure modes and the fast-user-switching edge cases need isolation-preserving handling; the fixed-at-creation NIC count means pool resize is a privileged, daemon-mediated operation. M21 also lands the lite (container) backend on macOS — the SSH-/headless-friendly path that needs no agent, no GUI session, and no socket_vmnet slot — and reuses the landed M13/M14/M17 infrastructure with the macOS substitutions (peer-cred identity, version equality, the macOS `sandbox doctor` rows + `/diagnostics` fields, workspace parity).

This is **not** the terminal milestone of the spec; no spec-delivery verification session here.

## M21-S1 — Gateway and socket_vmnet failure handling

**Entry criteria.** M20-S6 (a full-backend session boots with enforced egress).

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — § Failure handling → Gateway Lima instance crash; socket_vmnet process failure; Individual gateway container crash; Slot leak detection.

**Rationale.** The single gateway Lima instance hosts all gateway containers, so its crash is the only cross-session *availability* failure in the architecture and needs an explicit recovery path; this session implements the health poll, the transient `networking_status` (Degraded → Recovering → Healthy), and per-session recovery. socket_vmnet daemons are launchd-`KeepAlive`-supervised, so the daemon's job is to surface crash-loops and tolerate transient dark segments, not to re-supervise. These belong together as the platform's availability-failure story.

**In scope.**

- Implement gateway-crash detection + recovery per § Gateway Lima instance crash (poll, Degraded tag, force-stop/restart, 90s readiness, per-session container/network re-create + DNAT re-inject, per-session `Error` on failure), with `networking_status` as an in-memory / DTO-only field (no schema change).
- Implement socket_vmnet failure handling per § socket_vmnet process failure (lean on launchd `KeepAlive`; surface crash-loops; transient dark segment recovers with static addressing) and individual gateway-container crash per § Individual gateway container crash.
- Implement runtime slot-leak detection per § Slot leak detection (tie reclamation to VM-liveness reconcile).

**Explicitly deferred.**

- Agent-crash / fast-user-switch reconcile → M21-S2.
- Surfacing `networking_status` in `sandbox doctor` / `inspect --verbose` → M21-S5 (this session defines the field; S5 wires the diagnostic surface where `inspect` does not already show it).

**Exit criteria.**

- Integration / e2e tests cover gateway-crash recovery (sessions go Degraded → Recovering → Healthy; a session that fails recovery goes `Error` while others continue) and slot reclamation on a confirmed-dead VM.
- The recovery path is idempotent (tolerates not-found at every step); full suites green.
- Code review approved; the recovery sequence documented at its site.

## M21-S2 — Per-operator agent failure modes and fast user switching

**Entry criteria.** M21-S1.

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — § Failure handling → Per-operator agent unavailable (macOS-only); Fast user switching, re-login, and duplicate check-in (macOS-only); § Privilege model → Login-session constraints.

**Rationale.** The per-operator agent is a long-lived process with no Linux analog, so its failure modes — not-checked-in, mid-session crash + respawn, and the fast-user-switching edge cases — need explicit, isolation-preserving handling. The subtle correctness requirements (last-connection-wins eviction, switched-away operators still running, stale-uid via heartbeat + fresh-check-in-driven reconcile, `GuestSocat` re-establish on eviction) are where a naive implementation would either hang or mis-route work across uids. None of this may ever leak across operators.

**In scope.**

- Implement agent-unavailable handling per § Per-operator agent unavailable (actionable error for full-backend ops; lite unaffected; agent-crash respawn → reconcile that operator's sessions; no cross-operator impact).
- Implement the fast-user-switching edge cases per § Fast user switching: duplicate check-in last-connection-wins (with the `GuestSocat` stream severed → reconnected), switched-away-but-running operators not treated as failed, stale-captured-uid handled via heartbeat + fresh-check-in-tied VM-side reconcile.
- Tie VM-side reconcile (startup step 6b) to a fresh check-in event, not a possibly-stale long-lived connection, per § Stale captured uid / uid recycle.

**Explicitly deferred.**

- The `check_operator_agent` doctor row → M21-S5.

**Exit criteria.**

- Integration tests cover: actionable error when the agent is absent; lite ops succeed without an agent; duplicate-uid eviction drops and reconnects the `GuestSocat` stream; a switched-away operator's running session is not marked failed; reconcile fires on fresh check-in.
- **No cross-operator state crossing under any of these paths — a mandatory multi-uid test** (not "where feasible"); full suites green.
- An **interim security-review pass** of the eviction / stale-uid / last-writer-wins paths: this authentication-and-isolation logic lands *after* the M19-S6 broker review and must not ride unreviewed to M23-S1. Findings fixed in-session.
- Code review approved; the login-scoped lifetime documented at the reconcile site.

## M21-S3 — Pool resize handling

**Entry criteria.** M21-S2.

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — § Pool resize handling; § socket_vmnet pool → Initialization (privileged vmnet daemon regen); § Session lifecycle → NIC interface naming (defers here on mismatch).

**Rationale.** The gateway instance's NIC count is fixed at creation, so changing `max_macos_sessions` requires rebuilding the gateway and regenerating the root-owned vmnet daemon set — a privileged operation the unprivileged daemon cannot perform alone. This session makes the daemon the single authority on a config-vs-actual NIC mismatch: rebuild when safe, otherwise continue with the discovered count and surface the sudo-required resize path. It also makes "the daemon always starts" robust to a stale/changed config.

**In scope.**

- Implement the resize procedure per § Pool resize handling: detect the NIC-count mismatch at startup, rebuild the gateway when no sessions run and the vmnet daemon set already matches N, otherwise warn-and-continue with the discovered count.
- Make this handler the single authority the NIC-discovery step defers to (no hard-fail on mismatch) per § NIC interface naming → defers here.
- Surface the privileged vmnet-daemon-set regeneration requirement (daemon refuses the daemon-side rebuild until the sudo-driven plist regen has matched N) per § socket_vmnet daemon set must be regenerated too.

**Explicitly deferred.**

- The actual sudo install step that regenerates the vmnet plist set → M22-S2 (this session detects + guides; install owns the privileged regen).

**Exit criteria.**

- Integration tests cover: rebuild when no sessions + matching daemon set; warn-and-continue with running sessions; refuse the daemon-side rebuild when the vmnet daemon set does not yet match N.
- The daemon always starts under every mismatch case; full suites green.
- Code review approved; the resize authority documented at its site.

## M21-S4 — Lite-mode container backend on macOS

**Entry criteria.** M21-S3.

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — § M11 — lite-mode container backend on macOS (Architecture, Pool slot allocation, ContainerRuntime platform-awareness, Performance trade-off, Workspace modes for lite, Differences in the session lifecycle); § socket_vmnet pool → Concurrency.

**Rationale.** Lite mode is the SSH-/headless-friendly backend (no agent, no GUI session, no socket_vmnet slot) and is the only full workflow available to SSH-only operators on macOS, so it must work; it hosts sandbox containers inside the gateway Lima instance over the forwarded Docker socket, reusing the proven Linux container-backend interception path byte-for-byte. The key correctness points are that lite consumes a session-count token but no NIC slot, uses an internal Docker bridge (no macvlan/socket_vmnet), and routes via the in-VM `sandbox-route-helper`. The `shared:` TCC asymmetry (host-side access by `_sandbox`, not the operator) is a real lite-vs-Lima difference to implement and document.

**In scope.**

- Make `ContainerRuntime` macOS-aware per § ContainerRuntime platform-awareness (`is_macos`, `docker_socket` → forwarded gateway socket on every `docker` call; `docker_home` HOME/DOCKER_CONFIG pinning to base-dir).
- Implement the lite create/stop/start/rm flow per § Differences in the session lifecycle (internal per-session Docker bridge in the gateway Lima VM, two-NIC gateway container, sandbox container `--user op:gid`, in-VM route-helper repoint, no macvlan/socket_vmnet) and the egress path per § Architecture.
- Implement lite pool accounting per § Pool slot allocation and § Concurrency (session token only, no NIC) and lite workspace modes per § Workspace modes for lite (`shared:` double-hop + `_sandbox` TCC asymmetry caveat, `local:`, `clone:`). The lite `shared:` host exposure via Lima's own 9p `mounts:` stanza is added to the gateway Lima template **at install/template time, not per-session** — own that gateway-template touch here, coordinated with M20-S2's template generation.

**Explicitly deferred.**

- Tart / install-e2e coverage of lite on macOS → M22-S8.

**Exit criteria.**

- E2E (container cell, macOS) covers a lite session create/use/stop/rm over the forwarded gateway socket, egress gated through the gateway, and an SSH-only (no GUI) operator using lite end-to-end.
- Integration tests confirm lite consumes a session token but no NIC slot (mixed Lima + lite under N), and that the `shared:` host-side access runs as `_sandbox`.
- Code review approved; the lite-vs-Lima `shared:` difference documented in user docs.

## M21-S5 — M13/M14 integration: identity, version, doctor, diagnostics

**Entry criteria.** M21-S4.

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — § M13 — per-caller API session isolation; § M14 — version handling and `sandbox doctor` (incl. the consolidated checks table) and `/diagnostics`; § M17 — workspace modes and workspace-lock; § Disk usage (`check_disk_space`).

**Rationale.** macOS reuses the landed M13/M14 mechanisms with substitutions, and this session lands the diagnostic and identity surface operators rely on to triage a macOS install: the full set of macOS `sandbox doctor` rows (and cfg-gating the Linux-only ones), the macOS `/diagnostics` fields, and confirmation that peer-cred identity capture and version equality work unchanged. It also confirms M17 workspace parity, which the QEMU+9p decision makes essentially zero-net-work. Consolidating these here keeps the operator-facing diagnostic surface coherent.

**In scope.**

- Confirm / adjust per-caller identity per § M13: `peer_cred()` works on both platforms; `getgrouplist` is the authoritative membership gate; the `NGROUPS`/DAC note holds (no `0666` fallback). Version equality on connect is platform-independent (§ M14).
- Implement the macOS `sandbox doctor` rows and the Linux-only cfg-gating per the § M14 consolidated checks table (socket_vmnet installed/running, gateway lima instance, vmnet pool slots, gateway image inside lima, operator agent, lima version, socket_vmnet access, lima base image, disk space) and the macOS `/diagnostics` Option-typed fields.
- Confirm M17 workspace parity per § M17 (9p on both platforms, `local:` rsync over Lima SSH, workspace-lock platform-independent) — implement only the deltas, document the rest as verified-no-work.

**Explicitly deferred.**

- `check_tcc_workspace_access` real-chain behaviour (the informational verdict) → finalized in M22-S9 (this session lands the row; S9 settles the TCC attribution).
- The `check_gateway_image_inside_lima` self-heal trigger → M22-S5.

**Exit criteria.**

- Integration tests cover the macOS doctor rows' pass/skip behaviour under `cfg!(target_os)` gating and the `/diagnostics` macOS fields; `sandbox doctor` is all-green on a healthy dev macOS install.
- Workspace-lock + `local:`/`shared:` Lima-mode tests pass on macOS with no macOS-specific lock code; full suites green.
- Code review approved; the doctor checks table reflected in operator docs.
