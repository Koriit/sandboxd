# M15 — Release & install infrastructure

Spec 4 of the five-spec arc: the GitHub Actions release pipeline that signs and publishes sandboxd artifacts, and the `install.sh` / `uninstall.sh` scripts that put those artifacts onto an operator's host. The daemon-side deployment shape (the `sandbox` system user, the systemd unit, the state-dir layout, `sandbox doctor`) is fully specified by Spec 3 (M14) and merely consumed here — Spec 4 is the delivery mechanism, not the shape decision. M15 also ships the Lima-based E2E harness that exercises `install.sh` + `uninstall.sh` on real Linux VM distros, closing the `integration_systemd_unit_smokes` coverage deferred from M14. M15 is the terminal milestone for Spec 4.

**Spec references:**

- [`../specs/2026-05-11-release-and-install-infrastructure-design/2026-05-11-release-and-install-infrastructure-design-spec.md`](../specs/2026-05-11-release-and-install-infrastructure-design/2026-05-11-release-and-install-infrastructure-design-spec.md) — M15-S1..S3, M15-S4..S5

## M15-S1 — GitHub Actions release workflow + CI shellcheck

**Entry criteria.** M14 complete (Spec 3 — `sandboxd/contrib/systemd/sandboxd.service` artifact must exist for the tarball to include; `GATEWAY_VERSION`-tagged `make gateway-image` target must be in place from M14-S1 for the release step to tag correctly).

**Spec reference.** `../specs/2026-05-11-release-and-install-infrastructure-design/2026-05-11-release-and-install-infrastructure-design-spec.md` — §§ 2.1–2.4 (tarball naming, layout, MANIFEST format, size sanity), §§ 3.1–3.6 (GH Actions trigger, arch matrix, steps verbatim, pinned versions table, permissions, SLSA build provenance), § 8.2 (shellcheck step in `ci.yml`), § 12 (implementation notes: `release.yml`, `ci.yml` edit).

**Rationale.** The release pipeline and its output format (tarball layout, MANIFEST schema, cosign verification command) are the contract that `install.sh` (M15-S2) reads from the other end. Writing the pipeline first means `install.sh` can be written against a defined artifact structure rather than specifying it in parallel. The shellcheck step ships here because it lints the scripts that M15-S2 will write — getting the linter in CI before the scripts exist means the first script commit is linted from day one.

**In scope.**

- `.github/workflows/release.yml` per Spec 4 §§ 3.1–3.5: `push.tags` trigger, two-arch matrix (`x86_64-unknown-linux-gnu` on `ubuntu-22.04`, `aarch64-unknown-linux-gnu` on `ubuntu-22.04-arm`), `fail-fast: false`, per-arch job steps verbatim (checkout, version-resolve + Cargo.toml sanity check, Rust toolchain pin at `1.88.0`, cargo cache, `cargo build --workspace --release`, `make gateway-image` + `docker save`, tarball assembly with the exact Python MANIFEST generator from § 3.3, cosign install at `v2.4.1`, `cosign sign-blob --yes`, `actions/attest-build-provenance@v1`, `softprops/action-gh-release@v2`), permissions block (`contents: write`, `id-token: write`, `attestations: write`).
- Tarball layout per Spec 4 § 2.2: `bin/sandboxd`, `bin/sandbox`, `bin/sandbox-route-helper`, `libexec/` (empty), `systemd/sandboxd.service`, `images/sandbox-gateway-<ver>.tar`, `attestations/*.sigstore`, top-level `MANIFEST`.
- MANIFEST JSON per Spec 4 § 2.3: required keys (`version`, `arch`, `build_sha`, `build_time`, `artifacts` map with `path` + `sha256` per entry; `gateway-image` additionally has `docker_tag`).
- `publish-install-script` top-level job that runs after both matrix jobs succeed; publishes `scripts/install.sh` + `scripts/uninstall.sh` to GitHub Pages via the existing `docs.yml` pipeline shape per Spec 4 § 4.1 (or documents the convention for M15-S2's docs.yml edit).
- Shellcheck step in `.github/workflows/ci.yml` per Spec 4 § 8.2: `shellcheck -s sh -S style scripts/install.sh scripts/uninstall.sh` (scripts do not yet exist; the step is added now so it runs when they appear in M15-S2).

**Explicitly deferred.**

- `scripts/install.sh` and `scripts/uninstall.sh` → M15-S2.
- Lima E2E workflow (`.github/workflows/install-e2e.yml`) → M15-S3.

**Exit criteria.**

- `.github/workflows/release.yml` passes `act` (or equivalent local workflow dry-run) for a synthetic tag push — both arch jobs execute without errors, tarball and MANIFEST are produced.
- MANIFEST produced by the Python generator contains all required fields and the per-artifact sha256 values match `sha256sum` of the staged files.

## M15-S2 — `install.sh` + `uninstall.sh` + GitHub Pages hosting

**Entry criteria.** M15-S1 (the tarball layout and MANIFEST schema must be finalized so `install.sh` can reference them; the shellcheck CI step must be in place so the first commit of the scripts is immediately linted).

**Spec reference.** `../specs/2026-05-11-release-and-install-infrastructure-design/2026-05-11-release-and-install-infrastructure-design-spec.md` — §§ 4.1–4.7 (install.sh: hosting, invocation patterns, idempotency principle, step-by-step flow §§ 4.4.1–4.4.24, install-state file, log format, color/TTY), §§ 5.1–5.2 (uninstall.sh: flags, step-by-step flow §§ 5.2.1–5.2.12), § 7 (trust bootstrap), § 9 (backward compatibility — dev mode), §§ 10.3–10.8 (risks and open questions, to be addressed in code or docs), § 12 (implementation notes: `scripts/install.sh`, `scripts/uninstall.sh`, `site/public/` hosting, `docs.yml` edit), `docs/start/installation.md` update.

**Rationale.** `install.sh` and `uninstall.sh` are mirror images of each other — uninstall reads the install-state file that install writes and precisely undoes what install recorded. A reviewer who sees them together can verify the symmetry (every "we created this" state bit in install has a corresponding "remove only if we created it" guard in uninstall). They are 500-600 and 300-400 lines respectively — large but coherent, and the idempotency pattern across all 24 + 12 steps is the structural invariant that keeps them reviewable as one concern.

**In scope.**

- `scripts/install.sh` per Spec 4 §§ 4.2–4.7: POSIX shell (`-s sh` for shellcheck), ~500-600 lines; all 24 steps in § 4.4 verbatim (arg parsing, OS/arch detect, TTY/color, pre-existing-install detection, prereq check, disk check, cosign bootstrap with pinned `v2.4.1` + embedded sha256 constants, tarball fetch via curl, sigstore verification via cosign, extraction + MANIFEST verify, create `sandbox` system user, add invoking operator to group, install binaries via `install -D`, setcap on route-helper, qemu-bridge-helper probe, setuid, install `/etc/qemu/bridge.conf`, install `/etc/sandboxd/users.conf`, docker load gateway image, install systemd unit, `systemctl daemon-reload`, write `/var/lib/sandbox/.install-state.json`, print next-steps). All steps are idempotent per § 4.3 pattern.
- `scripts/uninstall.sh` per Spec 4 §§ 5.1–5.2: POSIX shell, ~300-400 lines; flags (`--purge`, `--force`, `--yes`, `--verbose`, `--quiet`, `--no-color`); 12 steps (arg parse, refuse-if-active-sessions, read install state, stop+disable unit, remove unit, revert bridge-helper setuid, remove bridge.conf rules, remove users.conf with sha256-based backup if modified, defer helper caps, remove binaries, `--purge` block, final report).
- `/var/log/sandbox-install.log` format per Spec 4 § 4.6 (both scripts write to the same file; install.sh creates it with `0640 root:root` on first run).
- `site/public/install.sh` and `site/public/uninstall.sh` linked or copied per Spec 4 § 4.1 (symlinks if Astro follows them; build-time copy step in `docs.yml` if not — implementation discretion per § 4.1).
- `docs.yml` edit to copy/link scripts into Pages output per Spec 4 § 4.1.
- `docs/start/installation.md` update per Spec 4 § 12: two-path structure ("Operator install (curl|bash)" + "Developer install (make setup-dev-env)"); replace the current dev-mode-only instructions with the operator path and a pointer to the dev path.

**Explicitly deferred.**

- `/etc/profile.d/sandbox.sh` for `SANDBOX_SOCKET` → deferred per Spec 4 § 10.7 (document as follow-up, do not implement).
- logrotate snippet → deferred per Spec 4 § 10.4.
- Lima E2E harness that exercises the scripts → M15-S3.

**Exit criteria.**

- `shellcheck -s sh -S style scripts/install.sh scripts/uninstall.sh` passes without suppression directives.
- `install.sh --help` prints usage and exits `0`; `install.sh --version-that-does-not-exist` exits `2` (unknown flag).
- `install.sh` run on a fresh CI VM with a local tarball (`--from`, `--yes`, `--verbose`) exits `0`, creates all expected artifacts (`/usr/local/bin/sandboxd`, `sandbox`, `/usr/local/libexec/sandboxd/sandbox-route-helper`, `/etc/systemd/system/sandboxd.service`, `/var/lib/sandbox/.install-state.json`), and all steps log `action=install` or `action=create`.
- Second run with the same args exits `0` with all steps logging `action=skip`.
- `uninstall.sh --yes` after install exits `0`; binaries and unit are absent; `/var/lib/sandbox/` is preserved (no `--purge`).
- `uninstall.sh --purge --yes` removes `/var/lib/sandbox/` and (if we created it) the `sandbox` user.
- Install-state file matches the JSON schema in Spec 4 § 4.5 (all required keys present, `jq` parses without error).
- `/var/log/sandbox-install.log` exists with mode `0640` and owner `root:root` after install; the file contains at least one line per install step in the Spec 4 § 4.6 format (`<ISO8601> install.sh step=<name> ... status=ok`).
- `docs/start/installation.md` updated with two-path structure.
- Code review approved.

## M15-S3 — Lima E2E test harness for install/uninstall

**Entry criteria.** M15-S2 (`install.sh` and `uninstall.sh` must exist for the harness to test them; the tarball format from M15-S1 must be finalised so `build-local-tarball.sh` can replicate it).

**Spec reference.** `../specs/2026-05-11-release-and-install-infrastructure-design/2026-05-11-release-and-install-infrastructure-design-spec.md` — §§ 6.1–6.4 (Lima harness: why Lima, harness shape, test matrix 10 parametrised cases, CI integration via `install-e2e.yml`), §§ 8.1, 8.3–8.6 (test plan: Lima E2E as primary gate, idempotency unit tests, partial-failure recovery, uninstall double-run, air-gapped path).

**Rationale.** The Lima harness is where Spec 4's scripts gain end-to-end correctness proof on real distros — Ubuntu, Debian, Fedora each expose different qemu-bridge-helper paths, different OVMF firmware layouts, and different systemd journald defaults. Unit tests on the script logic alone would miss every path-resolution divergence. The harness also closes the `integration_systemd_unit_smokes` gap deferred from M14-S3, since this harness provides exactly the "Lima-controlled VM with systemd inside" environment that test requires.

**In scope.**

- `tests/install-e2e/` directory: `conftest.py` (Lima helper functions `lima_shell`, `lima_cp`, `wait_for_socket`; session-scoped `release_tarball_x86_64` fixture that invokes `build-local-tarball.sh`; distro-template helpers), `build-local-tarball.sh` (local-build tarball script mirroring the release workflow's `assemble tarball` step: `cargo build --workspace --release`, `make gateway-image`, `docker save`, tar), `requirements.txt` (pytest + dependencies), test files per Spec 4 § 6.3.
- Test cases per Spec 4 §§ 6.3, 8.3–8.6 (10+ parametrised cases):
  - `test_install_fresh_then_doctor_passes` (ubuntu-22.04, debian-12)
  - `test_install_fresh_then_doctor_passes_rhel_paths` (fedora-40; bridge-helper at `/usr/libexec/`)
  - `test_install_idempotent_double_run` (all steps log `action=skip` on second run)
  - `test_install_partial_failure_recovery` (kill at step N, re-run completes)
  - `test_uninstall_after_install_clean` (no-purge leaves `/var/lib/sandbox/` intact)
  - `test_uninstall_with_purge_removes_user_and_state` (full purge; `getent passwd sandbox` empty)
  - `test_uninstall_double_run_idempotent` (second uninstall all-skip)
  - `test_install_air_gapped` (network disabled mid-install; `--from` + `--cosign-bundle` path)
  - `test_install_refuses_wrong_arch_tarball` (aarch64 tarball on x86_64 VM → die with clear error)
  - `test_install_refuses_when_preexisting` (second install.sh run refuses, points at `sandbox update`)
- `.github/workflows/install-e2e.yml` per Spec 4 § 6.4: `push.paths` trigger on `scripts/install.sh`, `scripts/uninstall.sh`, `tests/install-e2e/**`; `runs-on: [self-hosted, kvm]`; 60-minute timeout; steps (checkout, setup-python 3.12, Rust toolchain 1.88.0, cargo cache, `build-local-tarball.sh`, run pytest with 900s timeout).
- The `integration_systemd_unit_smokes` test from M14-S3 (deferred to this session): uses the Lima VM to install the unit, verify `systemctl enable --now sandboxd` reaches `active (running)`, confirm `/run/sandbox/sandboxd.sock` exists with mode `0660`, confirm `sandbox doctor` exits `0`.

**Explicitly deferred.**

- Multi-arch (aarch64) harness tests (the `release_tarball_x86_64` fixture covers the primary arch; aarch64 harness on an ARM runner is a follow-up per § 10.2).
- Fuller air-gapped test (pre-staged cosign on a fully-offline host) per Spec 4 § 10.6 — the `test_install_air_gapped` case covers the mid-install network-block scenario; the fully-offline scenario is documented as known-partial in § 10.6.

**Exit criteria.**

- All 10+ Lima E2E test cases in § 6.3 pass locally on a KVM-capable host.
- `integration_systemd_unit_smokes` passes inside the Lima VM harness.
- `test_install_idempotent_double_run` parses the second-run install log and asserts every step's `action` field is `skip`.
- `test_install_partial_failure_recovery` demonstrates at least 2 failure-points and confirms re-run convergence.
- `.github/workflows/install-e2e.yml` YAML is valid (`actionlint` or `act --dryrun`); the `[self-hosted, kvm]` runner requirement is documented in the workflow file.
- Code review approved.

## M15-S4 — Multi-track review

**Entry criteria.** M15-S3 complete — all three implementation sessions landed; Lima E2E harness tests pass; shellcheck passes; CI workflows are valid.

**Spec reference.** `../specs/2026-05-11-release-and-install-infrastructure-design/2026-05-11-release-and-install-infrastructure-design-spec.md` (all sections).

**Rationale.** Spec 4 is the first milestone whose primary deliverables are shell scripts and CI YAML rather than Rust code. Shell scripts have different quality failure modes (POSIX-ism leakage, unquoted variables, race conditions between idempotency checks and state mutation, trust-chain breaks) that the six review tracks are specifically calibrated to catch. Findings are collated, triaged, and only confirmed blockers fixed before M15-S5.

**In scope.**

Six tracks, each delegated to a separate review agent in parallel:

- **Track 1 — Implementation vs. spec**: Re-read Spec 4 §§ 1–10 section by section; compare every concrete requirement (step order, log line format, state file field name, flag name, cosign verification command verbatim, MANIFEST field names) against `scripts/install.sh`, `scripts/uninstall.sh`, and `.github/workflows/release.yml`; flag any spec claim missing from code or implemented differently.
- **Track 2 — Code quality**: Review shell scripts for POSIX compliance (no bashisms past what shellcheck approves), variable quoting (especially around `$SUDO_USER`, `$operator`, path variables), `sudo -k` placement (every privileged action has its own `-k`), `die` usage on recoverable vs. non-recoverable errors, and idempotency correctness (every state-check path truly skips on second run). Review `release.yml` for pinned action versions, correct permission scope, and MANIFEST generator correctness.
- **Track 3 — Unit/script test quality**: For every test in `tests/install-e2e/`, verify assertions are observable and non-tautological (e.g., `test_install_idempotent_double_run` must actually parse the log, not just check exit code; `test_install_air_gapped` must actually block network egress, not just pass `--from`). Verify the partial-failure recovery tests inject real mid-script failures (e.g., `kill -9` the script at a known step) rather than simulated ones. Check that every Spec 4 § 6.3 test case has a corresponding test file or parametrization.
- **Track 4 — E2E / integration test quality**: For the Lima harness tests, verify assertions in each test cover real post-condition state (filesystem presence, file mode, `getent passwd` output, systemctl status, `getcap` output, `sandbox doctor` exit code and stdout tokens) — not just "the script exited 0". Verify the `test_uninstall_with_purge_removes_user_and_state` case checks both the user absence AND the state-dir absence AND the gateway image absence.
- **Track 5 — Docs quality**: Verify `docs/start/installation.md` accurately presents both install paths (operator curl|bash + dev make); check that the trust-chain walkthrough in § 7 is accurately referenced or linked; verify the cosign sha256 constants in `scripts/install.sh` match the actual values from `https://github.com/sigstore/cosign/releases/download/v2.4.1/cosign-linux-{amd64,arm64}.sha256`.
- **Track 6 — Workarounds + deprecated patterns**: Grep `scripts/install.sh` and `scripts/uninstall.sh` for: (a) any bashism that shellcheck's `-s sh` would flag if it appeared after the first commit (check the full file, not just the diff); (b) any hardcoded path that differs from what Spec 4's step-by-step prescribes (e.g., a hardcoded `/usr/lib/qemu/qemu-bridge-helper` reference that bypasses the probe in § 4.4.16); (c) any `TODO`/`FIXME` introduced in M15 sessions; (d) any missing `sudo -k` before a privileged action; (e) any `uninstall.sh` step that removes something it did not verify it created (i.e., a step that ignores the `we_*` install-state flags). For tests: flag any tautological test.

**Explicitly deferred.**

- Spec-delivery claim-to-code map → M15-S5.

**Exit criteria.**

- All six review tracks complete; findings collated into a single prioritized list.
- Every "must-fix" finding addressed, re-implemented where needed, and re-tested.
- Shellcheck passes after any script changes.
- Lima E2E harness tests still pass after any script changes.
- Code review approved.
