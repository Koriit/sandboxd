# M17 — Workspace ergonomics

The 9p `securityModel` of the shared-workspace mount is currently hardcoded to `mapped-xattr`, which trades real-symlink interop for keeping the QEMU process unprivileged. M17 makes the model a per-session choice exposed via `sandbox create --workspace shared:<path>[:<model>]` so users who need real symlinks in both directions can opt into `securityModel: none` without us loosening the hardening posture for everyone. The milestone is the terminal milestone for the 2026-05-14 spec; it ends with a six-track review and a dedicated claim-to-code delivery verification.

## M17-S1 — Implement configurable 9p `securityModel` + docs

**Entry criteria.** Fresh.

**Spec reference.** [`../../../.tasks/specs/2026-05-14-configurable-9p-security-model/2026-05-14-configurable-9p-security-model-spec.md`](../../../.tasks/specs/2026-05-14-configurable-9p-security-model/2026-05-14-configurable-9p-security-model-spec.md) — §§ Target design (CLI surface, Domain types, Parsing, Lima template, Container backend, Persistence, Verification), Docs.

**Rationale.** The feature is small in surface but fan-out-y in destructuring sites: every `match` arm on `WorkspaceMode::Shared` across the workspace has to gain the new field in the same diff or the build breaks. Splitting domain change from template/integration would leave a non-compiling intermediate, so the implementation lands as one PR. Docs ship in the same session because the feature is invisible without them: an operator scanning `--help` will see the new compact-form syntax, and the `none`-vs-default trade-off is what justifies the option existing at all — both belong in `docs/guides/workspaces.md` and `docs/guides/hardening.md` next to the existing 9p paragraphs.

**In scope.**

- `WorkspaceSecurityModel` enum and `WorkspaceMode::Shared { …, security_model }` field per Spec § Target design → Domain types (kebab-case serde, `Default = MappedXattr`, `Option<T> + #[serde(default, skip_serializing_if = "Option::is_none")]` for the persisted-blob rule).
- `WorkspaceMode::parse_flag` per Spec § Target design → Parsing (recognise trailing `:mapped-xattr` / `:none`, `rsplit_once(':')` + closed-enum + leading-`/` check, fall through to whole-string-as-path on any other suffix).
- Match-site fan-out: every destructure of `WorkspaceMode::Shared` across the workspace (mapper, lima, container, daemon main, integration tests, spec tests) updated to include the new field.
- Lima template interpolation per Spec § Target design → Lima template (`sandbox-core/src/lima.rs:1446-1466`, replace the hardcoded line; `cache: mmap` unchanged).
- Container-backend rejection per Spec § Target design → Container backend (fail fast with `SandboxError::InvalidConfig` or existing equivalent and a message naming `security_model`; `None` remains accepted).
- Unit tests enumerated in Spec § Target design → Verification (parser, template, persistence forward-compat, container rejection).
- Docs updates per Spec § Docs (`docs/guides/workspaces.md` syntax + when-to-use, `docs/guides/hardening.md` per-session model + non-exposure of `passthrough`/`mapped-file`).

**Explicitly deferred.**

- Six-track review of the change → M17-S2.
- Spec-delivery verification (claim-to-code map, example replay, out-of-scope grep) → M17-S3.
- Container-backend bind-mount equivalent of the option (spec § Out of scope).
- `sandbox describe` UX work for surfacing the chosen model (spec § Out of scope; falls out of `WorkspaceMode::Shared` serialisation organically but no dedicated UX).

**Exit criteria.**

- All unit tests listed in Spec § Verification (parser × 4, template × 3, persistence forward-compat × 1, container rejection × 1) pass under the default nextest profile.
- `cargo nextest run --workspace` default profile clean; `cargo nextest run --workspace --profile integration` clean.

- Manual example replay from Spec § Verification → Example replay completes for both default and `:none` invocations against a live Lima session: default produces a host-side regular file with `user.virtfs.symlink.target`; `:none` produces a host-side real symlink (`stat -c %F` reports `symbolic link`).
- `docs/guides/workspaces.md` updated with the new syntax + when-to-use paragraph; `docs/guides/hardening.md` updated with the per-session note + non-exposure rationale.

## M17-S2 — Multi-track review

**Entry criteria.** M17-S1 complete — feature landed, default-profile and integration-profile nextest passing, docs updated.

**Spec reference.** [`../../../.tasks/specs/2026-05-14-configurable-9p-security-model/2026-05-14-configurable-9p-security-model-spec.md`](../../../.tasks/specs/2026-05-14-configurable-9p-security-model/2026-05-14-configurable-9p-security-model-spec.md) (all sections).

**Rationale.** The change touches a security-relevant surface (the hardening posture) and a persistence-relevant surface (the `config_json` blob shape), and the parser carries a documented footgun in the compact form. Six review tracks — calibrated to this spec's actual surface, not M16's — catch failure modes that compile-and-test cannot: drift between spec text and implementation, parser corner cases the unit tests miss, docs that contradict the code, and any premature widening that creeps in (e.g. an inadvertent `passthrough` value sneaking into the enum).

**In scope.**

Six tracks, each delegated to a separate review agent in parallel:

- **Track 1 — Implementation vs. spec**: Re-read the spec § by §; compare every concrete claim (CLI shape, default value, exposed model set, container-backend rejection, persistence semantics, exact match between `WorkspaceSecurityModel` variants and the model tokens accepted by the parser, exact match between `as_yaml()` output and the YAML interpolated into the Lima template) against the implementation; flag any gap or divergence.
- **Track 2 — Code quality**: Review the touched files (`sandbox-core/src/session.rs`, `sandbox-core/src/lima.rs`, `sandbox-core/src/backend/container.rs`, every match-site updated for the new field) for: idiomatic Rust, no superfluous clones in the parser, error-message clarity at the container-rejection site, no `unwrap`/`expect` in non-test paths around the new code, no accidental `pub` widening of the new enum's surface beyond what callers need. Confirm `WorkspaceSecurityModel` is `Copy` (it should be — single-byte unit-enum) and that the field name `security_model` is consistent across struct definitions, match patterns, JSON wire form, and YAML output.
- **Track 3 — Unit test quality**: For every unit test added in M17-S1: verify the assertion is non-tautological (the tested property is not reachable without the production code path). Specifically: parser tests must assert both the path payload and the `security_model` payload of the returned `WorkspaceMode::Shared`; template tests must assert the exact `securityModel: <token>` substring (not just that the YAML contains `9p:`); the persistence forward-compat test must construct a JSON blob without the `security_model` key (not one with `null`) and assert `Option::None` deserialises. Container-rejection test must assert the error message contains `security_model`.
- **Track 4 — Integration / E2E test quality**: Confirm the manual example replay from Spec § Verification was actually performed (artefacts recorded — Lima YAML excerpt, host-side `stat -c %F` output, host-side `getfattr -d` output for the default case). If the E2E suite has a `test_shared_workspace*` file, check whether it would benefit from a parameterised case across both models; if not, document the absence as a known gap on M17-S3's plate.
- **Track 5 — Docs quality**: Verify `docs/guides/workspaces.md` documents the compact-form syntax accurately (including the documented footgun for paths ending in `:none` or `:mapped-xattr`), explains when to pick `none`, and points at the trade-off (silent no-op of guest-side chown/mknod/setuid). Verify `docs/guides/hardening.md` notes the per-session model decision, that the default is unchanged, that `none` does not alter the QEMU privilege model, and that `passthrough`/`mapped-file` are deliberately not exposed with rationale. Verify the docs do not contradict the spec on any of these points.
- **Track 6 — Workarounds + deprecated patterns**: Grep the touched files for: (a) any new `unwrap()`/`expect()` in non-test paths; (b) any `TODO`/`FIXME` introduced in M17-S1; (c) any place where `passthrough` or `mapped-file` accidentally appears as a recognised model token (CLI parser, enum, YAML interpolation, docs sample); (d) any milestone tag (`M17` / `S1`) embedded in code or test comments per CLAUDE.md's "no milestone tags in code or tests" convention.

**Explicitly deferred.**

- Claim-to-code map + example replay write-up + out-of-scope grep → M17-S3.

**Exit criteria.**

- All six review tracks complete; findings collated into a single prioritised list.
- Every "must-fix" finding addressed, re-implemented where needed, and re-tested.
- `cargo nextest run --workspace` default and integration profiles clean post-fixes; `cargo clippy --workspace` clean; `cargo fmt --check` clean.
