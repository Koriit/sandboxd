# M17 â€” Workspace ergonomics

The Lima 9p mount and the container bind-mount that back `shared:` are currently double-hardcoded: the 9p `securityModel` is pinned to `mapped-xattr` (trading real-symlink interop for keeping the QEMU process unprivileged) and the guest mount point is uniformly remapped to `/home/agent/workspace` regardless of the operator's host path. The two backends also advertise only `Shared` and `Clone` workspace modes â€” there is no host-snapshot mode, so operators who want a one-time `rsync`-shaped seed without a long-running 9p surface fall back to `sandbox cp`/`sandbox sync` against a freshly-created session. Once a `local:` host-snapshot mode exists, operator-driven `sandbox workspace push` / `pull` becomes the natural way to move edits across the host/guest boundary on demand â€” and those operations need a daemon-side mutual-exclusion primitive so that two concurrent pushes (or a push interleaved with `sandbox stop`) cannot race. M17 closes all four gaps under one umbrella:

1. **`shared:` `securityModel` becomes a per-session choice** (`mapped-xattr` default, `none` opt-in for real-symlink interop both directions) â€” carried forward from the 2026-05-14 spec.
2. **`shared:` guest path becomes configurable** with a breaking-change default of *preserving the host path* inside the guest (so `compile_commands.json`, stack traces, and `pwd`-aware build caches survive a hostâ†”guest round-trip).
3. **A new `local:` workspace mode** â€” rsync-based host-snapshot at session create plus operator-driven `sandbox workspace push` / `pull`. No 9p or bind-mount surface; no live host writes; works for non-git trees; no network policy carve-out needed.
4. **A workspace-lock subsystem** â€” daemon in-memory per-session lock that serialises push and pull, atomically state-gates workspace ops at acquire time (collapsing the prior "check state then act" race window), and refuses `sandbox stop` / `sandbox delete` against a session with an active workspace op. New `sandbox workspace unlock --force` subcommand recovers orphan locks left by a crashed CLI; locks reset to `Unlocked` on daemon restart by design (no on-disk state).

The CLI grammar grows in lockstep:

```
sandbox create --workspace shared:<host>[:<guest>][:<security-model>]
sandbox create --workspace local:<host>[:<guest>] [--no-gitignore]
sandbox workspace push <session> {-f | -n} [--safe-links] [--no-gitignore]
sandbox workspace pull <session> {-f | -n} [--safe-links] [--no-gitignore] [--dest <path>]
sandbox workspace unlock <session> [--force]
```

The 2026-05-14 spec is superseded by the 2026-05-20 expansion; M17 is the terminal milestone for the new spec. Five sessions: S1 ships the `shared:` parser changes + guest-path + securityModel + breaking default + the DTO `workspace_mode_detail` surface; S2 ships `local:` mode core (variant, parser, daemon-side rsync orchestration at create, describe rendering, capability advertisement) on both backends; S3 ships the operator-driven push/pull CLI together with the workspace-lock subsystem they consume (acquire/release/force endpoints, lifecycle interaction with stop/delete, `unlock --force` CLI, orphan-lock recovery); S4 is the six-track review across S1+S2+S3; S5 is the spec-delivery verification.

## M17-S3 â€” Push/pull commands + workspace-lock subsystem

**Entry criteria.** M17-S2 complete â€” `local:` mode core landed, capability advertisement live on both backends, default-profile and integration-profile nextest passing, create-time push integration tests green.

**Spec reference.** [`../../../.tasks/specs/2026-05-20-m17-workspace-ergonomics/2026-05-20-m17-workspace-ergonomics-spec.md`](../../../.tasks/specs/2026-05-20-m17-workspace-ergonomics/2026-05-20-m17-workspace-ergonomics-spec.md) â€” Â§Â§ CLI shape (push / pull / unlock), Push/pull commands (full section), Workspace lock (full section: Goal, Lock state, API endpoints, Error mapping, CLI flow, New CLI command, Lifecycle interaction, Concurrency and races, Orphan locks, Persistence and serde), Tests (push/pull planner subset, workspace-lock unit + integration + E2E subset), Docs Changes (push/pull + orphan-lock recovery sub-sections).

**Rationale.** Push and pull are the operator-driven ops `local:` mode exists to enable; the workspace-lock subsystem is the daemon-side primitive that makes those ops safe to run concurrently and against a session that might also be stopped/deleted in parallel. The lock subsystem has its own self-contained surface (in-memory state machine, three API endpoints, dedicated DTOs, a new `unlock --force` CLI subcommand, lifecycle hooks on `stop`/`delete`, dedicated unit + integration + E2E tests) but is conceptually inseparable from the push/pull commands that drive it. Docs ship in the same session because the push/pull commands and the orphan-lock recovery flow are discoverability-zero without them â€” operators will not find `sandbox workspace unlock --force` from the `--help` output of an unrelated subcommand.
