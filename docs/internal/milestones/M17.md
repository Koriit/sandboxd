# M17 — Workspace ergonomics

The Lima 9p mount and the container bind-mount that back `shared:` are currently double-hardcoded: the 9p `securityModel` is pinned to `mapped-xattr` (trading real-symlink interop for keeping the QEMU process unprivileged) and the guest mount point is uniformly remapped to `/home/agent/workspace` regardless of the operator's host path. The two backends also advertise only `Shared` and `Clone` workspace modes — there is no host-snapshot mode, so operators who want a one-time `rsync`-shaped seed without a long-running 9p surface fall back to `sandbox cp`/`sandbox sync` against a freshly-created session. M17 closes all three gaps under one umbrella:

1. **`shared:` `securityModel` becomes a per-session choice** (`mapped-xattr` default, `none` opt-in for real-symlink interop both directions) — carried forward from the 2026-05-14 spec.
2. **`shared:` guest path becomes configurable** with a breaking-change default of *preserving the host path* inside the guest (so `compile_commands.json`, stack traces, and `pwd`-aware build caches survive a host↔guest round-trip).
3. **A new `local:` workspace mode** — rsync-based host-snapshot at session create plus operator-driven `sandbox workspace push` / `pull`. No 9p or bind-mount surface; no live host writes; works for non-git trees; no network policy carve-out needed.

The CLI grammar grows in lockstep:

```
sandbox create --workspace shared:<host>[:<guest>][:<security-model>]
sandbox create --workspace local:<host>[:<guest>] [--no-gitignore]
sandbox workspace push <session> {-f | -n} [--safe-links] [--no-gitignore]
sandbox workspace pull <session> {-f | -n} [--safe-links] [--no-gitignore] [--dest <path>]
```

The 2026-05-14 spec is superseded by the 2026-05-20 expansion; M17 is the terminal milestone for the new spec. Four sessions: S1 ships the `shared:` parser changes + guest-path + securityModel + breaking default, S2 ships `local:` mode end-to-end with push/pull, S3 is the six-track review across both, S4 is the spec-delivery verification.

## M17-S1 — `shared:` guest-path + `securityModel` + breaking default

**Entry criteria.** Fresh.

**Spec reference.** [`../../../.tasks/specs/2026-05-20-m17-workspace-ergonomics/2026-05-20-m17-workspace-ergonomics-spec.md`](../../../.tasks/specs/2026-05-20-m17-workspace-ergonomics/2026-05-20-m17-workspace-ergonomics-spec.md) — §§ CLI shape (shared variant), Domain types, Parser (shared variant), Backward Compatibility, Lima Backend (shared), Container Backend (shared), `sandbox describe`, Tests (shared subset), Docs Changes (shared subset).

**Rationale.** The shared-side changes are small in surface but fan-out-y in destructure sites: every `match` arm on `WorkspaceMode::Shared` across the workspace gains the new `guest_path` field (and the `security_model` field from the 2026-05-14 design) in the same diff or the build breaks. Splitting the domain change from the template/integration would leave a non-compiling intermediate, so the entire shared-mode surface lands as one PR. The breaking default change (`guest_path = host_path` instead of `/home/agent/workspace`) lands here too — it cannot be phased without confusing operators about which version they're on; the project is pre-0.1.0 and breaking defaults have been explicitly accepted. Docs ship in the same session because the new compact-form grammar and the trade-off between `mapped-xattr` and `none` are invisible without them — an operator scanning `--help` will see the new syntax, and the `none`-vs-default trade-off is what justifies the option existing at all.

**In scope.**

- `WorkspaceSecurityModel` enum and the `WorkspaceMode::Shared { host_path, guest_path, security_model }` shape per Spec § Domain types (kebab-case serde for the enum, `Default = MappedXattr`, `#[serde(default, skip_serializing_if = "Option::is_none")]` on the `Option<WorkspaceSecurityModel>` per the persisted-blob rule; `guest_path` with `#[serde(default)]` for legacy on-disk records).
- `WorkspaceMode::parse_flag` extended to the `shared:<host>[:<guest>][:<security-model>]` grammar per Spec § Parser: right-to-left token classification, strip-model-then-strip-guest, `~` expansion on both sides (host-side `~` against `std::env::home_dir`, guest-side `~` against `/home/agent`), absoluteness check, closed-enum tokens for `mapped-xattr` / `none`.
- Custom deserialiser shim per Spec § Backward Compatibility: legacy `Shared` records without `guest_path` recover with `guest_path = host_path` (aligning with the new default). No SQLite migration — the change is JSON-blob-only.
- Match-site fan-out: every destructure of `WorkspaceMode::Shared` across the workspace (mapper, lima, container, daemon main, integration tests, spec tests) updated to include the two new fields.
- Lima template interpolation per Spec § Lima Backend → shared mount block: `mountPoint` substituted from `guest_path`, `securityModel` substituted from the resolved model, `sanitize_yaml_path` applied to both host and guest paths (symmetric injection-prevention), `cache: mmap` unchanged.
- Container backend per Spec § Container Backend → shared bind-mount: `--mount type=bind,src=<host_path>,dst=<guest_path>` (the dst widens from the hardcoded `/home/agent/workspace`); rejection of `security_model: Some(_)` with `SandboxError::InvalidConfig` naming `security_model` retained from the 2026-05-14 design (no 9p layer for the container backend; `None` remains accepted).
- `sandbox describe` rendering of the `Workspace:` block per Spec § `sandbox describe` (shared and clone variants only — local lands in S2). Multi-line block with `Mode:` / `Host path:` / `Guest path:` / `Security:` keys; `Security:` renders `mapped-xattr (default)` when `security_model` is `None`.
- Unit tests enumerated in Spec § Tests (shared subset): parser matrix rows, `~` expansion both sides, relative-path rejection, empty-token rejection; backward-compat (legacy JSON without `guest_path` key, forward-compat round-trip with and without `security_model`); Lima template (default model, `none` model, custom guest path, both at once, YAML-injection probe); container backend argv (default path, custom guest path, `Some(_)` rejection); describe golden-form for shared and clone.
- Docs updates for the shared layer per Spec § Docs Changes: `docs/guides/workspaces.md` (the "Mount a host directory" section gains the guest-path paragraph and the existing `:<security-model>` paragraph, with the optional-token order documented as `:<guest-path>:<security-model>`); `docs/guides/hardening.md` (per-session model decision note + non-exposure of `passthrough` / `mapped-file` with rationale); `docs/internal/upgrade-notes.md` breaking-default note for the historical `/home/agent/workspace` operators.

**Explicitly deferred.**

- `local:` mode end-to-end → M17-S2.
- Six-track review of S1+S2 → M17-S3.
- Spec-delivery verification → M17-S4.
- Container-backend bind-mount support for guest paths outside `/home/agent` that survive the `--read-only` rootfs (Spec § Out of Scope; rsync/`docker create` errors at create time remain the diagnostic).
- A separate `--workspace-guest-path` / `--security-model` flag surface (Spec § Out of Scope; the colon-delimited grammar is the single surface).

**Exit criteria.**

- All shared-subset unit tests from Spec § Tests pass under the default nextest profile.
- `cargo nextest run --workspace` default profile clean; `cargo nextest run --workspace --profile integration` clean.
- `cargo clippy --workspace` clean; `cargo fmt --check` clean.
- Manual example replay (Lima): `sandbox create --workspace shared:/tmp/sbx-a:/srv/work:none` produces a guest mount at `/srv/work` with `securityModel: none` in the rendered YAML; guest-side `ln -s a b` in the mount round-trips as a real host symlink at `/tmp/sbx-a/b` (`stat -c %F` reports `symbolic link`). Default invocation (`shared:/tmp/sbx-default`) produces a host-side regular file with `user.virtfs.symlink.target` xattr (`getfattr -d` confirms).
- Manual example replay (container): `sandbox create --workspace shared:/tmp/sbx-a:/home/agent/work` records `--mount type=bind,src=/tmp/sbx-a,dst=/home/agent/work` in the daemon-emitted `docker create` argv (visible in daemon logs).
- `docs/guides/workspaces.md`, `docs/guides/hardening.md`, and `docs/internal/upgrade-notes.md` updated; code review approved.

## M17-S2 — `local:` mode end-to-end

**Entry criteria.** M17-S1 complete — shared layer landed, default-profile and integration-profile nextest passing, docs updated.

**Spec reference.** [`../../../.tasks/specs/2026-05-20-m17-workspace-ergonomics/2026-05-20-m17-workspace-ergonomics-spec.md`](../../../.tasks/specs/2026-05-20-m17-workspace-ergonomics/2026-05-20-m17-workspace-ergonomics-spec.md) — §§ CLI shape (local variant), Domain types (Local + WorkspaceModeKind::Local), Parser (local variant), `local:` Mode (full section), Push/Pull Commands, Lima Backend (local), Container Backend (local), `sandbox describe` (local), Tests (local subset, including integration + E2E), Docs Changes (local subset).

**Rationale.** `local:` mode is additive — new enum variant, new parser arm, new CLI subcommands, new daemon-side rsync orchestration at session-create. The parser surface from S1 is the foundation; the rsync orchestration and the workspace-aware CLI subcommands are isolated additions that do not destabilise S1's match-site fan-out. The integration and E2E tests require real backends (Lima VM, Docker container), so this session is heavier than S1 in test infrastructure but lighter in destructure fan-out. Docs ship in the same session because the new mode is discoverability-zero without them — operators will not find `sandbox workspace push` from the `sandbox --help` output of an unrelated subcommand.

**In scope.**

- `WorkspaceMode::Local { host_path, guest_path }` variant + `WorkspaceModeKind::Local` enum value per Spec § Domain types. The custom deserialiser shim from S1 extends to `Local` for consistency (no legacy records exist for the new variant, but the symmetric shape keeps the codepath uniform).
- `WorkspaceMode::parse_flag` extended to the `local:<host>[:<guest>]` grammar per Spec § Parser — shares the right-to-left token classifier with the shared grammar (no duplicated parsing logic); `~` expansion both sides; absoluteness check.
- `--no-gitignore` flag on `sandbox create` per Spec § `--no-gitignore` on `sandbox create`: top-level flag, meaningful only with `--workspace local:`; combined with `shared:` or `--repo` the daemon returns `InvalidConfig` naming `--no-gitignore`. Not persisted — governs the create-time push only.
- Daemon-side rsync orchestration per Spec § `local:` Mode and § Lima Backend (local) / § Container Backend (local): parent-dir creation inside the guest (`--mkpath` on rsync 3.2.3+, or `limactl shell sandbox-<id> -- mkdir -p` / `docker exec sandbox-<id> mkdir -p` as fallback); blocking initial push from `host_path` to `guest_path`; rsync invocation wrapped in `tokio::task::spawn_blocking` per CLAUDE.md async-handler convention; rsync stderr surfaced verbatim in the daemon's `InvalidState` / `Internal` response on failure; session record + VM/container torn down on failure (no orphan artefacts).
- Default rsync invocation per Spec § Default rsync invocation: `rsync -aL --delete --filter=':- .gitignore' -e <shell-transport> <src> <dst>` where the shell transport is `limactl shell` (Lima) or `docker exec -i` (container). `-L` swaps to `--safe-links` under `--safe-links`; `--filter` is dropped under `--no-gitignore`.
- `Capabilities::workspace_modes` for both backends advertises `Local` (Lima and container both gain the third variant; `EnumSet::all()` continues to express the full set).
- `sandbox workspace push <session> {-f|-n} [--safe-links] [--no-gitignore]` CLI subcommand per Spec § Push/Pull Commands.
- `sandbox workspace pull <session> {-f|-n} [--safe-links] [--no-gitignore] [--dest <path>]` CLI subcommand per Spec § Push/Pull Commands. `--dest` defaults to the session's recorded `host_path`; the dirname is `create_dir_all`-ed before the rsync spawn.
- Client-side state and mode checks (must be Running, must be `local:`) with the documented error messages and exit codes (1 for state, 2 for mode/usage). Daemon-side validation runs too on the happy path (defence in depth).
- `sandbox describe` rendering of the `Local` variant in the `Workspace:` block per Spec § `sandbox describe`: `Mode: local`, `Host path:`, `Guest path:`.
- Unit tests for the local-subset of Spec § Tests: parser matrix rows for `local:`, describe rendering for `Local`, push/pull planner full-argv assertions for both backends, container backend `Local` argv shape (no workspace `--mount`), capability-set assertions.
- Integration tests (`integration_*` prefix) per Spec § Tests → Integration tests: `integration_lima_local_create_and_push`, `integration_lima_local_pull`, `integration_container_local_create_and_push`, `integration_container_local_pull`, `integration_local_gitignore_filter`, `integration_local_create_failure_tears_down`, `integration_shared_guest_path_lima`, `integration_shared_guest_path_container`.
- E2E tests in `tests/e2e/test_workspace_local.py` and `tests/e2e/test_workspace_shared_guest_path.py` per Spec § Tests → E2E tests, both following the existing `tests/e2e/conftest.py` session-lifecycle pattern with `@pytest.mark.lima` / `@pytest.mark.container` per M12-S13's marker convention.
- Docs updates for the local layer per Spec § Docs Changes:
  - `docs/guides/workspaces.md` — new top-level "Snapshot a host directory (`local:` mode)" section between "Mount a host directory" and "Copy individual files with `sandbox cp`"; covers when-to-pick, `--no-gitignore` semantics, the `-f`/`-n` safety gate, the gitignored-`.env` filter-interaction note. The "`cp` vs. `sync`" table grows a row for `local:` push/pull.
  - `docs/guides/hardening.md` — new bullet under "Security trade-offs you choose": **`local:` snapshot.** No 9p surface, no live host writes; trade-off is staleness.
  - `docs/concepts/workspaces.md` — mode list grows from four to five (clone, shared, local, cp, git-remote); `local:` sits between `clone:` and `shared:` on the live-edits / isolation axis in the trade-off table.
  - `docs/internal/upgrade-notes.md` — rollback caveat for `local:` records (daemons predating M17 cannot deserialise `WorkspaceMode::Local`; the session is unloadable until rolled forward).

**Explicitly deferred.**

- Six-track review → M17-S3.
- Spec-delivery verification → M17-S4.
- Live two-way sync watcher for `local:` (inotify-driven push, filesystem-watcher daemon) — Spec § Out of Scope.
- Exposing rsync's `--include` / `--exclude` / `--filter` / `--info=progress2` as first-class flags on push/pull — Spec § Out of Scope; the fixed `--filter=':- .gitignore'` (or absent under `--no-gitignore`) is the complete filter surface.
- A `sandbox workspace status` subcommand — Spec § Out of Scope; `push -n` / `pull -n` already serve the diff need.
- Auto-detection of "this is a git repo, gitignore is fine" vs "this is not a git repo" — Spec § Out of Scope; rsync's `:- .gitignore` filter handles the no-gitignore case gracefully.

**Exit criteria.**

- All local-subset unit, integration, and E2E tests from Spec § Tests pass.
- `cargo nextest run --workspace` default profile clean; `cargo nextest run --workspace --profile integration` clean; `make test-e2e-matrix` clean (both backends).
- `cargo clippy --workspace` clean; `cargo fmt --check` clean.
- Manual example replay (Lima): `sandbox create --workspace local:/tmp/sbx-l` populates `/tmp/sbx-l` inside the guest at create time; `sandbox workspace push -f` after a host edit propagates the edit; `sandbox workspace pull -f` after a guest edit propagates back.
- Manual example replay (container): same shape against the container backend with `guest_path` inside `/home/agent` per the documented idiom for the `--read-only` rootfs.
- Manual example replay (filter): `local:` with a `.gitignore` excluding `excluded/` does not transfer that directory; the same source with `--no-gitignore` does.
- Docs landed and reviewed.

## M17-S3 — Six-track review (covering S1 + S2)

**Entry criteria.** M17-S2 complete — both shared and local layers landed, all tests green.

**Rationale.** The expanded M17 touches a security-relevant surface (the hardening posture via `securityModel`), a persistence-relevant surface (the `config_json` blob shape, with a custom deserialiser shim recovering legacy records), and a new orchestration code path (daemon-side rsync at session-create with blocking-failure tear-down). Six review tracks calibrated to this spec's actual surface — not M16's, not the 2026-05-14 spec's narrower scope — catch failure modes that compile-and-test cannot: drift between spec text and implementation, parser corner cases the unit tests miss, docs that contradict the code, premature widening (e.g. an inadvertent `passthrough` value sneaking into the enum), and `unwrap`/`expect` leaks in the new rsync orchestration path.

**In scope.**

Six tracks, each delegated to a separate review agent in parallel:

- **Track 1 — Implementation vs. spec.** Re-read the spec section-by-section; compare every concrete claim (CLI shape, default values, parser grammar table, persistence semantics, exact match between `WorkspaceSecurityModel` variants and accepted parser tokens, `as_yaml()` output vs YAML interpolated into the Lima template, container `--mount type=bind,src=...,dst=...` argv shape, push/pull argv shape, `sandbox describe` block format, capability set membership across both backends) against the implementation; flag any divergence. - **Track 5 — Docs quality.** Verify `docs/guides/workspaces.md` documents the three-token shared grammar accurately (including the path-with-colons footgun for `:none` / `:mapped-xattr` and `:<guest_path>`-shaped suffixes), explains when to pick `none`, and points at the trade-off (silent no-op of guest-side chown/mknod/setuid on `none`); the new `local:` section reads as a peer mode (not as an afterthought); the cp/sync/local-push/pull distinction is clear in the comparison table. Verify `docs/guides/hardening.md` covers both the per-session model decision (shared) and the no-9p-surface bullet (local); that the default is unchanged for `shared:`; that `passthrough` / `mapped-file` are deliberately not exposed with rationale; that `none` does not alter the QEMU privilege model. Verify `docs/concepts/workspaces.md` lists five modes. Verify `docs/internal/upgrade-notes.md` documents both breaking changes (shared default `guest_path`, `local:` rollback envelope). Verify the docs do not contradict the spec on any of these points.
- **Track 6 — Workarounds + deprecated patterns.** Grep touched files for: (a) any new `unwrap()`/`expect()` in non-test paths; (b) any `TODO`/`FIXME` introduced in S1+S2; (c) any place where `passthrough` or `mapped-file` accidentally appears as a recognised model token (CLI parser, enum, YAML interpolation, docs sample); (d) any milestone tag (`M17` / `S1` / `S2`) embedded in code or test comments per CLAUDE.md's "no milestone tags in code or tests" convention; (e) any hardcoded `/home/agent/workspace` that survived the shared-side guest-path migration; (f) any duplicated rsync orchestration between the daemon (create-time push) and the CLI (push/pull) that should share a planner.
