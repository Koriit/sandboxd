# M14 — Daemon productionization

Spec 3 of the five-spec arc: the deployment shape changes that Spec 4 will install and Spec 5 will operate on. Spec 3 is safe to implement only once M13 is complete (Spec 1's pair-check makes the dedicated `sandbox` user safe at the helper layer; Spec 2's per-caller filter makes it safe at the API layer). The changes here are all daemon-side: a hardened startup routine that enforces state-dir modes, version-pinned image references, the QEMU wrapper cleaned of its rootless-Docker branch, a strict CLI ↔ daemon version handshake, and the `sandbox doctor` diagnostic surface. The `sandbox` system user, systemd unit installation, and state-dir creation at `/var/lib/sandbox/` are Spec 4 territory — Spec 3 specifies their contracts and ships the unit file as a source artifact only. M14 is the terminal milestone for Spec 3.

**Rationale.** These changes share a single theme — hardening the daemon's deployment infrastructure without adding any new user-facing command. ## M14-S2 — `/version` endpoint + CLI version-equality check

**Entry criteria.** M14-S1 (`ensure_base_dir_layout` and `sessions.db` chmod must be in place so daemon startup does not fail before the `/version` route is reachable; `GATEWAY_IMAGE_REPOSITORY` + `gateway_image_tag_for_version` must exist so the daemon starts without a `missing image` hard failure during integration tests — the daemon still starts when the gateway image is absent, per Spec 3 § 8.5, but the constant must exist to form the tag).

**Spec reference.** `../specs/2026-05-11-daemon-productionization-design/2026-05-11-daemon-productionization-design-spec.md` — §§ 7.1–7.5 (the strict-equality rule, `/version` endpoint shape, error format, version constant location, bypass for `sandbox doctor` and `sandbox version`), § 7.6 (`sandboxd --version` output format pin), §§ 11.2–11.3 (unit tests — `/version` endpoint, CLI version-equality check), § 11.6 (integration tests — version endpoint real socket, CLI refuses on version skew).

**Rationale.** The CLI-daemon version handshake is a single, self-contained concern: one new route on the daemon side and one new guard in `send_request` on the CLI side. It is the lightest session in M14 and establishes the precondition that `sandbox doctor` (M14-S3) requires — doctor's C3 check calls `/version`, so the endpoint must exist before the doctor PR can be meaningfully tested.

**In scope.**

- `version_handler` returning `{"version": env!("CARGO_PKG_VERSION")}` + `.route("/version", get(version_handler))` in `sandboxd/src/main.rs` near the existing `/health` route per Spec 3 § 7.2; no authentication required.
- CLI `send_request` gains a version handshake immediately after `UnixStream::connect` per Spec 3 §§ 7.3-7.5: fetch `/version`, compare to `env!("CARGO_PKG_VERSION")`, refuse on mismatch with the verbatim stderr message (`version mismatch`, `CLI is`, `daemon is`, `both must match`) and exit code `2`.
- `sandbox doctor` and `sandbox version` subcommands bypass the strict-equality check per Spec 3 § 7.5; all other subcommands are gated.
- `sandbox version` subcommand returns immediately with the CLI's own `env!("CARGO_PKG_VERSION")` — no daemon connection at all per Spec 3 § 7.5.
- Confirm that `sandboxd --version` and `sandbox --version` produce exactly `<name> <semver>\n` (one line, two space-separated tokens) per Spec 3 § 7.6 — clap derives this from the `#[command(name, version)]` attributes; add a test that asserts the exact format.
- Unit tests per Spec 3 § 11.2 (`version_endpoint_returns_cargo_pkg_version`, `version_endpoint_returns_200_with_application_json`).
- Unit tests per Spec 3 § 11.3 (4 tests: proceeds on match, refuses on skew with verbatim stderr, bypassed for doctor, bypassed for version subcommand).
- Integration tests per Spec 3 § 11.6: `integration_version_endpoint_real_socket`, `integration_cli_refuses_on_version_skew`.

**Explicitly deferred.**

- Doctor C3 check (which calls `/version` as a check, not as a gate) → M14-S3.
- `sandbox update --check` consuming `/version` → M16 (Spec 5).

**Exit criteria.**

- All Spec 3 §§ 11.2, 11.3 unit tests pass.
- `sandbox version` prints the CLI semver without connecting to the daemon.
- `sandbox session list` (or any non-doctor/non-version subcommand) against a daemon with a different `CARGO_PKG_VERSION` exits with code `2` and the verbatim stderr message.
- `sandboxd --version` and `sandbox --version` each produce exactly one line matching `<name> <semver>` (verified by unit test).
- Integration tests `integration_version_endpoint_real_socket` and `integration_cli_refuses_on_version_skew` pass.
- `cargo nextest run --workspace` default and integration profiles clean; `cargo clippy --workspace` clean; code review approved.

## M14-S3 — `sandbox doctor` + `GET /diagnostics` endpoint

**Entry criteria.** M14-S2 (`/version` endpoint must exist because doctor's C3 check calls it; the CLI version-equality bypass for `doctor` must be in place so the test harness can invoke `sandbox doctor` against a running daemon without being refused by the version gate).

**Spec reference.** `../specs/2026-05-11-daemon-productionization-design/2026-05-11-daemon-productionization-design-spec.md` — §§ 6.1–6.5 (doctor CLI surface, check list C1-C13, output format examples, exit codes, code placement), § 12 (backward compatibility — dev mode: checks C4, C5, C10 degrade gracefully), § 13.2 (`GET /diagnostics` endpoint — system-level + per-operator scoped data, `/dev/kvm` access, C12 guest-version drift), § 13.6 (doctor parallel-check ordering: serial phase C1-C2, parallel phase C3-C13), §§ 11.4, 11.6, 11.7 (unit tests — doctor check registry; integration tests; notes on systemd integration harness).

**Rationale.** `sandbox doctor` is the operator's primary diagnostic surface for the system-service deployment. It is a pure CLI addition (new `doctor.rs` module + CLI dispatch arm) that calls back into the daemon via both the new `/version` endpoint (M14-S2) and the new `/diagnostics` endpoint introduced here. The `/diagnostics` endpoint delivers the daemon-uid-scoped data (KVM access, image presence, guest-version drift, substrate orphans) that the CLI process cannot check on behalf of the daemon's uid. The two surfaces are coupled and ship together.

**In scope.**

- New `sandboxd/sandbox-cli/src/doctor.rs` per Spec 3 §§ 6.5, 13.6: `Check` trait, check registry (C1-C13 enumerated), two-phase execution (serial C1-C2, then parallel C3-C13 if both pass), output formatter (colored `✓`/`~`/`✗` prefix, `--verbose` shows all, default suppresses passes, hint lines indented 4 spaces with `hint:` prefix, trailing summary line).
- `Command::Doctor { #[arg(long)] verbose: bool }` variant in `sandbox-cli/src/main.rs` + dispatch arm per Spec 3 § 6.5.
- Doctor checks C1-C13 as specified in Spec 3 § 6.2 (C1 falls back to `connect()` when no systemd; C4, C5, C10 degrade gracefully in dev mode per § 12.2; C12 skipped in default mode; C13 informational only).
- Exit codes 0/1/2 per Spec 3 § 6.4.
- `GET /diagnostics` endpoint in `sandboxd/src/main.rs` per Spec 3 § 13.2: system-level fields (`daemon_uid`, `daemon_user`, `kvm_readable`, `kvm_writable`, `gateway_image_present`, `lite_image_present`, `users_conf_pool`) returned to every connected operator; per-operator scoped fields (`guest_version_drift`, `substrate_orphans`) filtered by `caller_username` from `OperatorIdentity`.
- `docs/start/installation.md` note per Spec 3 § 16: brief description of the system-service install model and `sandbox doctor` as the diagnostic entry point.
- Unit tests per Spec 3 § 11.4 (doctor check registry: 10 unit tests covering individual check happy/fail paths, skip propagation, exit codes 0/1/2).
- Integration tests per Spec 3 § 11.6 (excluding `integration_systemd_unit_smokes` which requires Spec 4's Lima harness — see below): `integration_subdir_mode_correction_at_startup`, `integration_version_endpoint_real_socket` (already covered in M14-S2; re-run confirms no regression), `integration_doctor_hard_fails_on_missing_gateway_image`, `integration_doctor_informational_on_missing_lite_image`, `integration_doctor_full_pass_against_running_daemon`, `integration_kvm_check_via_daemon_diagnostics`.

**Explicitly deferred.**

- `integration_systemd_unit_smokes` → M15 (requires Spec 4's Lima-based E2E test harness). Per the project convention in `CLAUDE.md`, the test is named with the `integration_*` prefix so it runs only under `--profile integration` — no `#[ignore]` attribute, no env gate. Until M15's harness exists, this test will fail under `--profile integration` and is explicitly excluded from M14-S3's exit criteria gate; it is unblocked by M15-S3 which provides the harness and re-runs it to green.
- Doctor C12 display of stopped-session compatibility status → M16 (Spec 5 territory per Spec 3 § 14).
- Doctor on non-Linux systems (launchd / rc.d) → Spec 4+ per Spec 3 § 14.

**Exit criteria.**

- All Spec 3 § 11.4 unit tests pass under the default nextest profile.
- All in-scope § 11.6 integration tests pass under the `integration` nextest profile.
- `sandbox doctor` exit code is `0` when all checks pass, `1` when at least one fails, `2` when doctor itself cannot run.
- Output format exactly matches the three examples in Spec 3 § 6.3 (happy-path verbose, mixed with informational skip, partial-fail daemon-down) — message tokens and indentation are load-bearing for integration tests.
- `GET /diagnostics` returns the system-level fields verbatim and scopes `guest_version_drift` to the calling operator's sessions.
- `docs/start/installation.md` updated.
- `cargo nextest run --workspace` default and integration profiles clean; `cargo clippy --workspace` clean; code review approved.

## M14-S4 — Multi-track review

**Entry criteria.** M14-S3 complete — all three implementation sessions landed; `cargo nextest run --workspace` default and integration profiles pass.

**Spec reference.** `../specs/2026-05-11-daemon-productionization-design/2026-05-11-daemon-productionization-design-spec.md` (all sections).

**Rationale.** Spec 3 touches six distinct subsystems (startup enforcement, image tag pinning, QEMU wrapper, version contract, doctor, diagnostics endpoint) across five source crates. The six review tracks run in parallel to catch inter-subsystem drift, quality gaps, and any accidental re-introduction of patterns Spec 3 explicitly removes (rootless-Docker enablers, unpinned `:latest` references). Findings are triaged; only confirmed blockers are fixed before M14-S5.

**In scope.**

Six tracks, each delegated to a separate review agent in parallel:

- **Track 1 — Implementation vs. spec**: Re-read Spec 3 §§ 1–13 section by section; compare every concrete requirement (startup behavior, mode value, unit file directive, check name, exit code, error message token, wire shape) against the codebase; flag any spec claim missing from code or implemented differently from what the spec specifies.
- **Track 2 — Code quality**: Review `ensure_base_dir_layout`, `gateway_image_tag_for_version`, the QEMU wrapper edit, `version_handler`, `send_request` version gate, and `doctor.rs` for correctness, Rust/axum/tokio conventions per `CLAUDE.md`, unsafe unwraps, missing error propagation, and non-idiomatic patterns. No refactoring beyond what the specs planned.
- **Track 3 — Unit test quality**: For every unit test in M14-S1 through M14-S3, verify non-tautological assertions (the test's assertion is not derivable from its setup without exercising the production code). Run `cargo-llvm-cov` (or equivalent) on `sandboxd/src/main.rs` (base-dir layout, version handler), `sandbox-core/src/gateway.rs` (tag functions), `sandbox-core/src/lima.rs` (wrapper script string), and `sandbox-cli/src/doctor.rs` (check registry, output formatter). Report any uncovered line in these files with a reason or a new test.
- **Track 4 — E2E / integration test quality**: For integration tests in M14-S1 through M14-S3, verify each asserts real observable outputs (docker image tag strings, `--version` output byte-for-byte, daemon exit codes, `sandbox doctor` stdout message tokens, `GET /diagnostics` JSON field values). Verify no cases from Spec 3's test plans are missing. Verify no test passes trivially because it doesn't run the code it purports to test.
- **Track 5 — Docs quality**: Verify `docs/start/installation.md` update accurately describes the system-service shape; check that no doc file still references the old `:latest` gateway image reference or the old `helper=` QEMU argument in a way that contradicts Spec 3's changes; verify `sandboxd/contrib/systemd/sandboxd.service` matches the unit file content in Spec 3 § 4.1 verbatim.
- **Track 6 — Workarounds + deprecated patterns**: Grep the workspace for: (a) `GATEWAY_IMAGE` bare constant used in a docker-run position (must be replaced by `gateway_image_tag_for_version`); (b) any remaining `,helper=` in the QEMU wrapper string; (c) any `BRIDGE_HELPER` or `SANDBOX_BRIDGE_HELPER` references in `sandbox-core/src/`; (d) any `TODO`/`FIXME` introduced in M14 sessions; (e) any `#[ignore]` attribute on any integration test — the project convention uses only the `integration_*` naming prefix, never `#[ignore]`. For all tests: flag any tautological test.

**Explicitly deferred.**

- Spec-delivery claim-to-code map → M14-S5.

**Exit criteria.**

- All six review tracks complete; findings collated into a single prioritized list.
- Every "must-fix" finding addressed, re-implemented where needed, and re-tested.
- Code coverage report generated; uncovered lines in security-critical and deployment-critical paths have documented reasons or new tests.
- `cargo nextest run --workspace` default profile passes.
- `cargo nextest run --workspace --profile integration` passes.
