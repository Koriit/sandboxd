# M13 — Security foundations: helper identity + API session isolation

The two parallel security specs (Specs 1 and 2 of the five-spec arc) that make a dedicated `sandbox` system user safe. Spec 1 adds a pair-membership check to the cap'd route-helper and the `SO_PEERCRED`-derived operator identity that threads through every daemon invocation; Spec 2 adds per-caller API session ownership at the `SessionStore` boundary and the guest version compatibility infrastructure that `sandbox update` (Spec 5) will depend on. Together they close both specs' full scope. M13 is the terminal milestone for both.

## M13-S3 — Daemon SO_PEERCRED acceptor + `RuntimeStartArgs::for_user` threading

**Entry criteria.** M13-S2 (the helper's `--for-user` flag must exist and the pair-check must be implemented before the daemon meaningfully emits it; integration tests for propagation invoke the pair-check).

**Spec reference.** `../specs/2026-05-11-helper-identity-assertion-design/2026-05-11-helper-identity-assertion-design-spec.md` — §§ 6.1–6.5 (daemon-side changes: peer-credential read, threading to handlers, threading to helper invocation, invocation sites table, wire snapshot), § 7 (backward compatibility — dev mode), § 8.5 (integration tests — daemon → helper propagation), § 11 (implementation notes), § 12 (affected files: `sandboxd/src/main.rs`, backend files, `docs/start/installation.md`).

**Rationale.** With the helper's pair-check enforced, this session closes the Spec 1 loop on the daemon side: every accepted connection learns the connecting operator's identity from the kernel (`SO_PEERCRED`) — unforgeable by the client — and threads it through to every `runtime.start()` call site. The `OperatorIdentity` extension it introduces is the shared seam Spec 2 (M13-S4) consumes for session ownership; establishing it here avoids duplicating the SO_PEERCRED infrastructure across specs.

**In scope.**

- Custom `axum::serve` acceptor wrapping `tokio::net::UnixListener` per Spec 1 § 6.1: `stream.peer_cred()` immediately after accept, `nix::unistd::User::from_uid` resolution, refuse-on-failure (close stream, `warn!` with unresolved uid, continue accepting).
- `OperatorIdentity { uid: u32, name: String }` struct; attach via `Request::extensions_mut().insert(...)` per Spec 1 § 6.2.
- `RuntimeStartArgs::for_user: Option<String>` field with doc-comment per Spec 1 §§ 6.3–6.4 (Lima call sites L1, L2 thread the value through for forward-compatibility; `ContainerRuntime::start` errors if `for_user` is `None` when a helper path is configured).
- Thread `for_user: Some(operator.name)` from `create_session` (D1) and `start_session` (D2) handlers per Spec 1 § 6.4.
- `invoke_route_helper` signature gains `for_user: &str`; adds `.arg("--for-user").arg(for_user)` per Spec 1 § 6.3.
- Integration tests per Spec 1 § 8.5 (`--for-user` appears in helper argv from daemon; Lima backend does not invoke helper).
- `docs/start/installation.md` note per Spec 1 § 12: brief description of `_schema_version` field and the `"sandbox"` convention in `allow_users`, framed for operators editing the file by hand.

**Explicitly deferred.**

- `OperatorIdentity` consumed for `owner_username` stamping and per-caller filtering → M13-S4.
- Structured 4xx response on uid-without-passwd connection-close → Spec 2 § 4 notes this is acceptable bare-reset behavior; no change here.

**Exit criteria.**

- Integration tests per Spec 1 § 8.5 pass under the `integration` nextest profile.
- Dev-mode walkthrough per Spec 1 § 7.1 succeeds end-to-end: CLI connects → daemon reads `SO_PEERCRED` → helper invoked with `--for-user <operator>` → pool `["sandbox", "<operator>"]` → allowed.
- Wire snapshot per Spec 1 § 6.5: helper argv contains `--for-user <name>` between the helper binary and the two positional args.
- `docs/start/installation.md` updated.
- `cargo nextest run --workspace` default and integration profiles clean; `cargo clippy --workspace` clean; code review approved.

## M13-S4 — V006 migration, `SessionStore` per-caller filtering, handler wiring

**Entry criteria.** M13-S3 (`OperatorIdentity` extension must exist so HTTP handlers can extract the caller username for both owner stamping and per-caller filtering).

**Spec reference.** `../specs/2026-05-11-api-session-isolation-guest-compat-design/2026-05-11-api-session-isolation-guest-compat-design-spec.md` — §§ 2.1–2.6 (V006 migration, API isolation rule, affected endpoints H1-H12, `SessionStore` API changes, stable username identity, pinned decisions), § 4 (SO_PEERCRED plumbing — relationship to Spec 1), § 5 (wire snapshot — 404-on-foreign-id, request/response shape), § 6 (backward compatibility — dev mode: §§ 6.1–6.2), §§ 7.1–7.2 (migration tests, `SessionStore` filtering unit tests), § 7.3.1 (static-analysis test for `update_state_reconcile` callers).

**Rationale.** V006 and per-caller filtering are one atomic concern: the migration adds the `owner_username` column and the filter enforces it at the storage boundary. Splitting them would produce a PR where the column is unfiltered (a transient security regression) or the filter is present but unstamped (dead code). The `update_state_reconcile` rename ships here because its doc-comment and static-analysis test enforce the session-store safety rule that the per-caller filter depends on — a future HTTP handler reaching for "force this state" now sees the reconciler-only name, not a generic escape hatch. The changeset is wide but a reviewer holds one question: "when alice connects, does she see only her sessions?"

**In scope.**

- ~15 `SessionStore` methods gain `caller_username: &str` parameter with `AND owner_username = ?N` in their SQL per Spec 2 § 2.4 (row-exists-with-different-owner treated as not-found: `Ok(None)` for reads, `Err(SessionNotFound)` for mutations).

## M13-S5 — Guest version compat + refresh machinery

**Entry criteria.** M13-S4 (V006 columns `guest_protocol_version` and `guest_binary_version` must exist in the schema and be stamped by `create_session`; `SessionStore::get_session_by_name_or_id` must accept `caller_username`).

**Spec reference.** `../specs/2026-05-11-api-session-isolation-guest-compat-design/2026-05-11-api-session-isolation-guest-compat-design-spec.md` — §§ 3.1–3.10 (guest version compat: two version fields, constants, compatibility predicate, refresh decision tree, refusal error shape, embedded guest binary, `can_refresh_in_place` v1 body, per-backend refresh mechanics, atomic version update, on-demand version query), § 5.1 (guest wire protocol additions), § 7.3 (compat predicate unit tests), § 7.4 (integration tests — refresh path), § 7.5 (integration tests — refuse path + uid-without-passwd daemon).

**Rationale.** Guest version infrastructure has strong internal dependencies: the predicates gate `start_session`, the predicates call `refresh_guest_binary`, Rust requires both backend implementations to compile the trait simultaneously, and `GuestRequest::Version` must land alongside the constants it reports. The session is wide but a reviewer holds one question: "when a session's stored proto version doesn't match the daemon's, what happens?"

**In scope.**

- `DAEMON_GUEST_PROTO_VERSION: u32 = 1` and `SANDBOX_GUEST_VERSION: &str` constants in `sandbox-core/src/guest.rs` per Spec 2 § 3.2; update `create_session` to stamp real values (replacing S4's placeholder `0`/`""`).
- `is_protocol_compatible(session_proto: u32) -> bool` and `can_refresh_in_place(session_proto: u32) -> bool` predicates per Spec 2 §§ 3.3, 3.7.
- `GuestRequest::Version` + `GuestResponse::VersionResult` enum variants per Spec 2 §§ 3.10, 5.1; guest-side `handle_request` dispatch arm (returns compile-time constants).
- `SessionRuntime::refresh_guest_binary(&self, handle: &RuntimeHandle) -> Result<(), SandboxError>` trait method per Spec 2 § 3.8.
- Container backend `refresh_guest_binary`: `docker stop -t 5` (idempotent), write embedded guest binary to host tempfile (mode `0755`), `docker cp`, drop tempfile per Spec 2 § 3.8.1.
- Lima backend `refresh_guest_binary`: ensure VM running (`limactl start` if needed), write embedded binary to tempfile, `limactl copy` + `sudo mv` + `sudo chmod +x`, `systemctl restart sandbox-guest`, `limactl stop` per Spec 2 § 3.8.2.
- `GuestProtocolIncompatible` error variant + HTTP 409 mapping in `error.rs` per Spec 2 § 3.5 (message tokens `refresh is not viable`, `recreate the session` are load-bearing for integration tests).
- `update_guest_versions(caller_username, id, proto, binary_version) -> Result<(), SandboxError>` in `SessionStore` per Spec 2 § 3.9; called only after both refresh and `runtime.start` succeed.
- `start_session` compat gate per Spec 2 § 3.4 decision tree: `is_protocol_compatible` → fast path; `can_refresh_in_place` → `refresh_guest_binary` + `runtime.start` + `update_guest_versions`; else → 409.
- Unit tests per Spec 2 § 7.3 (compat predicate table, 6 rows).
- Integration tests per Spec 2 §§ 7.4, 7.5.

**Explicitly deferred.**

- Optional post-start cross-check (`GuestRequest::Version` after start to detect out-of-band binary edits) → implementer judgment per Spec 2 § 3.10; not a hard requirement for Spec 2.
- `GET /diagnostics` C12 guest-version drift check → M14 (Spec 3 § 13.2).

**Exit criteria.**

- All Spec 2 § 7.3 predicate unit tests pass under the default nextest profile.
- Integration test: `start_session` on a compatible session takes the fast path (no `refresh_guest_binary` call, no extra docker stop/start cycle).
- Integration test: `start_session` on an incompatible refreshable session runs `refresh_guest_binary` then starts; `guest_protocol_version` in DB updates to the daemon's current version.
- Integration test: `start_session` on a non-refreshable session (proto `= 0`) returns HTTP 409 with both message tokens `refresh is not viable` and `recreate the session`.
- `cargo nextest run --workspace` default profile clean; `--profile integration` clean; `cargo clippy --workspace` clean; code review approved.
