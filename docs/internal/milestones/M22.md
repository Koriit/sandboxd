# M22 — macOS install, release, update & e2e

The fourth and heaviest milestone of the macOS support arc (the **terminal** milestone is M23). M19–M21 built the runtime; M22 makes it *shippable, installable, and updatable* (verification is M23). It lands the `install.sh`/`uninstall.sh` Darwin branch, the launchd system-daemon + per-operator-agent + socket_vmnet-pool install machinery (with the root-exec foothold guarantees), the release pipeline that produces a cosign-signed macOS tarball with ad-hoc-signed Mach-Os and a `linux/arm64` `sandbox-guest`, the runtime image-management + operational lifecycle (golden base VM, gateway-image self-heal, reboot/wake, disk), `sandbox update` parity via launchctl, the full one-suite/thin-seam/marker e2e harness, the Tart-driven install-e2e, and the SIP/TCC resolution of the spec's open verification item. M22 ends at S9 (install/release/update/e2e work).

This is **not** the terminal milestone — **M23 (macOS polish and fixes)** follows with the six-track review and the relocated spec-delivery verification. (The verification moved here per the planning skill's relocation caveat: when a milestone stops being terminal, its verification session moves to the new terminal milestone.) M22 is the **heaviest milestone of the arc** (9 sessions, four themes); S4's release-pipeline output is a cascade source for S5–S9, so sequence with headroom. Per the arc execution model (§ `session-plan.md`), nothing here publishes independently — the release fires only after M23 closes.

## M22-S1 — install.sh macOS foundation + daemon libexec path move

**Entry criteria.** M21-S5 (runtime complete on both backends).

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — § M15 → install.sh changes; Install paths on macOS; Install daemon model: system launchd daemon; Per-platform deltas (the `sandboxd` install-path note).

**Rationale.** install.sh is the canonical install path on both platforms, and the macOS branch must create the `_sandbox` system user, the per-uid state tree, and run the macOS prerequisite checks; this session lands that foundation plus the cross-platform cleanup of moving the daemon binary off user PATH into `/usr/local/libexec/sandboxd/`. The system-user provisioning (`dscl`/`sysadminctl`, one `-create` per attribute, disabled login, uid below 500) is macOS-idiomatic and error-prone, so it is scoped here with idempotency. The path move is bundled because it touches the same install/update binary-placement surface.

**In scope.**

- Implement the `detect_os` / arch branch (Darwin → arm64-only, reject Intel with a clear message) and the macOS prerequisite checks per § install.sh changes (socket_vmnet installed/running, lima version, socket_vmnet access; skip `check_kernel_version`).
- Implement `_sandbox` user/group provisioning per § install.sh changes (dscl/sysadminctl, RealName/disabled-password, idempotent, uid < 500) and state-dir creation with owners/modes per § Install paths; operator group membership via `dseditgroup` with the log-out/in warning.
- Move the daemon binary to `/usr/local/libexec/sandboxd/sandboxd` on **both** platforms per § Per-platform deltas note (update install.sh + the systemd unit; the launchd plist and the `sandbox update` swap-location land in S2/S6).

**Explicitly deferred.**

- launchd plists, agent install, socket_vmnet pool daemon install → M22-S2.
- The full uninstall path → M22-S3.

**Exit criteria.**

- install-e2e (or a Tart smoke once S8 lands) confirms the macOS branch creates the user/group idempotently, creates state dirs with correct owner/mode, and rejects Intel + missing prerequisites with actionable messages.
- The Linux install + systemd unit still work with the daemon at the libexec path; full suites green.
- Code review approved; install docs updated for macOS.

## M22-S2 — launchd daemon + per-operator agent + socket_vmnet pool install; log rotation; graceful shutdown

**Entry criteria.** M22-S1.

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — § M15 → launchd plist; Install paths; install.sh changes (socket_vmnet pool daemons; per-operator agent install); Log rotation; Graceful shutdown contract; § Prerequisites → socket_vmnet access (root-exec foothold).

**Rationale.** This session lands the privileged install machinery: the system-daemon launchd plist, the per-operator agent + LaunchAgent (no setuid), and the N root-owned socket_vmnet pool daemons (production-staged at the root-exec-safe `/Library/sandboxd` path, with the launchd codesign pre-exec guard; the daemon's startup ancestry assertion itself lands in M20-S1). The boot-ordering and dir-ownership subtleties (`/var/run` cleared on boot; unordered jobs racing to create `/var/run/sandbox` with the right owner via `install -d -o _sandbox`) are load-bearing and specified precisely. In-process log rotation and the graceful-shutdown contract round out the launchd service model.

**In scope.**

- Ship and install `io.sandboxd.daemon.plist` per § launchd plist (UserName/KeepAlive/ThrottleInterval/ExitTimeOut/EnvironmentVariables incl. pinned `HOME`; note the no-`StartLimitBurst`-equivalent caveat → any give-up policy is in-daemon).
- Install the per-operator agent + `io.sandboxd.agent` LaunchAgent (no setuid; `LimitLoadToSessionType: Aqua`; clean-exit self-gate) and the N `io.sandboxd.vmnet.<n>` pool daemons with **production** root-exec staging at `/Library/sandboxd/socket_vmnet` + the launchd codesign/cdhash **pre-exec guard** + the race-safe `install -d -o _sandbox` socket-dir creation, per § install.sh changes and § socket_vmnet access (root-exec foothold). (The daemon's startup ancestry+ownership assertion is **not** re-implemented here — it landed in M20-S1; this session's install-e2e *exercises* it.) This includes the sudo-driven vmnet plist regen path M21-S3 guides operators toward.
- Implement in-process log rotation (tracing-appender, macOS branch) per § Log rotation and confirm the graceful-shutdown contract per § Graceful shutdown contract.

**Explicitly deferred.**

- `sandbox update`'s launchctl swap → M22-S6; uninstall teardown of these → M22-S3.

**Exit criteria.**

- install-e2e / Tart (S8) confirms the daemon `bootstrap`s + `enable`s and reaches healthy, the agent LaunchAgent + N vmnet daemons load, sockets appear `_sandbox:_sandbox 0750`, and the root-exec ancestry assertion passes; the agent binary is non-setuid `root:wheel 0755`.
- The daemon's startup ownership assertion + the launchd pre-exec guard both fire on a tampered binary (negative test where feasible); full suites green.
- Code review approved (with a pass on the root-exec foothold + dir-ownership race); install docs updated.

## M22-S3 — uninstall.sh macOS branch

**Entry criteria.** M22-S2.

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — § M15 → scripts/uninstall.sh on macOS; Release tarball contents → Shared notes (assembler / ui.sh; TUI parity).

**Rationale.** Uninstall must mirror install and tear down everything install created, idempotently and safely — notably **never** `pkill qemu-system` (it would kill the operator's personal Lima/Colima VMs) and never shelling `limactl` directly. The 8-step Darwin teardown (pre-flight refusal, bootout daemon + vmnet daemons, remove binaries/agent, optional state purge, user/group removal members-first) animates through the shared `ui.sh` TUI engine for parity with install. Pairing it right after the install sessions keeps the install/uninstall surface coherent.

**In scope.**

- Implement the Darwin uninstall sequence per § scripts/uninstall.sh on macOS (pre-flight refusal + `--force`; daemon bootout; leftover-VM handling via the purge step, not `pkill`; vmnet daemon bootout + plist removal; binary/agent removal incl. `/Library/sandboxd`; optional `--purge-state`; `_sandbox` user/group removal members-first; verification).
- Wire the macOS uninstall steps through the shared `ui.sh` TUI engine with byte-identical plain-path degradation per § Release tarball → Shared notes.
- Honor `--purge-state` / `--force` / `--yes` semantics per § Flags on uninstall.sh (force backed by `DELETE /sessions/{id}?force=true`).

**Explicitly deferred.**

- Tart-driven full install → uninstall → re-install e2e → M22-S8.

**Exit criteria.**

- install-e2e / Tart (S8) confirms uninstall removes the daemon + agent LaunchAgent + vmnet daemons, with `--purge-state` removing the state tree + user + group (members stripped first) and without it preserving state for re-install; no `pkill qemu` line exists anywhere.
- Idempotent re-run exits 0; full suites green.
- Code review approved; uninstall docs updated.

## M22-S4 — Release pipeline, tarballs, signing, cosign pinning, CI matrix

**Entry criteria.** M22-S3.

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — § M15 → Build artifact matrix; CI matrix; Apple code signing and notarization; Release tarball contents; Gateway image distribution; § M16 → Cosign verification (the pin-table gap).

**Rationale.** The release pipeline must produce a signed, verifiable macOS tarball with the right artifacts (incl. a `linux/arm64` `sandbox-guest` and per-arch gateway-image tarballs built on the Linux runner), ad-hoc-sign every Mach-O (an Apple-Silicon `execve` invariant, independent of notarization), and close the cosign pin-table gap so the macOS trust root is actually verified. The `build-macos` CI job runs the native macOS suites with no nested virtualization. This is the artifact/trust surface the install and update paths consume.

**In scope.**

- Implement the per-artifact build + tarball assembly per § Build artifact matrix and § Release tarball contents (macOS tarball: `sandboxd`/`sandbox`/`sandbox-agent`/`sandbox-guest` (linux/arm64 ELF), launchd plists, gateway-linux-arm64 image tarball; per-arch gateway images built on the Linux runner per § Gateway image distribution).
- Implement ad-hoc signing + verification of every shipped Mach-O per § Apple code signing (re-sign after any `strip`; cosign ships pre-signed; no entitlements needed — the HVF entitlement rides Homebrew's `qemu`).
- Add the `build-macos` CI job per § CI matrix (native `test-e2e-macos`, no nested virt) and close the cosign pin-table gap per § Cosign verification (add `cosign-darwin-arm64` version + SHA256 to `lib.sh` + the Rust pin table + the drift test).

**Explicitly deferred.**

- The e2e harness those CI jobs invoke → M22-S7; the Tart install-e2e → M22-S8.

**Exit criteria.**

- The pipeline produces a cosign-signed macOS tarball whose every Mach-O passes `codesign -v` and whose MANIFEST re-verifies after extraction; `cosign_pin_for_arch("aarch64-apple-darwin")` resolves and the drift test passes.
- `build-macos` is green in CI (hermetic + integration for macOS paths; e2e/install-e2e gated on S7/S8); the Linux release path is unchanged.
- Code review approved; release docs updated for the macOS tarball.

## M22-S5 — Runtime image management and operational lifecycle on macOS

**Entry criteria.** M22-S4.

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — § M15 → Golden base VM (`sandbox-base`) on macOS; Gateway image distribution (per-arch tarballs); Reboot and wake-from-sleep; Coexistence with user-owned Lima and Colima; Disk usage.

**Rationale.** Several runtime/operational behaviors are macOS-specific and cohere as "how images, disk, and VMs behave across the daemon's life on a Mac": the per-operator lazy `sandbox-base` build, the gateway image `docker load`-into-Lima + self-heal from the retained tarball, disk-usage accounting + `check_disk_space`, reboot/wake reconcile, and coexistence with personal Lima/Colima. None requires new privilege; they round out the macOS runtime so an operator's first-create, reboot, and disk-pressure experiences are correct.

**In scope.**

- Implement per-operator lazy `sandbox-base` build + version-bump refresh per § Golden base VM, and gateway-image `docker load`-into-Lima + automatic self-heal from the retained tarball per § Gateway image distribution (incl. the `check_gateway_image_inside_lima` self-heal trigger deferred from M21-S5).
- Implement reboot / wake-from-sleep reconcile wiring per § Reboot and wake-from-sleep (sessions → stopped after reboot, slot release, gateway-health poll on wake — reusing the M20-S5 / M21-S1 reconcile).
- Implement `check_disk_space` + the APFS accounting notes per § Disk usage; document coexistence per § Coexistence. Mark the gateway DNS-resolver-staleness remedy (re-read `scutil --dns` after a host network change / wake) as **docs-only this cut** per § Upstream DNS resolver discovery, so M23-S2's claim-to-code map has an owner for that spec claim rather than flagging it unowned.

**Explicitly deferred.**

- `sandbox update`'s gateway-image refresh sequencing → M22-S6 (this session owns the load/self-heal; S6 sequences it into update).

**Exit criteria.**

- E2E confirms an operator's first create lazily builds `sandbox-base` (with the one-time progress line) and subsequent creates clone fast — with a **coarse clone-fast assertion** (clone completes in a few seconds / produces a sparse file) guarding against a silent APFS-copy-on-write → full-copy regression if `/var/lib/sandboxd` ever lands cross-volume or non-APFS; the gateway image self-heals from the retained tarball when missing inside the Lima VM.
- `check_disk_space` warns below threshold; a post-reboot reconcile marks running sessions stopped and releases slots; full suites green.
- Code review approved; coexistence + disk docs shipped.

## M22-S6 — `sandbox update` on macOS

**Entry criteria.** M22-S5.

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — § M16 — `sandbox update` (Pre-flight; Service control via launchctl; Gateway Lima instance handling; Tarball download; Cosign verification; Backup retention / Backup contents; Agent/helper handling after swap).

**Rationale.** `sandbox update` must work on macOS with the same UX as Linux, with platform differences confined to the launchctl service-control step, the agent/binary swap (no re-privilege — there is no setuid on macOS), and the gateway-image refresh inside the persistent Lima instance. The backup set is redefined for macOS artifacts, and the cosign pin table (from S4) is what makes the verified download path safe. This is the last operational CLI surface to reach macOS parity.

**In scope.**

- Implement macOS service control + swap per § Service control via launchctl and § Agent/helper handling after swap (bootout/bootstrap; `was_running`; agent-binary swap with `launchctl kickstart -k` or next-login pickup; no setcap step).
- Implement the macOS gateway-image refresh per § Gateway Lima instance handling (load the staged tarball into the persistent instance on first start post-update) and the macOS pre-flight additions per § Pre-flight.
- Wire macOS into the existing auto-download (v-prefix tag / bare-filename asset URL contract; `aarch64-apple-darwin` asset) and cosign verification per § Tarball download and § Cosign verification (reusing the S4 pin table), plus the macOS backup set per § Backup contents.
- Update **`sandbox update`'s binary-swap target to `/usr/local/libexec/sandboxd/sandboxd` on *both* platforms** — the cross-platform half of the M22-S1 daemon-path move (spec § Per-platform deltas, "`sandbox update` is updated to swap the binary at the new location"). Without this a post-move update swaps the wrong path on Linux. (This closes the ownership gap the review flagged between M22-S1/S2/S6.)

**Explicitly deferred.**

- Tart-driven update scenario e2e (rollback, retention, multi-version) → M22-S8.

**Exit criteria.**

- A macOS `sandbox update` end-to-end (exercised by Tart in S8) performs daemon stop → verified download → binary + agent swap → daemon start → gateway-image reload → health, under one minute, with `was_running` honored.
- The macOS backup set restores a rollback (incl. the previous gateway-image tarball); an unverified cosign is refused (no unverified trust root); full suites green.
- Code review approved; update docs updated for macOS.

## M22-S8 — Tart install / uninstall / update e2e

**Entry criteria.** M22-S7.

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — § M15 → install / uninstall e2e coverage (Tart); § Prerequisites → Tart.

**Rationale.** install/uninstall/update mutate system state (users, `/Library/LaunchDaemons`, the pool daemons) that must not touch a developer's real machine, so the macOS install-e2e mirrors the Linux scenario set against throwaway Tart macOS VMs. Because the install *steps* differ per platform, the shared layer is the scenario shape and the observable end-state assertions, not the step implementations. This is the session that actually exercises the S1–S6 install/uninstall/update paths against the `build.sh`-assembled scripts.

**In scope.**

- Implement the Tart-based install-e2e mirroring per § install/uninstall e2e coverage: happy-path install assertions, idempotency, refusals, uninstall (with/without `--purge-state`), and the headline install → create-session → uninstall → re-install cycle.
- Implement the macOS `sandbox update` scenarios per the same section (launchctl swap, agent-binary swap, gateway-image reload, backup retention, rollback).
- Run against the `build.sh`-assembled `install.sh` / `uninstall.sh` (ui.sh inlined) so the suite guards the assembler's Darwin branch too.
- Add a **`/var/run` reboot-race test** (reboot the Tart VM, or `rm -rf /var/run/sandbox` + relaunch the vmnet daemons and the daemon in *both* orders) asserting `/var/run/sandbox` ends up `_sandbox:_sandbox 0750` regardless of which unordered job wins — the most macOS-idiomatic boot hazard, currently only install-asserted.
- Add a **coexistence negative test**: plant a dummy personal `~/.lima/<instance>` and a Homebrew `socket_vmnet` socket, run install + uninstall, and confirm both survive untouched (guards the "never `pkill qemu-system`" rule and the separate-namespace claim).

**Explicitly deferred.**

- The TCC real-chain verification (needs a GUI Aqua session, not a headless Tart step) → M22-S9.

**Exit criteria.**

- `make test-install-e2e` passes on the `build-macos` CI job (Tart) across the full scenario set incl. the install → session → uninstall → re-install cycle and the update scenarios.
- The assembled scripts (not the dev-tree sources) are what the suite exercises; full suites green.
- Code review approved; install-e2e docs updated for the macOS matrix.

## M22-S9 — SIP & TCC (resolve the open verification item)

**Entry criteria.** M22-S8.

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — § M15 → SIP and TCC (the open verification item); § Apple code signing (agent must be a signed Mach-O, never a script).

**Rationale.** `shared:` into TCC-protected dirs is made *possible* by the per-operator-agent model, but the spec carries an explicit open verification item: which binary in the agent→`limactl`→`qemu-system` chain TCC attributes the protected read to, and whether a prompt appears at all (since `limactl start` typically daemonizes qemu, detaching it from the agent's responsibility chain). This session runs the real-chain verification on hardware, finalizes `check_tcc_workspace_access` as informational, and documents the outcome (agent-attributed grant vs the qemu/limactl re-prompt-on-upgrade fallback / Full Disk Access). It is scoped late because it needs the signed agent (S4) and a real Aqua login session.

**In scope.**

- Confirm no SIP carve-outs are needed (all written paths are SIP-allowed) per § SIP.
- Run the real-chain TCC verification per § SIP and TCC → Open verification item (launchd-at-login, drive `limactl start` of a `shared:~/Documents/…` session, record whether a prompt appears and which binary it names) and encode the outcome: agent-attributed (desired) or qemu/limactl-attributed (document re-prompt-on-`brew upgrade` + the Full Disk Access fallback).
- Finalize `check_tcc_workspace_access` as the informational doctor verdict per § SIP and TCC, and confirm the agent remains a dedicated signed Mach-O (never reduced to a script wrapper).

**Explicitly deferred.**

- Apple Developer ID signing / notarization → out of scope (spec Non-goal).

**Exit criteria.**

- The real-chain TCC behavior is verified on a real Apple-Silicon GUI session — a **manual, non-CI hardware verification** (like the M18 carry-forward #17; a headless `macos-14` runner and a headless Tart step provide no Aqua session, so this cannot be a green CI gate) — and its outcome (prompt presence + responsible binary + upgrade-invalidation behavior) is **recorded in the spec/docs** for M23-S2's amendment verification to check. The **automated** assertion CI runs is `check_tcc_workspace_access` returning its informational verdict.
- Non-protected-path `shared:` needs no consent (confirmed); full suites green.
- Code review approved; TCC operator docs (one-time prompt / FDA fallback / zero-prompt default) shipped.
