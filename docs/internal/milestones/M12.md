M12 starts here because every subsequent UDP improvement (S2's e2e tests, S3's spec coherence) depends on having correct deny-event attribution first.

- Fix #29: replace the IP_ORIGDSTADDR-only path in the UDP deny-logger handler with a conntrack netlink lookup for the original destination (likely via `nfnetlink_conntrack` or equivalent) so deny events carry the pre-DNAT 5-tuple.
- Rewrite the Phase 8 test 2 assertion in `sandboxd/sandboxd/tests/m10_s3_end_to_end.rs` to expect the pre-DNAT tuple. Remove the explanatory comment that documented the workaround.

## M12-S7 — Native `cp` via `limactl cp` / `docker cp`

**Spec reference.** Ad-hoc. Anchored on todo #101: "shouldn't sandbox cp actually use limactl cp on lima and docker cp on docker backend?"

**Rationale.** Today `sandbox cp` implements file transfer through an in-tree path (likely vsock copy / ssh-tee / generic byte-pump) that duplicates effort `limactl cp` and `docker cp` already do well. Backend-native tooling handles edge cases (sparse files, attribute preservation, large files, error message clarity) that an in-tree implementation tends to gloss over. Refactoring to dispatch to the backend's native cp tool is a behavior-preserving (or behavior-improving) refactor that also clears the path for S8's rsync-style sync — that command will dispatch to `limactl shell -- rsync` and `docker exec rsync` on the same dispatch surface this session establishes. Doing the cp refactor before sync means S8 doesn't need to design dispatch from scratch, and a single PR boundary keeps the behavior change reviewable in isolation from the new feature.

**In scope.**

- Audit the current `sandbox cp` implementation: where it lives, what it currently does, what edge cases it handles or doesn't.
- Refactor to dispatch by backend kind: Lima sessions invoke `limactl cp`, container sessions invoke `docker cp`. Keep the user-facing CLI surface unchanged for typical invocations.
- Establish a small dispatch trait or helper for "transfer file from host to/from session" that S8 can extend for rsync.
- Update or add tests exercising the new dispatch path on both backends. Cover attribute preservation, large-file behavior, and error-message clarity (native tools have different error styles than ad-hoc Rust).
- Update docs that describe `sandbox cp` to reflect any user-visible behavior changes — likely improvements (better errors, sparse-file handling).

**Explicitly deferred.**

- rsync-like full-tree sync → M12-S8.
- Behavior changes beyond "use native tooling" (progress bars, multi-host transfer, etc.) — out of M12 scope.

**Exit criteria.**

- `sandbox cp` invokes `limactl cp` for Lima and `docker cp` for container backends; no in-tree byte-pumping path remains.

## M12-S8 — rsync-like directory sync

**Entry criteria.** M12-S7 (uses the dispatch surface S7 establishes).

**Spec reference.** Ad-hoc. Anchored on todo #104: "cp is fine but we need also full rsync-like sync for dirs; not just simple copy".

**Rationale.** `cp -r` is fine for one-shot transfers but doesn't suit directory-sync workflows: incremental updates, deletion mirroring, attribute preservation across re-runs, and the ability to bail out cleanly on partial failures. These are all properties rsync already provides. Adding a `sandbox sync` (or extending `sandbox cp` with an `--rsync`/`--archive` mode) on top of S7's backend dispatch lets us inherit the full rsync feature set without reimplementing. Lima dispatches via `limactl shell -- rsync`; Docker dispatches via `docker exec rsync` if rsync is installed in the session image, with a chosen fallback strategy for images that don't ship it. The user explicitly asked for "full rsync-like sync, not just simple copy" — this session is the deliverable.

**In scope.**

- Decide CLI surface: new `sandbox sync` command vs. flag on `sandbox cp` (`--rsync` / `-a` / `--archive`). Capture decision and rationale in commit body.
- Implement on both backends:
  - Lima: dispatch via `limactl shell -- rsync` or `limactl cp`-driven round-trip; reuse the existing Lima ssh surface; ensure ssh-key handling matches.
  - Docker: dispatch via `docker exec rsync` if rsync is installed; surface a clear error or fall back to `tar | docker exec tar` if not — choose one strategy with rationale captured in commit body and user-visible doc.
- Add tests on both backends covering: full-tree sync, incremental sync (re-run produces no copy), deletion mirroring (`--delete` or equivalent), attribute preservation.
- Document the new command/flag in user-facing CLI reference and add an example in the relevant guide.

**Explicitly deferred.**

- Sync-as-watch (continuous syncing) — out of M12 scope.

**Exit criteria.**

- New CLI surface ships and is documented.
- E2E tests on both backends prove: full sync, incremental sync, deletion mirroring, attribute preservation.
- Container backend's rsync availability story is decided, implemented, and documented (require in image / fallback to tar / error early).
