# M23 — macOS polish and fixes (terminal)

The terminal milestone of the macOS support arc. M19–M22 built the platform and made it shippable; M23 is a dedicated polish-and-fixes pass that holds the whole arc to its promises before it closes. It is two sessions: a **six-track multi-track review** — each track handled by a separate subagent — whose findings are fixed in-session, followed by the **spec-delivery verification** that was relocated here from M22 when M22 stopped being terminal (the planning skill's relocation caveat). The review tracks mirror the established M18-S8 / M18-S14 model, with explicit emphasis on non-tautological test assertions, coverage-gap analysis, and standing up code-coverage tooling where it is missing.

This **is** the terminal milestone for the `2026-05-02-macos-support-design.md` spec. M23-S2 produces the claim-to-code delivery map and flips the macOS arc to complete under a single conjunctive hard gate.

## M23-S1 — Six-track arc review and fixes

**Entry criteria.** M22-S9 (the macOS arc's implementation is complete across M19–M22).

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — whole spec; § Security posture; § Privilege model on macOS; all M13–M18 integration sections; § M15 macOS e2e harness adaptation (the test surfaces under review).

**Rationale.** A 28-session, cross-platform, security-critical arc accumulates drift between what was promised and what shipped, tautological tests, and stop-gap workarounds; a dedicated multi-track review — one subagent per track — catches these while the arc is fresh, and fixes them in-session so the spec-delivery gate that follows is measuring a clean tree. The test tracks carry a special burden because the macOS suite is largely *reused* Linux bodies across new cells (a tautological or low-coverage test would silently pass on macOS while asserting nothing); so non-tautological assertion review, coverage-gap analysis, and code-coverage tooling are first-class here, not afterthoughts.

**In scope (six parallel tracks, one subagent each).**

- **Track 1 — implementation vs. plan/promise.** Review the M19–M22 implementation against what the spec and the milestone session intents originally planned and promised; flag scope drift, unimplemented promises, and silent behavior changes.
- **Track 2 — code quality.** Review the macOS code surface for clarity, conventions, and the project's idioms (per `CLAUDE.md`), incl. the `spawn_blocking` carve-outs and the privilege-model conventions.
- **Track 3 — unit-test quality.** Review unit tests with a special focus on whether they assert *real behavior* (non-tautological) and whether coverage is adequate (are any cases missing?). **Set up code-coverage tooling if it is not already present** and use it to surface gaps.
- **Track 4 — e2e-test quality.** Review the e2e (and install-e2e / Tart) tests with the same non-tautological + coverage-gap focus; confirm the marker matrix exercises the cells it claims to and that no platform-specific behavior silently skips.
- **Track 5 — docs quality.** Review user / operator / contributor / workspace docs for accuracy against the shipped macOS behavior.
- **Track 6 — workarounds & deprecated patterns.** Find workarounds and uses of deprecated things that should be replaced with proper solutions; replace them rather than annotate them.
- Aggregate all tracks' findings and **fix every finding in-session** under a no-open-findings rule.

**Explicitly deferred.**

- The claim-to-code spec-delivery map → M23-S2.

**Exit criteria.**

- All six review tracks complete; every finding fixed in-branch (no open findings carried out of the session). [your criterion 1: *review done, findings fixed*]
- Code-coverage tooling is in place (set up if it was missing), and the coverage gaps the review identified in unit and e2e tests are closed.
- All unit and e2e tests pass (full hermetic + integration + native macOS e2e + install-e2e (Tart) + Linux suites green); clippy `-D warnings` + fmt clean on both platforms. [your criterion 2: *all unit and e2e tests pass*]
- Code review approved across all tracks; any spec inaccuracy found is corrected in the spec in-branch.

## M23-S2 — Spec-delivery verification (terminal)

**Entry criteria.** M23-S1.

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — the entire spec: every concrete claim, Non-goals, Known gaps / deferred decisions, and the applied networking-design.md corrections. Delivery artifact: `../../../.tasks/specs/2026-05-02-macos-support-design-delivery.md`.

**Rationale.** This is the terminal session of the macOS arc; its job is to make the code match the *spec*, not merely the session bullets. It produces the claim-to-code verification map, replays every prescribed example, confirms out-of-scope items are genuinely absent, reconciles known gaps to tracked follow-ons, and verifies the networking-design.md corrections are reflected. The exit is a single conjunctive hard gate — the incentive gap between "bullets shipped" and "spec delivered" closes here. It runs after M23-S1 so the map is built against a reviewed, fixed tree.

**In scope.**

- Produce the claim-to-code map at `.tasks/specs/2026-05-02-macos-support-design-delivery.md` (force-added past the gitignore on the spec location): every concrete claim → (a) a code+test locator, (b) a Non-goals citation, or (c) a tracked follow-on milestone/issue, per the claim-to-code format.
- Example replay (e.g. the `sandbox inspect` slot output, the install/uninstall observable end-states, the egress delete-route regression); out-of-scope conformance (vzNAT/SLIRP never an egress NIC, no setuid on the host, no `/etc/sudoers.d/lima`, no Intel build, headless full-backend); known-gap reconciliation (#14 / #15 / #63 / #17, the TCC follow-up) to tracked issues.
- Amendment verification: confirm the five networking-design.md corrections are reflected and any in-branch spec amendments (the argv-rewrite shape, the TCC attribution outcome) are written back into the spec.

**Exit criteria (single conjunctive hard gate — ALL must hold).**

- Every concrete spec claim maps to (a) code+test, (b) a Non-goals citation, or (c) a named tracked follow-on, with zero unexplained gaps; every prescribed example replays to the spec's stated output.
- Out-of-scope items confirmed absent from the tree; known gaps each tracked as a `progress` todo or filed issue; networking-design.md corrections + in-branch spec amendments verified present.
- Full e2e matrix (`make test-e2e-linux` + `make test-e2e-macos`), install-e2e (Tart + Linux), integration, hermetic, clippy `-D warnings`, and fmt all green on both platforms; the delivery map committed — only then is M23, and the macOS arc, marked complete.
