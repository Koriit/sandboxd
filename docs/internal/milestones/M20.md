# M20 — macOS networking & VM runtime: first booting session with enforced egress

The second milestone of the macOS support arc. On the foundation M19 laid (the `LimaExecutor` seam, the per-operator agent + broker), M20 builds the networking and VM-runtime machinery that lets a **full-backend macOS session actually boot** — and proves the property that is sandboxd's entire reason to exist: **absolute, unbypassable egress control**. The full backend uses `vmType: "qemu"` + socket_vmnet (never VZ — VZ offers no interceptable isolated L2 segment), an isolated socket_vmnet slot per session as the gateway-facing NIC, and a fail-closed qemu-wrapper that locks Lima's unavoidable slirp management NIC to `-netdev user,restrict=on`. M20 lands the socket_vmnet pool, the persistent gateway Lima instance, the sandbox VM template + wrapper, the session lifecycle, the daemon startup sequence, and — as the milestone's headline deliverable — an end-to-end egress regression that confirms a root guest cannot escape the gateway.

This is **not** the terminal milestone of the spec; no spec-delivery verification session here. The egress E2E (M20-S6) carries a conjunctive gate because the core promise is non-negotiable.

## M20-S1 — socket_vmnet pool and the vmnet launchd daemon set

**Entry criteria.** M19-S6 (foundations complete).

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — § socket_vmnet pool (Purpose, Instance naming, Subnet allocation, Slot lifecycle, Pool exhaustion, Initialization, Concurrency); § Prerequisites → socket_vmnet access; § M15 install.sh → socket_vmnet pool daemons.

**Rationale.** socket_vmnet slots are the isolated L2 segments the whole macOS egress model depends on; the pool is the macOS counterpart to Linux's `SubnetAllocator` and must encode the /29 layout, the NIC-slot-vs-session-token distinction, and the unmanaged per-VM `socket:` attach (no `networks.yaml` anywhere). Because the N `io.sandboxd.vmnet.<n>` launchd daemons are root network infra, this session also lands their plist generation and a dev-time bootstrap so later sessions are testable. The per-slot `--vmnet-network-identifier` isolation and `--socket-group=_sandbox` are load-bearing for the inter-slot-isolation invariant the security model rests on.

**In scope.**

- Implement `VmnetPool` per § Concurrency: claimed-slot set + `max_slots` + per-session network map, tracking the N NIC slots and the cross-backend session-count cap as two distinct things (Lima claims a NIC slot + a token; lite claims only a token).
- Implement /29 subnet allocation and the per-slot address roles relative to base `B` per § Subnet allocation, plus slot claim/release per § Slot lifecycle and the § Pool exhaustion error.
- Generate the `io.sandboxd.vmnet.<n>` launchd plists (host mode + deterministic per-slot network-identifier UUID + `--socket-group=_sandbox`, `RunAtLoad`+`KeepAlive`, sockets under `/var/run/sandbox/vmnet/`) per § Initialization and § M15 install.sh socket_vmnet pool daemons.
- Land a **`make setup-dev-env` Darwin dev bring-up as an explicit deliverable** (vmnet-pool bootstrap + the dev agent-registration path) that M19-S3 / M20-S2 / M20-S6 *consume* rather than each re-inventing — this is the single owner of the dev-time bring-up the review flagged as ownerless. **Stage `socket_vmnet` to the root-owned `/Library/sandboxd` path and land the daemon's startup ancestry+ownership assertion here** (pulled forward from M22-S2 so the dev/CI root-exec posture is not weaker than production for the rest of the arc; only the launchd codesign pre-exec guard + full release install stay in M22-S2), per § socket_vmnet access (root-exec foothold).
- Record the slot→socket-path map for the template generator to embed the per-VM `socket:` entry (no `networks.yaml` written), per § Initialization.

**Explicitly deferred.**

- The launchd codesign/cdhash **pre-exec guard** and the full release-shaped install of the pool daemons → M22-S2. (The root-owned `/Library/sandboxd` staging + the daemon startup ancestry assertion are pulled *into this session* — see In scope — so the dev/CI root-exec posture matches production across the arc.)
- Pool resize when `max_macos_sessions` changes → M21-S3.
- Consuming a slot from a real session create → M20-S4.

**Exit criteria.**

- Integration tests cover slot claim/release, pool exhaustion, the NIC-slot-vs-session-token split (lite tokens consume no NIC), and that no `networks.yaml` is ever written.
- On a dev host the N vmnet daemons bootstrap from the root-owned `/Library/sandboxd` path and their slot sockets appear under `/var/run/sandbox/vmnet/`, reachable by an `_sandbox`-group member; the daemon's startup ancestry+ownership assertion **refuses a non-root-owned / admin-writable** socket_vmnet binary (negative test).
- **Inter-slot L2 isolation is a mandatory automated regression** — two slots, a same-subnet ARP/datapath across distinct `--vmnet-network-identifier`s fails — *not* a "where feasible" assertion. It is a core topological control and must not silently degrade; re-run it after the M21-S3 / M22-S2 pool-plist regen.
- Code review approved; the /29 role table documented at the allocator definition.

## M20-S2 — Gateway Lima instance: template, lifecycle, NIC discovery, egress plumbing

**Entry criteria.** M20-S1. (Testing this session's NIC discovery additionally requires the M20-S1 dev vmnet daemons to be *running and reachable*, not merely their plists generated — an empty `limaN` set means missing infrastructure, not a discovery bug.)

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — § Gateway Lima instance (all subsections); § socket_vmnet pool → Gateway → internet egress (sandboxd-egress bridge, baseline egress nft); § Daemon startup sequence (step 4); § Upstream DNS resolver discovery; § Session lifecycle → NIC interface naming.

**Rationale.** The gateway Lima instance is the persistent daemon-owned Linux VM that hosts every session's gateway containers and provides the only sanctioned egress uplink; it must come up before any sandbox VM and expose its Docker socket to the host. This session builds its template (N pre-attached socket_vmnet NICs, forwarded Docker socket, network hardening), its lifecycle and readiness, runtime NIC discovery by name, the shared `sandboxd-egress` NAT bridge, and the MUST-level baseline egress nft policy that bounds a compromised-gateway blast radius. These are one coherent subsystem, so they land together.

**In scope.**

- Generate and manage the gateway instance template + lifecycle + readiness per § Lima template structure, § Lifecycle, § Readiness check (forwarded socket, persistence across daemon restarts, MTU clamp / IPv6-disable / no-DHCP netplan hardening).
- Implement runtime NIC discovery (`limactl shell … ip -json link`, select `limaN` by name excluding `eth0`, build the slot→NIC map) per § NIC interface naming, run as `_sandbox` via the seam's degenerate path.
- Create the shared `sandboxd-egress` NAT bridge and apply the baseline default-drop egress nft policy on the gateway VM's `eth0`, per § Gateway → internet egress (defense-in-depth on the shared uplink) and § Daemon startup steps 4f/4g.
- Rely on Lima+Docker resolver propagation for upstream DNS (no macOS-specific DNS work) per § Upstream DNS resolver discovery.

**Explicitly deferred.**

- Gateway-image `docker load` into the instance + self-heal → M22-S5.
- Gateway crash recovery → M21-S1.
- Pool-resize-driven gateway rebuild → M21-S3.

**Exit criteria.**

- Integration tests / a dev bring-up confirm the gateway instance creates, starts, exposes a reachable forwarded Docker socket (`docker -H … info`), discovers `limaN` NICs by name, creates `sandboxd-egress`, and applies the baseline egress nft policy.
- The `eth0` baseline egress policy is verified to allow only re-originated upstream + DNS (the gateway VM is never `restrict=on`), per the MUST in § Gateway → internet egress.
- Code review approved; docs updated where they describe gateway topology on macOS.

## M20-S3 — Sandbox VM template + the narrow `restrict=on` qemu-wrapper

**Entry criteria.** M20-S2.

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — § Sandbox VM template (macOS) (vmType and network; vzNAT and user-mode SLIRP must never be used; No Linux seccomp/cgroup wrapper; Operator-uid alignment; Workspace mount type; Everything else is identical); § Networking design corrections #5 (rationale record).

**Rationale.** This is where the core egress promise is enforced at the VM layer: the sandbox VM attaches its gateway-facing NIC via socket_vmnet, and Lima's unavoidable slirp management `eth0` is locked down with `-netdev user,restrict=on` via a narrow, fail-closed qemu-wrapper — a QEMU/slirp-layer hard block, not an overridable route metric (the model Linux already shipped at commit `4b134ec`). The wrapper is host-side string-surgery on Lima-generated argv, so it must fail closed and emit a marker for broker-mediated post-boot verification. Operator-uid alignment and the 9p shared mount fall out of the QEMU decision and are template-generation concerns, so they land here.

**In scope.**

- Generate the macOS sandbox VM template per § vmType and network (`vmType: "qemu"`, per-VM socket_vmnet `socket:` entry, single aarch64 image, shared URL constants) with the Linux seccomp/cgroup wrapper gated off per § No Linux seccomp/cgroup wrapper.
- Implement the narrow `restrict=on` qemu-wrapper per § vzNAT and SLIRP → The rewrite (fail-closed in-place rewrite of the management `user` netdev preserving `hostfwd`; firmware-path / HVF-via-exec specifics; marker file), gated on **instance kind** (sandbox VM only — never the gateway instance). **Reuse the Linux in-place rewrite logic** — the spec's note that the Linux wrapper only *appends* is inaccurate (Linux's `lima.rs` qemu-wrapper already rewrites every `-netdev user,…` token to splice `restrict=on`), so the genuine macOS deltas are the firmware-path + HVF-via-exec handling, **not** the rewrite algorithm; correct the spec's "append vs rewrite" wording in-branch.
- Implement operator-uid-alignment cloud-init (groupmod/usermod/chown, elided at uid 1000) and the `0600` SSH-key umask path per § Operator-uid alignment; emit `mountType: "9p"` with the `securityModel` sub-stanza per § Workspace mount type.
- Wire the brokered `Create`/`Start` payload to carry the `--qemu-wrapper` path the agent asserts (the daemon-authored side of M19-S4's validation).

**Explicitly deferred.**

- The broker-mediated post-boot `restrict=on` verification step + the mandated delete-route E2E regression → M20-S6.
- The create/start flow that invokes the wrapper → M20-S4.

**Exit criteria.**

- Unit/integration tests cover the wrapper's fail-closed behaviour (zero or >1 `user` netdev → abort; `restrict=on` confirmed in the rewritten token; `hostfwd` preserved) and operator-uid-alignment cloud-init emission (present off-default-uid, elided at 1000). **Scope note:** S3 proves the wrapper's *argv-surgery + firmware-dir staging shape* only — the first actual HVF boot *through* the wrapper is M20-S4, and the egress proof is M20-S6.
- The generated template is the daemon-authored blob the agent re-validates (socket_vmnet-only NICs, asserted wrapper) — round-trips through M19-S4's validation in test, **and updates M19-S4's validator fixtures to this real rendered output** (so the validator is exercised against ground truth, not the synthetic stand-in).
- Code review approved with attention to the wrapper as an egress-enforcement surface; spec corrected if the argv-rewrite shape diverges from current Lima output.

## M20-S4 — macOS session lifecycle (create/start/stop/rm) + `vmnet_slot` schema

**Entry criteria.** M20-S3.

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — § Session lifecycle on macOS (NIC interface naming, create, start, stop, rm); § Session store schema changes (network_info JSON blob, sandbox inspect output).

**Rationale.** This session assembles the pieces into the actual session machinery: the create flow (slot claim → macvlan network → two-NIC gateway container → deny-all → health → DNAT → wrapped `limactl start` → verify), and the start/stop/rm flows, all routed through the operator-execution seam. The egress happens-before invariant — the VM's NIC never carries traffic until its gateway has deny-all *and* DNAT — is load-bearing and the step ordering enforces it. The only new persisted state is `vmnet_slot` inside the existing `NetworkInfo` JSON blob, requiring no migration.

**In scope.**

- Implement create per § create (operator-identity capture + `ensure_operator_lima_home`, slot claim, macvlan `sandbox-net-<id>` with the correct name the orphan-reaper parses, two-NIC gateway container `sandbox-gw-<id>`, deny-all → health → DNAT, wrapped `limactl start` via the seam, restrict-verification gate, partial-create rollback) and the § Egress happens-before invariant.
- Implement start (fresh non-sticky slot, YAML rewrite, gateway-before-VM), stop (DNAT flush, VM stop, gateway/network teardown, slot release), and rm per § start / § stop / § rm.
- Add the `vmnet_slot: Option<VmnetSlot>` field to `NetworkInfo` with `#[serde(default)]` (no migration — confirmed against `store.rs`) and the `sandbox inspect` three-line output per § Session store schema changes.

**Explicitly deferred.**

- The broker-mediated post-boot `restrict=on` verification implementation + proof → M20-S6 (this session calls the verification step; S6 proves it end-to-end).
- Daemon-startup reconcile of these sessions → M20-S5.
- Failure-mode recovery (gateway crash, agent crash) → M21.

**Exit criteria.**

- Integration tests cover create/start/stop/rm slot accounting, the macvlan/gateway-container naming the orphan-reaper parses, partial-create rollback (no leaked slot/container/VM), and `vmnet_slot` round-tripping in the JSON blob against an old-row deserialize.
- **Integration tests assert the gateway container's hardened config** — no `--privileged`, read-only rootfs, no host mounts, exactly the one added `NET_ADMIN` cap, the in-VM Docker socket never bind-mounted in — i.e. the § Security posture / Known-deltas blast-radius controls are *asserted, not assumed*; and `net.inet.ip{,6}.forwarding == 0` is re-checked at create (defense-in-depth against a runtime flip by a VPN client, not only at daemon startup).
- `sandbox inspect` renders the macOS slot lines; full hermetic + integration suites green.
- Code review approved; the happens-before invariant documented at the create flow.

## M20-S5 — macOS daemon startup sequence + reconcile

**Entry criteria.** M20-S4.

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — § Daemon startup sequence (macOS) (the eight steps); § Failure handling → Slot leak detection (startup portion); § socket_vmnet pool → Gateway → internet egress (host IP-forwarding assertion).

**Rationale.** The startup sequence wires every subsystem together and is where a macOS daemon first comes fully alive: prerequisite checks (incl. the hard IP-forwarding-off assertion the egress guarantee depends on), pool init, gateway init, the two-halves reconcile (unconditional gateway-side cleanup vs agent-gated VM-side reconcile), and pool-state restore. The reconcile split is subtle and load-bearing — a logged-out operator's leftover gateway containers must be reclaimed without waiting for that operator to log back in. This session makes "the daemon always starts" true.

**In scope.**

- Implement the eight-step startup sequence per § Daemon startup sequence (macOS): executor/broker bind (from M19-S5), prerequisite checks incl. the `net.inet.ip{,6}.forwarding == 0` hard assertion, pool init (slot→socket map only), gateway init (from M20-S2), session load, the two-halves reconcile, pool-state restore, accept requests.
- Implement the reconcile split per § step 6: gateway-side cleanup runs unconditionally for every non-running session + orphaned containers; VM-side reconcile is agent-gated and tied to a fresh check-in.
- Implement startup slot-leak release (stopped/nonexistent sessions hold no slot) per § Slot leak detection.

**Explicitly deferred.**

- Live failure recovery (gateway crash poll loop, agent-crash respawn) → M21-S1 / M21-S2.
- Pool resize on NIC-count mismatch → M21-S3 (this session defers a config-vs-actual mismatch to that handler rather than aborting).

**Exit criteria.**

- A macOS daemon starts end-to-end against a dev pool + gateway, reconciles a pre-existing stopped session, and refuses to start with the actionable error when host IP-forwarding is on.
- Integration tests cover the reconcile split (gateway-side cleanup runs for a not-logged-in operator's session; VM-side reconcile waits for check-in) and startup slot-leak release.
- Code review approved; the startup sequence + reconcile split documented at the startup site.

## M20-S6 — First-light egress E2E + post-boot `restrict=on` verification + egress hardening review

**Entry criteria.** M20-S5.

**Spec reference.** [`../../../.tasks/specs/2026-05-02-macos-support-design.md`](../../../.tasks/specs/2026-05-02-macos-support-design.md) — § Sandbox VM template → vzNAT and SLIRP (Post-boot verification; "macOS MUST carry the same E2E regression"); § Security posture → Topological controls; § Session lifecycle → create → Egress happens-before invariant; § M15 → macOS e2e harness adaptation (minimal bring-up).

**Rationale.** M20's deliverable — a full-backend macOS session that boots with *provably*-enforced egress — is not real until an E2E test proves the egress block holds against a root guest, mirroring the landed Linux #65 regression. This session implements the broker-mediated post-boot verification (the agent reads the wrapper marker / queries QMP, since the QEMU process is operator-owned and unreadable by the daemon), brings up the minimal macOS e2e needed to boot one session, and runs the delete-the-gateway-route-still-blocked regression. It closes with a focused review of the egress-enforcement invariants because they are sandboxd's core promise.

**In scope.**

- Implement the broker-mediated post-boot `restrict=on` verification per § Post-boot verification (using the reviewed broker op from M19-S4), **preferring the live QMP `query-netdev` query over the operator-writable marker file** as the authoritative evidence (the marker is fallback); daemon refuses to mark `Running` unless confirmed, tear-down on failure.
- Stand up the minimal macOS e2e capability (provisional; folded into the conftest `_platform` seam at M22-S7) per § macOS e2e harness adaptation, scoped to what proves egress. It must already distinguish the **GUI/Aqua-agent-present** mode (full backend) from the agent-bring-up plumbing, since M21-S4's SSH-only lite path depends on that distinction; the full harness adaptation is deferred.
- **Pull a minimal `build-macos` CI lane forward into this session** — build + **ad-hoc-sign every Mach-O** (required: AMFI `SIGKILL`s any unsigned arm64 Mach-O at `execve`, before Gatekeeper) + run this egress regression on the `macos-14` runner — so the core-promise gate runs in CI on real Apple-Silicon as soon as M20 code exists. The full CI matrix + Tart + signing pipeline stay in M22-S4.
- Land the mandated macOS egress regression per § vzNAT and SLIRP: boot a session, delete the gateway default route inside the guest, confirm slirp `eth0` still cannot reach the internet; confirm the isolated-segment topology (default route = gateway, on-link `B+1` dead-end) per § Topological controls.
- Egress-enforcement hardening review of the wrapper + verification + happens-before ordering + baseline gateway nft, fixing findings in-session.

**Explicitly deferred.**

- The full one-suite / thin-seam / marker harness adaptation across all four cells → M22-S7.
- Lite-mode egress path → M21-S4.

**Exit criteria (conjunctive for the egress claim).**

- The delete-gateway-route E2E regression passes **in CI on the `macos-14` runner** (not only a developer's local Mac): a root guest cannot egress via `eth0`; the socket_vmnet gateway path is the only route out.
- The post-boot verification refuses `Running` when `restrict=on` is unconfirmed (negative test); partial-create rollback fires on verification failure.
- Egress hardening review findings fixed in-branch; full hermetic + integration green; code review approved.
