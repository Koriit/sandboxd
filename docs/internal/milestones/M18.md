# M18 — Cross-user CLI access

Six CLI subcommands (`sandbox ssh`, `cp`, `sync`, `workspace push`, `workspace pull`, and transitively `git-remote-sandbox`) shell out to backend-native tools that are user-scoped — `limactl` walks the calling user's `~/.lima/`, so when the daemon (running as the `sandbox` system user) registered a VM under `/home/sandbox/.lima/sandbox-<id>/`, an operator running the CLI as their own UID sees an empty Lima registry and the lookup fails. The gap is currently masked in CI because the e2e harness launches the daemon as the test user; the cross-user code path is never exercised. M18 adopts the daemon-mediated approach used by Docker, libvirt, LXD, Kubernetes, and systemd-machined: every CLI operation that needs to reach inside a session goes through the daemon socket. SSH-shaped operations tunnel through a new `GET /sessions/{id}/proxy` WebSocket endpoint via `ProxyCommand`; the daemon distributes a per-session SSH config block (under `~/.ssh/sandbox/`, `Include`-d at the top of `~/.ssh/config`) so external SSH tooling — VS Code Remote-SSH, JetBrains Gateway, ad-hoc `scp`/`rsync` — works against either backend through the same generated config without our CLI being in the data path beyond the `ProxyCommand` shim. To make a single CLI code path cover both backends, the lite-image gains a minimal sshd (and its in-container guest user renames from `agent` to `sandbox` so the SSH-config template's `User sandbox` line works without per-backend branching).

Test infrastructure is reworked **before** any production code change: M18-S1 lands the cross-user e2e harness (sandboxd launched via its systemd unit, or `sudo -u sandbox sandboxd` where systemd is unavailable) and the three Phase-1 acceptance tests (Lima ssh, Lima git-remote-sandbox, container ssh) so the bug is reproducible in CI before fixes land. M18-S2..S7 implement Phase 2 of the spec: S2 bakes sshd into the lite-image; S3 ships the keypair migration (V007), per-session keypair generation, tmpfs `authorized_keys` injection, and the `GET /sessions/{id}/ssh-config` endpoint with `SshConfigDto`; S4 ships the WebSocket proxy endpoint with the explicit `tokio::process::Command` carve-out from the project's `spawn_blocking` convention; S5 ships the CLI `sandbox proxy` hidden shim and the persistent `~/.ssh/sandbox/` management module (flock + tempfile+rename atomic writes, marker-delimited `Include` at the top of `~/.ssh/config`); S6 rewrites the six broken CLI commands to use the `sandbox-<id>` alias and wires the single-retry drift-recovery path (matched only at outermost CLI dispatch); S7 wires the lifecycle hooks — `sandbox rm` cleanup, `sandbox ls` opportunistic reconcile against the daemon's session list, and `sandbox proxy <id>` lazy-404 cleanup. M18-S8 is the **six-track review** across S1..S7, each track delegated to a separate subagent in parallel: (1) implementation vs. spec, (2) code quality, (3) unit-test quality, (4) e2e-test quality, (5) docs quality, (6) workarounds + deprecated patterns. M18-S9 closes the milestone: full e2e matrix under the new harness, manual `git-remote-sandbox` and VS Code Remote-SSH sanity checks, plus the spec-delivery claim-to-code verification gate. M18 is the **terminal milestone** for the 2026-05-24 spec.

## M18-S3 — Daemon: keypair migration + injection + `GET /sessions/{id}/ssh-config`

**Entry criteria.** M18-S2 complete — the lite-image accepts a pubkey at `/run/sandbox/authorized_keys` and runs sshd against it.

**Spec reference.** [`../../../.tasks/specs/2026-05-24-cross-user-cli-access-design/2026-05-24-cross-user-cli-access-design-spec.md`](../../../.tasks/specs/2026-05-24-cross-user-cli-access-design/2026-05-24-cross-user-cli-access-design-spec.md) — §§ Architecture → Daemon: per-session SSH credentials, Daemon API: two new endpoints → `GET /sessions/{id}/ssh-config`, Phase 2 — implementation (steps 2 and 3), Persisted state forward-compat, Security considerations (trust model statement).

**Rationale.** Steps 2 and 3 in the spec are tightly coupled: the migration + keypair generation exists to populate state that the endpoint serves, and the endpoint exists to serve state the migration produces. Splitting them creates a half-useful intermediate where the daemon stores keys it cannot expose, or exposes an endpoint that has nothing to serve — neither is a useful checkpoint. Landing them as a single session also keeps the persisted-state contract (Spec § Persisted state forward-compat) in one diff: the new SQLite column, the `Option<SshKeypair>` field with `#[serde(default)]`, and the `404 SSH_NOT_AVAILABLE` branch for pre-migration rows all describe one coherent forward-compat story. The dedicated `SshConfigDto` (not a flattened domain struct) follows the project's `feedback_api_dto_separation` convention; the trust-model restatement in Spec § Security considerations ("members of the `sandbox` OS group are trusted with every session's private key") is the load-bearing premise that justifies storing keys plaintext at rest under DB file permissions and is documented inline in the new code.

**In scope.**

- New forward-only migration `V007__add_ssh_keypair.sql` per Spec § Persisted state forward-compat: adds `ssh_keypair_json BLOB NULL` to the `sessions` table; nullable so existing rows continue to deserialise.
- `SessionRow` (or equivalent persisted struct) gains an `ssh_keypair: Option<SshKeypair>` field with `#[serde(default)]` per Spec § Persisted state forward-compat — older daemons reading newer rows skip the field; newer daemons reading older rows see `None`.
- `SshKeypair` domain type holding the public (`ssh-ed25519 …`) and private (PEM) halves per Spec § Architecture → Daemon: per-session SSH credentials.
- Container session-create path generates an ed25519 keypair, persists it in the new column, and injects the pubkey into the container at start via a tmpfs bind-mount at `/run/sandbox/authorized_keys` per Spec § Architecture → Daemon: per-session SSH credentials.
- Lima session-create path: no new persistence — the daemon reads Lima's own per-VM SSH credentials on demand when serving the endpoint per Spec § Architecture → Daemon: per-session SSH credentials.
- `GET /sessions/{id}/ssh-config` endpoint per Spec § Daemon API → `GET /sessions/{id}/ssh-config`: response is `SshConfigDto { config: String, private_key: String }` placed under the daemon's `api::dto` module (dedicated DTO, not a flattened domain serialisation, per the project's DTO-separation convention). Config text shape exactly as in Spec § Daemon API (`Host sandbox-<id>` … `ProxyCommand sandbox proxy <id>` … `IdentityFile <CLI-rewrites-this>` … `ControlMaster`/`ControlPath`/`ControlPersist` block).
- Session-ownership check per Spec § Architecture → Daemon: per-session SSH credentials → "single-operator throughout": the endpoint reuses the same ownership check pattern as every other `/sessions/{id}/...` handler (existing `owner_username` column).
- `404 Not Found` with typed error code `SSH_NOT_AVAILABLE` per Spec § Daemon API → `GET /sessions/{id}/ssh-config` for pre-existing container sessions without a keypair (no lazy keypair generation — injecting a new `authorized_keys` into a running container would require sshd hot reload that the lite-image is not designed for).
- `integration_*` per backend per Spec § Phase 2 step 3 (test layer): real session created against each backend, fetch ssh-config, decode the DTO shape, assert config text contains the documented placeholders (`HostName`, `Port`, `User sandbox`, `ProxyCommand sandbox proxy <id>`, `IdentityFile <CLI-rewrites-this>`, multiplexing block).
- Inline code-comment restating the trust model from Spec § Security considerations at the keypair-generation site (so a future reviewer touching the DB schema does not accidentally weaken the "members of the `sandbox` OS group are trusted with every session's private key" premise).

- CLI `proxy` shim + `~/.ssh/sandbox/` module → M18-S5.
- Rewrite of the six broken CLI commands + drift recovery → M18-S6.
- Lifecycle hooks → M18-S7.
- Six-track review across S1..S7 → M18-S8.
- Phase 3 + spec-delivery verification → M18-S9.

## M18-S4 — Daemon: `GET /sessions/{id}/proxy` WebSocket byte mover

**Entry criteria.** M18-S3 complete — the ssh-config endpoint exists and the integration substrate is green; sessions on both backends have the means to authenticate an incoming SSH connection.

**Spec reference.** [`../../../.tasks/specs/2026-05-24-cross-user-cli-access-design/2026-05-24-cross-user-cli-access-design-spec.md`](../../../.tasks/specs/2026-05-24-cross-user-cli-access-design/2026-05-24-cross-user-cli-access-design-spec.md) — §§ Daemon API: two new endpoints → `GET /sessions/{id}/proxy`, Architecture → Daemon API → Async I/O note, Phase 2 — implementation (step 4), Decision → Why unify both backends on sshd.

**Rationale.** The proxy endpoint is the load-bearing piece of the daemon-mediated design: without it the SSH config block has nowhere to dial. It is isolated as its own session because the long-lived async I/O contract (Spec § Architecture → Async I/O note) is the one place in the codebase where the project's `spawn_blocking` convention is deliberately not followed — wrapping a long-lived `docker exec`/`socat` byte pump inside `spawn_blocking` would occupy a blocking task slot for the entire SSH session (potentially hours for an IDE) and deadlock the executor under load. Landing this carve-out as a focused diff makes the deviation visible to reviewers and lets the inline code comment land in the same change that justifies it, so a future "spawn_blocking everywhere" sweep does not silently regress it. The endpoint is a dumb byte mover — no SSH-protocol parsing, no channel-id framing (the spec explicitly excludes channel-id framing here; SSH does its own multiplexing inside the tunnel) — and the per-backend transport plumbing is the only non-trivial code.

**In scope.**

- HTTP-to-WebSocket upgrade handshake per Spec § Daemon API → `GET /sessions/{id}/proxy`: client performs the standard upgrade; daemon responds with `101 Switching Protocols`; binary frames carry raw SSH bytes (no per-frame framing layer).
- Lima backend forwarding per Spec § Daemon API → `GET /sessions/{id}/proxy` (Lima bullet): discover the per-VM SSH port by running `limactl list --format=json` and reading the `sshLocalPort` field on the matching instance (Lima's documented machine-readable surface, stable across Lima minor versions); cache the port in memory per session for the duration of the proxy connection; re-query on each new proxy request (Lima may reassign on instance restart). Open a `tokio::net::TcpStream` to `127.0.0.1:<sshLocalPort>` and `tokio::io::copy_bidirectional` between the WebSocket and the TCP stream.
- Container backend forwarding per Spec § Daemon API → `GET /sessions/{id}/proxy` (container bullet): spawn `tokio::process::Command` running `docker exec <ctr> socat - TCP:127.0.0.1:22` with async pipes; `tokio::io::copy_bidirectional` between the WebSocket and the spawned child's stdio.
- **Async I/O carve-out** per Spec § Architecture → Async I/O note: the long-lived byte-pump uses `tokio::process::Command` (async pipes, no `spawn_blocking` wrapper) because a blocking-task-slot capture for the SSH session's full duration would deadlock the executor under load. The one-shot `limactl list` invocation continues to follow the existing `spawn_blocking` convention. **Inline code comments at both call sites** explain the carve-out so a future drive-by sweep does not regress either decision.
- Session-ownership check reused from the existing `/sessions/{id}/...` handler pattern per Spec § Daemon API → `GET /sessions/{id}/proxy`: the endpoint enforces session ownership before opening the WebSocket; bytes after the handshake are not inspected, but cannot reach a session the calling operator does not own.
- **No** SSH-protocol parsing inside the proxy — the endpoint is a dumb byte mover per Spec § Daemon API → `GET /sessions/{id}/proxy`.
- **No** channel-id framing per Spec § Daemon API → `GET /sessions/{id}/proxy`: a single raw byte stream; SSH does its own multiplexing inside the tunnel. (Channel-id framing is explicitly out of scope here; a future non-SSH streaming-exec endpoint would adopt it, but that endpoint is itself out of scope per Spec § Non-goals.)
- WebSocket library choice per Spec § Daemon API → `GET /sessions/{id}/proxy` (the K8s precedent paragraph): standard binary-frame WebSocket over the daemon's existing HTTP-on-Unix-socket surface; no SPDY, no bespoke upgrade token. Concrete library selection follows whatever the daemon's existing HTTP stack already uses.
- `integration_*` per backend per Spec § Phase 2 step 4 (test layer): open a proxy connection to a real session, exchange bytes through a stub-sshd inside the session (e.g. an in-VM `cat` or a deliberately-trivial sshd stand-in), assert byte-perfect round-trip across both directions; close from either end propagates correctly.

**Explicitly deferred.**

- CLI `proxy` shim + `~/.ssh/sandbox/` module → M18-S5.
- Rewrite of the six broken CLI commands + drift recovery → M18-S6.
- Lifecycle hooks → M18-S7.
- Six-track review across S1..S7 → M18-S8.
- Phase 3 + spec-delivery verification → M18-S9.
- Streaming-protocol design for the non-SSH `sandbox exec` path (Spec § Non-goals; Spec § Alternatives → Custom streaming protocol).
- Unix-domain-socket-per-session transport (Spec § Alternatives → Unix-domain-socket transport — rejected to preserve CLI forward-compat with macOS and remote daemons).

**Exit criteria.**

- WebSocket handshake succeeds for both backends against a real session; daemon responds with `101 Switching Protocols`.
- Bidirectional byte transfer is byte-perfect for payloads of varied sizes (small, MTU-crossing, large enough to exercise WebSocket fragmentation) on both backends — verified by `integration_*`.
- Close from either end (client → daemon and session → daemon) propagates to the other side and tears down the spawned child / TcpStream cleanly with no resource leak (verified by inspecting daemon process state or container `ps` snapshot in the integration test).
- Session-ownership rejection (`403`) fires before the WebSocket handshake completes for a non-owner caller; the daemon never opens the per-backend transport for a rejected request.
- **Async I/O carve-out comment** present at both the `tokio::process::Command` call site (container path) and the `limactl list` `spawn_blocking` call site (Lima path), each referencing Spec § Architecture → Async I/O note inline so the reviewer sees the rationale without leaving the file.
- The Phase-1 acceptance set from M18-S1 still has its expected M18-S1 pattern (S4 ships substrate; the CLI does not consume the proxy yet).
- `cd sandboxd && cargo nextest run --workspace` default profile clean; `cd sandboxd && cargo nextest run --workspace --profile integration` clean.
- `cd sandboxd && cargo clippy --workspace -- -D warnings` clean; `cd sandboxd && cargo fmt --check` clean.
- Code review approved; relevant internal-doc references (`CLAUDE.md` convention block, or sibling notes) updated to call out the async-I/O carve-out so the project-wide `spawn_blocking` rule no longer appears unqualified.

## M18-S5 — CLI: `sandbox proxy` hidden shim + `~/.ssh/sandbox/` management module

**Entry criteria.** M18-S4 complete — the daemon-side proxy endpoint accepts WebSocket connections and forwards bytes to sshd on both backends.

**Spec reference.** [`../../../.tasks/specs/2026-05-24-cross-user-cli-access-design/2026-05-24-cross-user-cli-access-design-spec.md`](../../../.tasks/specs/2026-05-24-cross-user-cli-access-design/2026-05-24-cross-user-cli-access-design-spec.md) — §§ Architecture → CLI: persistent ssh-config + thin `sandbox proxy` (file layout, mode bits, `Include` line marker block, concurrency and atomicity), Phase 2 — implementation (steps 5 and 6).

**Rationale.** The CLI shim and the management module ship together because step 5 alone is useless without the generated SSH config block that names `ProxyCommand sandbox proxy <id>`, and step 6 alone has nothing to integrate against until the shim exists. Both are hermetically testable against a tempdir `$HOME` and a mock WebSocket, so the combined session has a clean self-contained verification surface — no real daemon or VM is required for S5 to land. The `~/.ssh/sandbox/` module is the only place in the project that mutates files under the operator's home, so the load-bearing safety contracts (flock for concurrency, tempfile + atomic rename for crash-safety, `Include` line at the very top of `~/.ssh/config` to avoid first-match-wins shadowing by an existing `Host *` block) all live in one diff under one set of tests. The `sandbox proxy` subcommand name is treated as wire format from this point on per Spec § Architecture → CLI: persistent ssh-config — once it ships, changes to it must be coordinated with the generated config template.

**In scope.**

- `sandbox proxy <id>` registered as a **hidden** subcommand (`#[command(hide = true)]` or equivalent) per Spec § Architecture → CLI: persistent ssh-config — does not appear in `sandbox --help`; the generated SSH config is the only intended caller.
- Subcommand body per Spec § Architecture → CLI: persistent ssh-config: open a WebSocket against the daemon's `/sessions/{id}/proxy` endpoint, perform the handshake, bidirectionally splice its own stdin/stdout with binary WebSocket frames. Thin shim — no business logic, no SSH parsing, no retry; failures propagate to the operator's `ssh` client as a `ProxyCommand` exit.

## M18-S7 — CLI lifecycle hooks: `rm` cleanup + `ls` reconcile + `proxy` lazy-404

**Entry criteria.** M18-S6 complete — the six broken commands work cross-user; per-session entries are created in `~/.ssh/sandbox/` on first command for a session.

**Spec reference.** [`../../../.tasks/specs/2026-05-24-cross-user-cli-access-design/2026-05-24-cross-user-cli-access-design-spec.md`](../../../.tasks/specs/2026-05-24-cross-user-cli-access-design/2026-05-24-cross-user-cli-access-design-spec.md) — §§ Architecture → CLI: persistent ssh-config (Per-session entry removal, Lazy cleanup, Reconcile on listing), Phase 2 — implementation (steps 9 and 10).

**Rationale.** With per-session entries being created in M18-S6, the milestone needs the three lifecycle hooks that keep `~/.ssh/sandbox/` consistent with the daemon's authoritative session list. The spec explicitly rejects a separate `gc` subcommand (Spec § Architecture → CLI: persistent ssh-config) — the three mechanisms here (explicit removal on `rm`, lazy cleanup on proxy 404, reconcile on `ls`) cover every realistic code path, and stale entries are harmless because the proxy endpoint 404s if anyone attempts to use one. Landing them together is the right shape: each is small, they share the same flock and the same marker-delimited-block primitive from M18-S5, and verifying them in isolation is much cheaper than verifying any single one with realistic out-of-band scenarios. The reconcile-firing rules (only on full-list query, not on single-id queries, silently skipped if the daemon is unreachable, `--no-reconcile` opt-out for machine-readable pipelines) are guard rails against the reconcile turning into a footgun in tooling consumers.

**In scope.**

- `sandbox rm <id>` per-session entry removal per Spec § Architecture → CLI: persistent ssh-config (Per-session entry removal): remove `~/.ssh/sandbox/keys/<id>`; remove the marker-delimited block for `<id>` from `~/.ssh/sandbox/config`. Both mutations go through the existing flock + tempfile + rename primitive from M18-S5.
- `sandbox proxy <id>` lazy-404 cleanup per Spec § Architecture → CLI: persistent ssh-config (Lazy cleanup): on a `404` response from the daemon (session does not exist), the CLI removes the local entry before exiting. This is the **one** place lifecycle work happens inside `sandbox proxy` — it does not violate the "thin shim" principle from M18-S5 because cleanup happens after the WebSocket fails, not as part of normal operation.
- `sandbox ls` opportunistic reconcile per Spec § Architecture → CLI: persistent ssh-config (Reconcile on listing): when the daemon returns the full session list for the calling operator (the default `ls` invocation), remove local entries the daemon does not know about. **Does not** fire on single-id queries (`sandbox inspect`, `sandbox describe`) where the absence of one id does not imply the others are stale. **Silently skipped** if the daemon is unreachable — `ls` falls through to whatever cached/local behaviour the existing command already exposes; no regression in error mode. `--no-reconcile` flag opt-out for tooling consumers that need strict read-only semantics (e.g. machine-readable `--output json` pipelines). The reconcile pass acquires the **same** flock on `~/.ssh/sandbox/.lock` as the write path so a concurrent `sandbox ssh` cannot race against entry removal.
- **No** separate `gc` subcommand per Spec § Architecture → CLI: persistent ssh-config — the three mechanisms above cover every realistic code path.
- ControlMaster sockets at `~/.ssh/sandbox/sockets/` are **not** explicitly cleaned per Spec § Architecture → CLI: persistent ssh-config — OpenSSH cleans them on master exit; leaked stale sockets are harmless and SSH overwrites them on next master start.
- Hermetic tests for each helper: `rm` removal helper against a tempdir `$HOME` with both the key file and the marker block present; `ls` reconcile against a mocked daemon list response with stale entries; `proxy` lazy-404 path against a mock daemon returning `404`. Add a hermetic test for the `--no-reconcile` opt-out and for the silent-skip-on-unreachable-daemon branch.
- E2E coverage per Spec § Phase 2 steps 9 and 10: a create / use / `rm` cycle proves the full lifecycle; an out-of-band session deletion followed by `sandbox ls` proves the reconcile path.

**Explicitly deferred.**

- Six-track review across S1..S7 → M18-S8.
- Phase 3 validation + spec-delivery verification → M18-S9.
- Eager `gc` subcommand (Spec § Architecture → CLI: persistent ssh-config rejects this).
- Cleanup of leaked ControlMaster socket files (Spec § Architecture → CLI: persistent ssh-config defers this — OpenSSH and the next master start handle it).
- Multi-tenant per-operation authorization on the lifecycle paths (Spec § Non-goals: daemon enforces session ownership and nothing more in this milestone).

**Exit criteria.**

- `sandbox rm` removes both `~/.ssh/sandbox/keys/<id>` and the marker-delimited config block for the removed session; verified by hermetic tempdir-`$HOME` test and by e2e create/use/rm cycle.
- `sandbox proxy <id>` on a daemon `404` removes the local entry before exiting; verified by hermetic mock-daemon test.
